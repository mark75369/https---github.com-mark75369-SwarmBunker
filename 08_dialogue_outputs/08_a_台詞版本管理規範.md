# 1. 文件目的

本文件用於管理台詞版本。

長篇遊戲劇本在使用 AI / Codex 生成台詞時，常見問題包括：

- 試寫版覆蓋定稿版
- 不知道哪一版是最新可用版
- 多版本之間沒有比較依據
- QA 修正後無法追蹤改了什麼
- 台詞功能、聲線、資訊揭露在修稿中被破壞
- 人類修改與 AI 修改混在一起
- 舊版台詞被誤當作正式版本
- 場景任務包更新後，台詞版本沒有同步
- 禁用詞或稱呼解禁規則改了，但舊台詞未重新檢查

本文件的目的，是建立一套穩定的版本管理規範，讓每個台詞檔案都能回答：

- 這是哪一場戲的台詞？
- 這是試寫、修稿、QA 還是定稿？
- 這版根據哪份任務包生成？
- 這版是否通過聲線、資訊、禁用詞與反 AI 味檢查？
- 這版是否可作為正式交付？
- 這版與前一版差異是什麼？
- 這版是否已被棄用？

# 本文件專屬檢查維度

> 速查卡（v0.2 新增）：本文件在版本管理流程中需檢查的具體維度。詳細規則仍以後續各章為準。

| 維度 | 說明 |
|---|---|
| 版本狀態 | Lifecycle Status (`DRAFT` / `ALT` / `REVIEW` / `QA_PASS` / `FINAL` / `LOCKED` / `DEPRECATED`) 是否正確 |
| 版本來源 | Mode Tag (`ORGANIZED` / `DRAFT_TRIAL` / `EXPERIMENTAL` / `CONVERGENCE` / `QA_REVIEW`) 是否標記 |
| 是否可進 QA | 任務包是否就位、上游資料是否齊全、版本是否非 `DEPRECATED` |
| 是否可交稿 | FINAL 條件是否全數通過、人類裁決項是否處理完畢 |
| 是否可 LOCKED | FINAL 已成立、上游無衝突、變更紀錄已存檔 |
| 是否可由 AI 直接修改 | 視 Lifecycle Status 與 Mode Tag 而定（`FINAL` / `LOCKED` 一律不可） |
| 是否需要回寫上游 | 視 final 台詞內容是否新增/改動世界觀、角色、關係、資訊 |

本文件的所有檢查項目，最終都必須回答：「這版能不能進下一個生命週期狀態？」

# 3. 基本原則

## 3.1 不覆蓋原始版本

除非使用者明確要求，AI / Codex 不得直接覆蓋既有台詞版本。

修稿應產生新版本，或在明確的版本欄位中記錄變更。

禁止：

- 直接把 `v1` 改成新的內容但仍叫 `v1`
- 直接覆蓋 `final`
- 在沒有變更紀錄的情況下修改 LOCKED 台詞
- 把試寫版覆蓋交稿版

---

## 3.2 每一版都必須有來源

每一份台詞版本都應記錄：

- 來源任務包
- 生成日期
- 生成模式
- 使用的角色聲線卡版本
- 使用的關係 / 資訊 / 禁用詞資料狀態
- QA 狀態
- 修改者
- 是否可用於正式劇本

若無法追溯來源，該版本不得標為 final。

---

## 3.3 final 不是「看起來最好」

台詞 final 版必須同時滿足：

- 場景功能完成
- 角色聲線一致
- 關係階段正確
- 資訊揭露正確
- 禁用詞通過
- 反 AI 味檢查通過
- 格式符合交稿規範
- 人類確認可用

不是文筆最順的版本就一定是 final。

# 4. 台詞版本狀態

## 4.1 狀態定義

| 狀態 | 意義 | 可否正式使用 |
|---|---|---|
| DRAFT | 草稿版，允許大幅修改 | 否 |
| ALT | 備選版本，可供比較 | 否 |
| REVIEW | 待審版，需要 QA 或人類審核 | 否 |
| QA_PASS | 通過 QA，但未最終定稿 | 視情況 |
| FINAL | 定稿版，可交付 | 是 |
| LOCKED | 鎖定版，不得擅自修改 | 是 |
| DEPRECATED | 棄用版，不得正式引用 | 否 |

---

## 4.2 狀態轉換

| 原狀態 | 可轉換為 | 條件 |
|---|---|---|
| DRAFT | REVIEW | 完成基本生成，可供審核 |
| REVIEW | QA_PASS | 通過必要 QA |
| QA_PASS | FINAL | 人類確認可交稿 |
| FINAL | LOCKED | 正式定稿並納入劇本 |
| 任意狀態 | DEPRECATED | 被新版本取代或設計變更 |
| FINAL / LOCKED | REVIEW | 需重大返修時，應另存新版 |

LOCKED 版不得直接改寫。  
若需修改，應建立新版本並保留原 LOCKED 版。

---

## 4.3 模式系統版本規則

### 4.3.1 探索版 / 破格版 / 收斂版 / QA_PASS / FINAL 的版本管理規則

| 版本類型 | 狀態 | 命名建議 | 可否直接升為 FINAL | 後續處理 |
|---|---|---|---|---|
| 探索版 | EXPERIMENTAL | CHxx_Sxx_explore_v01.md | 否 | 需經試寫或收斂整合 |
| 破格版 | EXPERIMENTAL | CHxx_Sxx_broken_v01.md | 否 | 必須經收斂與 QA |
| 收斂版 | REVIEW | CHxx_Sxx_converge_v01.md | 否 | 需 QA 後可升為 QA_PASS |
| QA_PASS | QA_PASS | CHxx_Sxx_QA_PASS_v01.md | 否 | 需人類確認後升為 FINAL |
| FINAL | FINAL | CHxx_Sxx_final.md | 是 | 可升為 LOCKED |

### 4.3.2 EXPERIMENTAL 或 ALT 版本不得直接升為 FINAL

EXPERIMENTAL 或 ALT 版本必須經以下流程才可升為 FINAL：

```text
EXPERIMENTAL / ALT
  ↓
收斂整合 (REVIEW)
  ↓
QA 檢查 (QA_PASS)
  ↓
人類確認 (FINAL)
```

不得跳過收斂或 QA 直接將探索 / 破格 / 試寫版本標記為 FINAL。

### 4.3.3 破格試寫必須經收斂與 QA

破格試寫版本：

- 不得直接作為正式台詞。
- 必須先經收斂模式整合亮點。
- 收斂後必須通過 QA 檢查。
- QA 通過後仍需人類確認是否保留破格亮點。

破格版本應標記為 EXPERIMENTAL，並在收斂報告中記錄哪些亮點被保留。

# 三層標籤體系（Lifecycle Status / Mode Tag / QA Decision）

> v0.2 新增。本節用於明確版本管理三層標籤的職責分工，**三層互不取代**。

## 層 1：Lifecycle Status / 版本正式狀態

沿用 §4.1 既有定義，這是版本的生命週期狀態：

```text
DRAFT / ALT / REVIEW / QA_PASS / FINAL / LOCKED / DEPRECATED
```

只有 Lifecycle Status 才能決定該版本是否可交稿、是否可鎖定、是否可被 AI 直接修改。

---

## 層 2：Mode Tag / 產出來源標籤

記錄該版本主要由哪個創作模式產出，用於辨識版本的「來源風險特性」：

| Mode Tag | 對應模式 | 中文 suffix | 限制強度 | 風險特性 |
|---|---|---|---|---|
| `ORGANIZED` | 整理模式 | `v0X_整理` | 中 | 結構化轉錄，不應改劇 |
| `DRAFT_TRIAL` | 試寫模式 | `v0X_試寫` | 中 | 多版本草稿，需收斂 |
| `EXPERIMENTAL` | 探索模式 / 破格模式 | `v0X_探索` / `v0X_破格` | 低 | 高自由度，QA 須走 `REVIEW_REQUIRED` |
| `CONVERGENCE` | 收斂模式 | `v0X_收斂` | 高 | 已整合多版本，準備 QA |
| `QA_REVIEW` | QA 模式 | `v0X_QA中` | 很高 | QA 進行中，未判定 |

Mode Tag 不等於 Lifecycle Status：

- `EXPERIMENTAL` 版本的 Lifecycle Status 通常為 `DRAFT` 或 `ALT`，不會直接是 `FINAL`。
- `CONVERGENCE` 版本的 Lifecycle Status 通常為 `REVIEW`。
- `QA_PASS` / `FINAL` / `LOCKED` 不是 Mode Tag，而是 Lifecycle Status。

---

## 層 3：QA Decision / QA 判定結果

QA 報告（09_a/b/c/d/f/g/h/i；v0.3 / D-043 — 原 09_a–09_d 範圍 supersede）的判定值，**僅在 QA 報告中使用**，不混入版本 Status，也不直接改變 Mode Tag：

```text
PASS / FAIL / REVIEW_REQUIRED / HUMAN_ARBITRATION_REQUIRED
```

| QA Decision | 意義 | 對 Lifecycle Status 的影響 |
|---|---|---|
| `PASS` | 該版本通過此項 QA | 滿足進入 `QA_PASS` 的必要條件之一 |
| `FAIL` | 該版本未通過此項 QA | 必須返修，不得升 `QA_PASS` |
| `REVIEW_REQUIRED` | 該版本需進一步檢視 | 不得自動升 `QA_PASS`；需另行裁決 |
| `HUMAN_ARBITRATION_REQUIRED` | 必須由人類裁決 | 不得自動升 `QA_PASS`；裁決結果記入 09_e |

特別規則：

- Mode Tag 為 `EXPERIMENTAL`（即 `_破格` / `_探索`）的版本，QA 不得自動 `PASS` 或 `FAIL`，**一律標記為 `REVIEW_REQUIRED` 或 `HUMAN_ARBITRATION_REQUIRED`**。詳見「§ 破格版本處理規則（REVIEW_REQUIRED）」。
- 在 09_d 資訊控制 QA 中，`EXPERIMENTAL` 版本不得因為是破格而放行；資訊提前揭露的破格必須走 `HUMAN_ARBITRATION_REQUIRED`。

# 模式系統承接表

> v0.2 新增。本文件對 10 種模式的具體承接：

| 模式 | 限制強度 | 本文件對此模式的具體承接 |
|---|---|---|
| 診斷模式 | 低 | 不產生版本檔；診斷報告另存於專案指定報告位置，本文件不直接承接 |
| 整理模式 | 中 | 可建立 `v0X_整理` 標記版本；Mode Tag = `ORGANIZED`；不可標記 `QA_PASS` |
| 探索模式 | 低 | 多版本 `v0X_探索A/B/C` 並列保留，不互相覆寫；Mode Tag = `EXPERIMENTAL` |
| 試寫模式 | 中 | 標記 `v0X_試寫`；Mode Tag = `DRAFT_TRIAL`；需備註場景 ID 與動機 |
| 破格模式 | 低 | 標記 `v0X_破格`；Mode Tag = `EXPERIMENTAL`；必須附「為何破格」一句話；後續 QA 一律進 `REVIEW_REQUIRED` 或 `HUMAN_ARBITRATION_REQUIRED` |
| 收斂模式 | 高 | 標記 `v0X_收斂`；Mode Tag = `CONVERGENCE`；可整合多版本為主候選 |
| QA 模式 | 很高 | 通過 → Lifecycle Status `QA_PASS`；不通過不可升級；判定結果存於 QA 報告 |
| 交稿模式 | 最高 | 升 `FINAL`；任何後續修改必須開新版本；FINAL 前例外確認須完成 |
| Codex 執行模式 | 依任務 | 不直接動版本 Status / Mode Tag；只記錄 commit SHA 對應；目前實作端為 Claude，見「§ Repo Execution / Claude 權限」 |
| 人類裁決模式 | 不適用 | 可越級任意升降版本；必須在 09_e 變更紀錄留記裁決者、理由與後續限制 |

# 7. 版本文件頭格式

| 版本 | 方向 |

# 9. 修稿規則

## 9.1 小修與大修

| 修稿類型 | 定義 | 版本處理 |
|---|---|---|
| 小修 | 詞句、節奏、少量禁用詞修正 | v01 → v01.1 |
| 中修 | 局部重寫、角色語氣調整 | v01 → v02 |
| 大修 | 場景功能、資訊揭露、關係狀態重構 | 重新生成 v02 或更新任務包後 v01 |

---

## 9.2 需要先更新任務包的情況

若修稿涉及以下內容，必須先更新任務包，而不是只改台詞：

- 場景功能改變
- 出場角色改變
- 關係狀態改變
- 資訊揭露改變
- 稱呼解禁改變
- 角色弧線階段改變
- 伏筆放置或回收改變
- 玩家選項或分支改變

# 13. DEPRECATED 規則

## 13.1 棄用原因

版本可標記為 DEPRECATED 的原因：

- 任務包重寫
- 場景刪除
- 角色聲線變更
- 資訊揭露時點改變
- 關係狀態改變
- 台詞品質不通過
- 被新版本取代
- 已不符合最新禁用詞規範

---

## 13.2 棄用版處理

DEPRECATED 版本不得刪除到無法追蹤，除非專案另有清理策略。  
應保留：

- 棄用原因
- 被哪個版本取代
- 是否仍可作參考
- 是否有可保留句子

# 破格版本處理規則（REVIEW_REQUIRED）

> v0.2 新增。

> **核心原則：破格句不是違規豁免，而是機械刪除豁免。**

Mode Tag 為 `EXPERIMENTAL`（`_破格` 或 `_探索` 來源）的版本，後續 QA 不得自動 `PASS`，也不得自動 `FAIL`，必須一律標記為：

```text
REVIEW_REQUIRED 或 HUMAN_ARBITRATION_REQUIRED
```

QA 報告（09_a/b/c/d/f/g/h/i；v0.3 / D-043 — 原 09_a–09_d 範圍 supersede）中必須列出五欄：

| 破格內容 | 命中的風險維度 | 可保留理由 | 修正建議 | 需人類裁決原因 |
|---|---|---|---|---|
|  | AI 味 / 聲線 / 禁用詞 / 資訊控制 / 其他 |  |  |  |
|  |  |  |  |  |

特別規則：

- 09_d 資訊控制檢查不得因為是 `EXPERIMENTAL` 而放行；資訊提前揭露的破格必須走 `HUMAN_ARBITRATION_REQUIRED`。
- QA 階段標記為 `REVIEW_REQUIRED` / `HUMAN_ARBITRATION_REQUIRED` 的項目，進入定稿前須整理至 09_e §8.3 的人類裁決紀錄表。
- 「不自動 PASS」不等於「自動退回」；版本 Lifecycle Status 在被裁決前維持原狀（通常為 `DRAFT` 或 `ALT`）。

# 15. Codex 使用規則

> **註（2026-05-08，v0.2）**：本節原名「Codex 使用規則」。目前 repo 落地與檔案修改端已改為 **Claude**；條文中的「Codex」可視為「Repo Execution 端」之歷史代稱。新增的 Claude 權限規則請見後續「§ Repo Execution / Claude 權限」一節。

當 Codex 修改或管理台詞版本時，應遵守：

1. 不得擅自覆蓋 FINAL 或 LOCKED 檔案。
2. 不得直接把 DRAFT 標記為 FINAL。
3. 不得在沒有 QA 的情況下標記 QA_PASS。
4. 若修改台詞內容，必須更新版本號或變更紀錄。
5. 若修改涉及場景功能、資訊、關係或弧線，必須先提醒更新任務包。
6. 若發現檔名不符合規範，應建議更名，不得擅自大量重命名 LOCKED 檔案。
7. 若有多版本，應先比較再整合，不得任意刪除。
8. 若棄用版本，應標記 DEPRECATED 並記錄原因。
9. 修改後必須列出影響範圍。

# 18. 最終原則

台詞版本管理的目標，不是增加行政負擔，而是保護高品質台詞不在反覆修稿中失真。

好的版本管理應該讓任何人或 AI 都能快速知道：

- 哪一版可以看
- 哪一版可以改
- 哪一版不能動
- 哪一版已經棄用
- 哪一版通過 QA
- 哪一版是正式交稿
- 每一次修改為什麼發生

若版本無法追蹤，台詞品質就無法穩定。  
商業級長篇劇本尤其不能依靠「我記得那版比較好」來管理台詞。
