# 1. 文件目的

本文件用於規範每一份實際生成出來的台詞檔案應如何保存。

它解決的問題包括：

- 台詞正文與設計註解混在一起
- 不知道台詞依據哪份任務包生成
- 不知道本版本是否通過 QA
- 玩家選項與分支格式不一致
- 修改後沒有變更紀錄
- 不知道哪些句子是 final，哪些仍是草稿
- 後續 Codex 無法安全修稿
- 人類審稿時無法快速理解本場功能

本文件不是台詞生成規則本身。  
台詞生成前應使用：

- `07_scene_tasks/07_a_單場台詞任務包模板.md`

台詞版本管理應依照：

- `08_dialogue_outputs/08_a_台詞版本管理規範.md`

# 2. 使用原則

## 2.1 台詞檔案必須可獨立閱讀

每份台詞檔案至少要讓讀者知道：

- 這是哪一場戲
- 這是第幾版
- 目前狀態是什麼
- 來源任務包是哪份
- 本場功能是什麼
- 哪些角色出場
- 正文從哪裡開始
- 哪些內容是註解，不是正式台詞
- 是否通過 QA
- 是否可作為正式劇本使用

---

## 2.2 正文與註解必須分離

正式台詞正文應清楚獨立。  
設計註解、QA 註解、修改建議、未採用版本不得混入 final 正文。

可使用區塊區分：

- 台詞正文
- 玩家選項
- 分支內容
- 動作 / 演出註記
- 設計註解
- QA 狀態
- 變更紀錄

若檔案是交稿版，可只保留乾淨正文與必要元資料。

---

## 2.3 台詞檔案不應補寫設定

生成台詞檔案不得擅自新增重大設定。  
若台詞中需要暫定設定，應標記：

```md
<!-- DRAFT_SETTING: 此處為暫定設定，需回寫上游文件確認 -->
```

正式台詞檔案不得把未確認設定當成 LOCKED 設定。

```text

```

# 7. 台詞正文格式

## 7.1 基本格式

```md
## 台詞正文

角色A：台詞。
角色B：台詞。
角色A：台詞。
```

## 7.2 動作與停頓格式

可使用以下格式之一，依專案統一：

```md
角色A：（看向門口）台詞。
角色B：……台詞。
角色A：（停頓）台詞。
```

或：

```md
[角色A看向門口]
角色A：台詞。
```

需在專案中統一，不得同一份文件混用多種動作格式。

## 8.1 玩家選項限制

玩家選項文字不得：

- 透露玩家尚未知道的資訊
- 替角色說出不該說的真相
- 把道德選擇標成正確答案
- 用作者視角整理劇情
- 比角色台詞更像旁白

# 17. Codex 使用規則

當 Codex 建立或修改台詞檔案時，應遵守：

1. 必須使用本文件定義的文件頭。
2. 必須記錄來源任務包。
3. 不得擅自覆蓋 FINAL 或 LOCKED 台詞檔案。
4. 不得把設計註解混入正式正文。
5. 若新增玩家選項或分支，需記錄資訊差異與後續影響。
6. 若修改台詞內容，必須更新版本號或變更紀錄。
7. 若台詞使用暫定設定，必須標記 DRAFT_SETTING。
8. 若 QA 未完成，不得標記為 FINAL。
9. 若版本棄用，需填寫棄用紀錄。
10. 修改後必須列出影響範圍。

# 20. 最終原則

生成台詞檔案的目標，不只是保存一段對話，而是保存一段可追蹤、可審核、可修訂、可交付的劇本文本。

一份合格的台詞檔案應該讓後續任何人或 AI 都能清楚知道：

- 這段台詞從哪裡來
- 它要完成什麼功能
- 它目前是不是正式版本
- 它是否通過 QA
- 它是否可被修改
- 修改時要小心什麼

台詞正文可以是藝術文本，但台詞檔案必須是工程化資產。  
商業級長篇劇本需要的不是一次寫出漂亮對話，而是能持續產出、追蹤、修正與保護品質的台詞系統。
