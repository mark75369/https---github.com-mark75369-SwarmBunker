# 1. 文件目的

本文件用於將章節結構拆解為可執行的場景清單。

`05B_章節結構模板.md` 定義章節層級；本文件則管理章節內每一場戲的索引與功能，讓後續能建立單場台詞任務包與台詞腳本。

本文件回答：

- 這一章有哪些場景？
- 每場戲為什麼存在？
- 每場戲從什麼狀態開始？
- 每場戲到什麼狀態結束？
- 每場戲出場哪些角色？
- 每場戲推動主線、角色、關係或資訊的哪一部分？
- 哪些場景需要台詞任務包？
- 哪些場景只需要環境敘事、系統文字或任務文本？
- 哪些場景可合併、刪除或移動？

場景索引不是台詞文件。  
它是台詞生成前的場景管理層。

# 3. 場景管理原則

## 3.1 每場戲必須有功能

一場戲至少應推動以下其中一項：

- 主線
- 任務
- 角色弧線
- 角色關係
- 資訊揭露
- 伏筆放置或回收
- 世界觀壓力
- 玩家選擇
- 情緒節奏
- 玩法目標

若一場戲沒有推動任何項目，應考慮刪除、合併或改為更輕量的文本形式。

---

## 3.2 場景必須有進場狀態與出場狀態

每場戲都應明確定義：

- 玩家進場時知道什麼
- 玩家出場時知道什麼
- 角色進場時想要什麼
- 角色出場時失去或得到什麼
- 關係是否改變
- 資訊是否改變
- 任務是否推進

若進場與出場狀態沒有差異，該場景可能缺乏戲劇功能。

---

## 3.3 場景不是台詞長度單位

一場戲不等於一段台詞。  
場景可以包含：

- 對話
- 行動
- 戰鬥
- 探索
- 任務選擇
- 環境敘事
- UI 文本
- 過場演出
- 玩家選項
- 無台詞沉默場景

因此本文件要先判斷「場景需要什麼文本形式」，而不是直接要求每場都產出台詞。

# 4. 場景ID規則

## 4.1 建議格式

```text
CH01_S01
CH01_S02
CH01_S03
CH02_S01
CH02_S02
```

格式說明：

```text
CH = Chapter
S = Scene
CH01_S03 = 第 1 章第 3 場
```

若有分支，可使用：

```text
CH01_S03A
CH01_S03B
CH01_S03C
```

若有可選支線，可使用：

```text
CH01_SIDE01_S01
CH01_SIDE01_S02
```

---

## 4.2 場景ID穩定性

場景ID一旦建立，不應因場景改名而改變。  
若場景被刪除，應標記為 DEPRECATED，而不是重用ID。

# 19. Codex 使用規則

當 Codex 修改或使用本文件時，應遵守：

1. 不得擅自修改 LOCKED 場景的核心功能、資訊與出場狀態。
2. 不得重用已 DEPRECATED 的場景ID。
3. 若新增場景，必須填寫場景ID、場景功能、進出狀態與是否需要任務包。
4. 若刪除或合併場景，需列出受影響的主線、角色弧線、資訊揭露、伏筆與任務包。
5. 若場景涉及重要台詞，需提醒建立 `07A_單場台詞任務包模板.md` 格式的任務包。
6. 若場景改變資訊揭露，需提醒更新 `05D_資訊揭露表.md`。
7. 若場景改變伏筆，需提醒更新 `05E_伏筆與回收表.md`。
8. 若場景改變關係，需提醒更新 `04A_角色關係矩陣.md` 與 `04B_關係變化時間線.md`。
9. 若場景改變角色弧線，需提醒更新 `05C_角色弧線表.md`。
10. 修改後必須列出影響範圍。

# 21. 最終原則

場景索引的目標，是讓每場戲在進入台詞生成前就已經知道自己的責任。

好的場景索引應該能回答：

- 這場戲為什麼存在？
- 這場戲改變了什麼？
- 這場戲不能說什麼？
- 這場戲需要什麼文本？
- 這場戲是否真的需要台詞？
- 這場戲之後，下一場為什麼必須發生？

若一場戲只是讓角色交換資訊，而沒有改變任務、角色、關係、世界壓力或玩家認知，它還不是成熟的場景。
