# 1. 文件目的

本文件用於建立定稿變更紀錄流程。

長篇遊戲劇本在多輪 AI / Codex / 人類協作後，最容易出現以下問題：

- 不知道 final 版相對前一版改了什麼
- 不知道為什麼改
- 不知道是 AI 改的、人類改的，還是混合修改
- 不知道修改是否通過 QA
- 不知道修改是否影響角色聲線、資訊揭露、關係時間線或伏筆
- 不知道是否需要回寫上游文件
- 不知道某句 final 台詞是否引入新設定
- 不知道 locked 版是否可被修改
- 舊任務包與新台詞互相矛盾

本文件的目的，是讓每次定稿都能被追蹤、驗證與回溯。

定稿變更紀錄不是額外負擔，而是保護商業級劇本品質的必要工程記錄。

# 3. 使用時機

## 3.1 必須建立變更紀錄的情況

以下情況必須建立或更新本文件格式的紀錄：

- 台詞版本從 REVIEW 進入 QA_PASS
- 台詞版本從 QA_PASS 進入 FINAL
- 台詞版本從 FINAL 進入 LOCKED
- FINAL 版需要返修
- LOCKED 版需要建立新版
- 任務包內容更新後重新生成台詞
- QA 指出重大問題並完成修正
- 人類對 AI 生成台詞做了重要改寫
- QA 報告標記需要人類裁決
- 人類決定保留違反通用規則但符合作品風格的內容
- 收斂或定稿時保留破格句、破格亮點或不安全但有效的語氣
- FINAL 前仍存在任何例外、保留、風險註記或人類裁決項
- LOCKED 前需要確認例外已被正式接受或移除
- 台詞修改導致上游設定需要更新
- 台詞修改新增稱呼、伏筆、角色關係變化或資訊揭露
- 台詞版本被標記為 DEPRECATED

---

## 3.2 可簡化紀錄的情況

以下情況可使用簡化紀錄：

- 修正錯字
- 調整標點
- 統一角色名格式
- 統一換行或 Markdown 格式
- 不影響語意、聲線、資訊或關係的小幅格式整理

即使是簡化紀錄，也應記錄日期、修改者與修改範圍。

# 4. 定稿狀態定義

| 狀態 | 意義 | 修改規則 |
|---|---|---|
| REVIEW | 待審版 | 可修改，需保留版本紀錄 |
| QA_PASS | 通過必要 QA | 可作 final 候選，不等於 final |
| FINAL | 人類確認定稿 | 不得直接覆蓋，修改需新版本或變更紀錄 |
| LOCKED | 已納入正式劇本 | 不得擅自修改，若需修改必須建立新版本 |
| DEPRECATED | 已棄用 | 不得正式引用，需記錄取代版本 |

## 8.3.4 與 09_a–d QA 階段的承接

> v0.2 新增。

09_a–09_d 的 QA 報告若標記為 `REVIEW_REQUIRED` 或 `HUMAN_ARBITRATION_REQUIRED` 的項目，**應在進入定稿前整理至本節 §8.3.1–§8.3.3 的裁決紀錄表**。

承接規則：

- 09_a–d 的 5 欄表（破格內容 / 風險維度 / 可保留理由 / 修正建議 / 需人類裁決原因）用於 QA 階段的判定。
- 09_e 的 6 欄表（位置 / 破格句 / 原破格方向 / 收斂後處理 / 保留理由 / 風險 / 人類是否確認）用於定稿 / 裁決階段的紀錄。
- 兩者**不互相取代**；定稿階段必須將 QA 階段的 `HUMAN_ARBITRATION_REQUIRED` 項目逐項轉錄至本節並完成裁決，才可進入 FINAL。

# 16. Codex 使用規則

> **註（2026-05-08，v0.2）**：本節原名「Codex 使用規則」。目前 repo 落地與檔案修改端已改為 **Claude**；條文中的「Codex」可視為「Repo Execution 端」之歷史代稱。新增的 Claude 權限規則請見後續「§ Repo Execution / Claude 權限」一節。

當 Codex 建立或修改定稿變更紀錄時，應遵守：

1. 不得在 QA 未完成時標記 FINAL。
2. 不得在 FINAL 未確認時標記 LOCKED。
3. 不得擅自覆蓋 LOCKED 版台詞。
4. 若修改 final 台詞，必須建立新版本或記錄明確變更原因。
5. 必須列出上游文件是否需要回寫。
6. 必須列出影響範圍。
7. 若發現 final 台詞與任務包衝突，必須回報，不得直接忽略。
8. 若人類已確認保留某句，不得因 AI 偏好自行刪除。
9. 若棄用版本，必須填寫 DEPRECATED 紀錄。
10. 修改後必須列出本次處理結果。
11. 若 QA 標記人類裁決項，必須記錄裁決者、裁決結果與保留理由。
12. 若違規但保留，必須明確列出違反規則與可保留原因，不得只寫「風格需要」。
13. 若保留破格句或破格亮點，必須記錄原破格方向、收斂後處理與人類確認狀態。
14. 進入 FINAL 前，必須完成 FINAL 前例外確認。
15. 進入 LOCKED 前，必須完成 LOCKED 前例外確認。

# Repo Execution / Claude 權限

> v0.2 新增。repo 落地端已從 Codex 移到 Claude。本節用於明確 Claude 在定稿變更紀錄流程中的權限分層。詳細執行規則沿用 §16「Codex 使用規則」（歷史代稱），本節為其在 Claude 實作端的權限補充。

> **定稿層特別提醒**：09_e 是人類裁決與 final / locked 例外紀錄的核心文件。Claude 在本流程中的權限上限**比其他文件更嚴格**：可協助建立草稿與整理裁決項，但**不得替人類完成審美、canon、`FINAL`、`LOCKED` 的決定**。

## Claude 可寫

- 09_e 變更紀錄草稿（含逐項變更紀錄、QA 結果整合、上游回寫檢查清單）
- 09_e §8.3 的人類裁決草稿（將 09_a–d 的 `HUMAN_ARBITRATION_REQUIRED` 項目轉錄成可裁決條目）
- DEPRECATED 紀錄草稿

## Claude 可建議但不可自行決定

- 版本 tag（Lifecycle Status / Mode Tag）變更
- `FINAL` 狀態
- `LOCKED` 狀態
- 破格句是否最終保留
- canon 是否成立
- 上游文件是否需回寫（提建議；實際寫回需人類確認）

## Claude 不可自行做

- 把 `REVIEW` 升 `FINAL`
- 把 `FINAL` 升 `LOCKED`
- 把破格句直接刪除或簽字保留
- 把台詞內容直接回寫 Bible（02–06）
- 把推測寫成 canon
- 修改已確認 `FINAL` / `LOCKED` 紀錄
- 推翻人類裁決結果
- 在 §8.3 表格中以 Claude 身份填入「裁決者」（裁決者必須是人類）

# 19. 最終原則

定稿變更紀錄的核心，不是記錄「改過了」，而是記錄：

- 改了什麼
- 為什麼改
- 誰改的
- 依據什麼改
- 是否通過 QA
- 是否有人類裁決
- 是否違規但保留
- 是否保留破格句，以及為什麼保留
- final / locked 前例外是否已確認
- 是否需要回寫設定
- 是否影響後續場景
- 是否真的可以交付

商業級長篇遊戲劇本中，台詞不是一次性文字，而是會被反覆修改、接續、翻譯、配音、實作與維護的資產。

如果 final 無法追蹤，就不是真正的 final。  
如果 locked 可以被隨意覆蓋，就不是真正的 locked。

定稿變更紀錄是讓創作判斷變成可維護資產的最後一道保護層。
