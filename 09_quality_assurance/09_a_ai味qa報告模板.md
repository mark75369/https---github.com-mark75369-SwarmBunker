# 1. 文件目的

本文件用於建立「AI 味」的可檢查 QA 流程。

AI 味不是單純指文字不好，而是指台詞出現以下問題：

- 說話過於平均、禮貌、完整
- 每個角色都像同一個成熟旁白
- 台詞過度解釋心理
- 台詞過度總結主題
- 句型過於工整
- 轉折太乾淨
- 情緒太直接說出
- 衝突太快和解
- 世界觀被角色朗讀
- 所有角色都使用同一批詞
- 高頻詞與套語反覆出現
- 沒有角色身份、階級、關係與場景壓力

本文件的目的，是讓 AI 味檢查從主觀感覺變成具體可追蹤的審稿項目。

# 3. AI 味判定原則

## 3.1 AI 味不等於語句流暢

流暢本身不是問題。  
問題是角色在不該流暢時過度流暢。

高風險情況：

- 角色在崩潰時仍能完整自我分析
- 角色在危急時刻講出漂亮長句
- 角色剛衝突完就成熟和解
- 角色用太乾淨的轉折整理情緒
- 角色把複雜關係說成抽象結論

---

## 3.2 AI 味通常來自過度補洞

當任務包、角色卡、關係資料不足時，AI 會自動用通用劇本語氣補洞。

常見補洞方式：

- 用心理總結補角色動機
- 用主題句補情緒重量
- 用對稱句補文學感
- 用正式名詞補世界觀
- 用成熟道歉補關係進展
- 用「我明白了」補角色成長
- 用「你不是一個人」補安慰功能

因此檢查 AI 味時，也要檢查是否是上游任務包資訊不足造成。

---

## 3.3 好台詞不一定漂亮

商業級台詞應優先滿足：

1. 符合角色
2. 符合關係
3. 符合場景壓力
4. 符合資訊控制
5. 完成場景功能
6. 有可讀性
7. 具備情緒或潛台詞

漂亮句子排在後面。  
一句不好看的句子，如果符合角色、關係與壓力，可能比漂亮句更有價值。

---

## 3.4 反 AI 味不是反風格

反 AI 味檢查的目的不是消滅漂亮句、宣言句、情緒直說、重複句或風格化語言。

這些元素在某些作品中可能是缺陷，在另一些作品中可能是風格核心。

因此本文件所有檢查項都應理解為：

```text
風險提示，不等於絕對禁用。
```

除非某詞、句型或表達已在 `02_vocabulary/02_c_禁用詞與慎用詞表.md` 或單場任務包中明確標為禁用，否則 QA 不得自動刪除。

---

## 3.5 高風險句型示例，不等於禁用句

以下句型在通用情況下高風險，但不代表永遠禁用：

- 我會守護你。
- 你不是一個人。
- 這就是我的選擇。
- 我終於明白。
- 真正重要的是。
- 我不想失去你。
- 我一直都相信你。
- 你一定要活下去。
- 這就是我們的選擇。
- 這就是我們的命運。

這些句型應根據以下條件判斷：

| 判斷項目 | 問題 |
|---|---|
| 作品風格 | 本作品是否允許宣言式、誓言式、情緒直說？ |
| 角色聲線 | 這個角色是否會用這種句型？ |
| 關係階段 | 此時兩人的關係是否允許說到這麼滿？ |
| 弧線階段 | 角色是否已經能理解或承認這件事？ |
| 場景壓力 | 場景是否支撐這種表達？ |
| 代價 | 這句話是否有行動或代價支撐？ |
| 重複度 | 同類句型是否已過度使用？ |

---

## 3.6 可保留的不規則亮點

反 AI 味 QA 必須加入「可保留亮點」檢查。

```md
## 可保留亮點

| 位置 | 內容 | 為何值得保留 | 是否違反規則 | 建議處理 |
|---|---|---|---|---|
```

可保留亮點包括：

- 不漂亮但很像角色的句子。
- 語法不完整但符合情緒的句子。
- 看似突兀但能形成記憶點的反應。
- 重複但有角色執念感的詞。
- 粗糙但符合階級、職業、年齡的說法。
- 不完全合規但符合本作品美學的句子。

---

## 3.7 不建議機械修正項

```md
## 不建議機械修正項

| 位置 | 內容 | 為何不應機械修正 | 需誰確認 |
|---|---|---|---|
```

以下情況不應由 AI 自行修掉：

- 人類明確指定保留的句子。
- 具有強烈角色感但語法不標準的句子。
- 本作品風格需要的宣言式台詞。
- 角色刻意說謊、表演或自我欺騙的句子。
- 為回收伏筆而重複的句子。
- 為形成儀式感而使用的固定句。

---

## 3.8 需要人類裁決項

```md
## 需要人類裁決項

| 項目 | 內容 | 需要人類決定 | 原因 |
|---|---|---|---|
```

若某句違反通用規則但符合角色、作品風格或資訊設計，應標記為人類裁決，不得自動刪除。

# 破格版本處理規則（REVIEW_REQUIRED）

> v0.2 新增。**破格句不是違規豁免，而是機械刪除豁免。**

Mode Tag 為 `EXPERIMENTAL`（`_破格` 或 `_探索` 來源）的版本，本文件 QA 不得自動 `PASS`，也不得自動 `FAIL`，必須一律標記為：

```text
REVIEW_REQUIRED 或 HUMAN_ARBITRATION_REQUIRED
```

QA 報告中必須列出五欄：

| 破格內容 | 命中的風險維度 | 可保留理由 | 修正建議 | 需人類裁決原因 |
|---|---|---|---|---|
|  | 通用成熟腔 / 心理總結 / 主題宣言 / 工整轉折 / 世界觀朗讀 / 詞彙重複 / 情緒模板 / 過度文藝 / 其他 |  |  |  |
|  |  |  |  |  |

AI 味專屬補充：

- AI 味破格可能是有生命力的不規則亮點（值得保留），也可能只是 AI 套語誤判（應修正）。
- 既有 §3.6「可保留亮點」、§3.7「不建議機械修正項」、§3.8「需要人類裁決項」三張小表**仍適用**，提供更細的處理指引；本節為跨模板的統一判定骨架。
- 本表內容須在進入定稿前整理至 09_e §8.3 的人類裁決紀錄表。

# 4. AI 味風險分類

## 4.1 主要類型

| 類型 | 說明 | 常見表現 |
|---|---|---|
| 通用成熟腔 | 每個角色都像理性成人 | 道歉完整、理解太快、會自我分析 |
| 心理總結腔 | 直接說出心理原因 | 我只是害怕、我一直在逃避 |
| 主題宣言腔 | 台詞替作者說主題 | 真正重要的是、這就是選擇的代價 |
| 工整轉折腔 | 句型過度對稱 | 不是A，而是B；我以為X，直到Y |
| 世界觀朗讀腔 | 角色像設定文件 | 長篇解釋歷史、制度、魔法規則 |
| 情緒模板腔 | 用固定安慰或告別句 | 你不是一個人、我不能失去你 |
| 詞彙均質化 | 所有角色用同一套詞 | 角色身份與階級消失 |
| 過度文藝腔 | 危急場景突然詩化 | 抽象比喻過多、意象不屬於角色 |
| 任務簡報腔 | NPC 像導覽或 UI | 直接告知目標、背景、風險 |
| 無潛台詞直說 | 所有情緒都說出來 | 我很生氣、我很難過、我在乎你 |

# 7. 句型 AI 味檢查

## 7.1 高風險句型

檢查是否出現以下句型或相似結構：

| 句型 | 是否出現 | 問題 | 處理 |
|---|---|---|---|
| 不是 A，而是 B |  | 工整、像總結 | 改為具體行動或角色式反駁 |
| 我以為 X，直到 Y |  | 回顧式心理整理 | 改為不完整承認或沉默 |
| 我終於明白 |  | 提前覺悟 | 改為行動或短句 |
| 真正重要的是 |  | 主題宣言 | 刪除或改成場景選擇 |
| 你不是一個人 |  | 通用安慰 | 改為角色專屬關心方式 |
| 我不能失去你 |  | 情緒直說 | 改為阻止、轉移、命令、物件 |
| 我一直都相信你 |  | 關係過滿 | 改為具體讓步或信任行動 |
| 這就是我的選擇 |  | 主題式決意 | 改為直接行動或更短反應 |
| 從今以後 |  | 結尾套語 | 刪除或改成未完成感 |
| 這一切都是為了 |  | 反派解釋腔 | 改為局部理由或行動證明 |

# 13. 修改建議格式

## 13.1 不直接重寫時

```md
## 修改建議

1. 第 X 行：問題是【問題類型】，建議改成【修正方向】，不要直接說【應避免內容】。
2. 第 X-Y 行：兩人太快互相理解，建議保留衝突，只讓其中一方退一步。
3. 第 X 行：此句像主題宣言，建議改成角色的具體行動或物件反應。
```

```md
## 局部重寫建議

原句：
角色A：

重寫方向：

建議版本：
角色A：
```

局部重寫應保留原場景功能，不得擅自改變資訊揭露、關係狀態或角色弧線。

# 15. Codex 使用規則

> **註（2026-05-08，v0.2）**：本節原名「Codex 使用規則」。目前 repo 落地與檔案修改端已改為 **Claude**；條文中的「Codex」可視為「Repo Execution 端」之歷史代稱。新增的 Claude 權限規則請見後續「§ Repo Execution / Claude 權限」一節。

當 Codex 使用本文件檢查台詞時，應遵守：

1. 不得只給模糊評語，如「有點 AI 味」。
2. 必須指出具體行號、問題類型與修正方向。
3. 不得在 QA 模式下擅自大幅改寫，除非使用者要求。
4. 若 AI 味來自任務包缺漏，應指出上游缺漏。
5. 若問題其實屬於聲線、資訊、禁用詞，應提醒使用對應 QA 模板。
6. 若台詞有可保留句，應標記，不得全盤否定。
7. 若建議重寫，應說明是因為場景功能、聲線、資訊或 AI 味哪一類問題。
8. 檢查後不得擅自標記 FINAL。

# Repo Execution / Claude 權限

> v0.2 新增。repo 落地端已從 Codex 移到 Claude。本節用於明確 Claude 在 AI 味 QA 流程中的權限分層。詳細執行規則沿用 §15「Codex 使用規則」（歷史代稱），本節為其在 Claude 實作端的權限補充。

## Claude 可寫

- 09_a AI 味 QA 報告草稿（含逐句問題標記、可保留句、需重寫句、高頻詞統計）
- 09_e 中對應的人類裁決草稿

## Claude 可建議但不可自行決定

- QA Decision 變更（`PASS` / `FAIL` / `REVIEW_REQUIRED` / `HUMAN_ARBITRATION_REQUIRED`）
- 是否升 Lifecycle Status（`REVIEW_CANDIDATE` 等）
- 破格 AI 味句是否最終保留
- `02_vocabulary/02_c_禁用詞與慎用詞表.md` 是否需要新增高頻詞

## Claude 不可自行做

- 把 `REVIEW` 升 `FINAL`
- 把 `FINAL` 升 `LOCKED`
- 把破格 AI 味句直接刪除或改寫
- 把台詞內容直接回寫高頻詞表 / Bible
- 修改已確認 `FINAL` / `LOCKED` 紀錄
- 推翻人類已下達的 AI 味裁決

# 17. 最終原則

AI 味 QA 的目的不是把所有台詞改得粗糙，而是把不屬於角色、不屬於場景、不屬於關係的通用漂亮句剔除。

好的 AI 味 QA 應該能區分：

- 這句話本身漂亮，但不是這個角色
- 這句話資訊正確，但不該在此時說
- 這句話情緒明確，但說得太滿
- 這句話有文學感，但不屬於此場景壓力
- 這句話流暢，但缺少人類對話中的錯位、遮掩與不完整

降低 AI 味不是追求故意破碎，而是讓台詞重新回到角色、關係、場景與代價之中。
