# 1. 文件目的

本文件用於建立資訊控制 QA 流程。

資訊控制檢查的核心問題是：

> 這個資訊，在這個時間點，能不能由這個角色、這種文本形式、這種方式出現？

本文件檢查範圍包括：

- 玩家此時是否知道太多
- 玩家此時是否知道太少
- 角色是否知道不該知道的資訊
- 角色是否說出知道但不能說的資訊
- NPC 是否使用作者視角
- 反派是否透露太多計畫或理念
- 玩家選項是否提前知道真相
- UI / 任務文本是否提前劇透
- 專有名詞是否提前出現
- 真名、稱呼、血緣詞是否提前解禁
- 本場必須揭露的資訊是否完成
- 本場只能暗示的資訊是否被說破
- 本場禁止揭露的資訊是否誤觸
- 伏筆與誤導是否公平

本文件是台詞進入 FINAL 前的必要 QA 之一。

# 3. 資訊控制判定原則

## 3.1 正確資訊也可能是錯誤台詞

一句台詞的內容即使符合設定，也可能因為出現時間錯誤而失敗。

常見失敗情況：

- 角色提前說出世界真相
- 玩家選項提前知道反派目的
- NPC 使用只有作者才知道的設定詞
- 反派在前期完整解釋自己的計畫
- 親密場景中過早說破關係秘密
- 任務描述提前暴露委託人謊言
- UI 文字直接標明選項後果

資訊正確不等於資訊可用。

---

## 3.2 角色知道不等於角色能說

角色可能知道某個資訊，但因以下原因不能說：

- 害怕
- 誓言
- 利益
- 被監視
- 不信任對方
- 不願承認
- 自我欺騙
- 不知道對方是否知道
- 說出後會改變關係
- 說出後會造成劇情過早崩塌

因此檢查時要分開判斷：

1. 角色是否知道？
2. 角色是否能說？
3. 角色是否有動機說？
4. 角色是否會用這種方式說？

---

## 3.3 區分自動刪除與人類裁決

資訊控制檢查應區分：

- **自動刪除**：明確提前劇透、角色不可能知道的資訊、玩家選項直接答案。
- **人類裁決**：若某句雖提前暗示但符合角色動機、作品節奏或資訊設計，應標記為人類裁決，不得自動刪除。

若難以判定，應標記為「需人類確認」。

---

## 3.4 暗示不是說破

本場若標記為「只能暗示」，不得把資訊用明確台詞說出。

暗示可以是：

- 稱呼停頓
- 角色迴避
- 物件出現
- 環境異常
- NPC 誤稱
- 反應過度
- 文件缺頁
- 任務結果矛盾
- 角色說半句後改口

說破則是：

- 直接說出真相
- 使用真名
- 使用正式名詞
- 說出因果關係
- 解釋前因後果
- 讓角色完整確認玩家猜測

# 破格版本處理規則（REVIEW_REQUIRED）

> v0.2 新增。**破格句不是違規豁免，而是機械刪除豁免。**

> **09_d 特例（最重要）**：資訊控制 QA 是 09 系列中風險最高的一項。Mode Tag 為 `EXPERIMENTAL`（`_破格` 或 `_探索` 來源）的版本，**若涉及資訊提前揭露、真相提前、名詞 / 稱呼提前、結局條件提前、反派目的提前**，一律走 `HUMAN_ARBITRATION_REQUIRED`，**不得只標記 `REVIEW_REQUIRED` 後放行**。

Mode Tag 為 `EXPERIMENTAL` 的版本，本文件 QA 不得自動 `PASS`，也不得自動 `FAIL`，必須一律標記為：

```text
REVIEW_REQUIRED 或 HUMAN_ARBITRATION_REQUIRED（資訊揭露相關必為後者）
```

QA 報告中必須列出五欄：

| 破格內容 | 命中的風險維度 | 可保留理由 | 修正建議 | 需人類裁決原因 |
|---|---|---|---|---|
|  | 真相提前 / 名詞提前 / 稱呼提前 / 因果提前 / 反派目的提前 / 結局條件提前 / 弧線理解提前 / 必須揭露缺漏 |  |  |  |
|  |  |  |  |  |

資訊控制專屬補充：

- 資訊控制破格是 09 系列中**最危險**的破格類型，可能提前劇透、破壞伏筆，或讓角色說出他不該知道的資訊。
- §3.3 既有「區分自動刪除與人類裁決」、§3.4「暗示不是說破」、§4.1 資訊狀態 enum (`HIDDEN` / `FORESHADOW` / `MISLEAD` / ...)、§7.1「提前揭露命中表」等仍適用，作為更細的處理指引；本節為跨模板的統一判定骨架。
- 本表內容須在進入定稿前整理至 09_e §8.3 的人類裁決紀錄表。

# 4. 資訊狀態對照

## 4.1 資訊狀態表

| 狀態 | 說明 | 台詞允許程度 |
|---|---|---|
| HIDDEN | 完全隱藏 | 不可說，不可明顯暗示 |
| FORESHADOW | 可伏筆 | 可用物件、反應、氣氛暗示 |
| MISLEAD | 可誤導 | 可提供錯誤或半真半假說法 |
| SUSPECT | 玩家可懷疑 | 可提供矛盾線索，但不可確認 |
| PARTIAL | 部分揭露 | 可說一部分，不可說完整因果 |
| REVEALED | 明確揭露 | 可由角色或事件說明 |
| PUBLIC | 世界內公開 | 多數角色可自然提及 |
| LOCKED_SECRET | 永久或後期秘密 | 主線中不可說破，除非使用者指定 |

# 17. Codex 使用規則

> **註（2026-05-08，v0.2）**：本節原名「Codex 使用規則」。目前 repo 落地與檔案修改端已改為 **Claude**；條文中的「Codex」可視為「Repo Execution 端」之歷史代稱。新增的 Claude 權限規則請見後續「§ Repo Execution / Claude 權限」一節。

當 Codex 使用本文件檢查台詞時，應遵守：

1. 不得只說「有劇透」，必須指出具體行號、資訊、目前狀態與應揭露時間。
2. 必須區分「提前揭露」與「必須揭露但缺漏」。
3. 必須檢查角色知道、角色能說、玩家知道、文本形式是否適合。
4. 不得為了修正劇透而擅自改變主線真相。
5. 不得為了補足缺漏資訊而新增未確認設定。
6. 若資訊狀態不明，應標記「需人類確認」，不得直接判定通過。
7. 若問題涉及禁用詞、稱呼或名詞解禁，需提醒同步使用 `09_quality_assurance/09_c_禁用詞檢查報告模板.md`。
8. 若問題涉及伏筆或回收，需提醒更新 `05_plot/05_e_伏筆與回收表.md`。
9. 通過資訊 QA 不代表可直接 FINAL，仍需其他 QA。

# Repo Execution / Claude 權限

> v0.2 新增。repo 落地端已從 Codex 移到 Claude。本節用於明確 Claude 在資訊控制 QA 流程中的權限分層。詳細執行規則沿用 §17「Codex 使用規則」（歷史代稱），本節為其在 Claude 實作端的權限補充。

> **資訊控制特別提醒**：09_d 是 09 系列中風險最高的 QA。Claude 在本文件下的權限上限**比其他 QA 模板更嚴格**：所有與資訊揭露相關的判定皆以「可建議但不可自行決定」為主。

## Claude 可寫

- 09_d 資訊控制 QA 報告草稿（含提前揭露命中表、缺漏資訊檢查、角色認知檢查）
- 09_e 中對應的人類裁決草稿（資訊提前揭露相關項目須清楚標明）

## Claude 可建議但不可自行決定

- QA Decision 變更（`PASS` / `FAIL` / `REVIEW_REQUIRED` / `HUMAN_ARBITRATION_REQUIRED`）
- 是否升 Lifecycle Status（`REVIEW_CANDIDATE` 等）
- 破格資訊是否最終保留
- `05_plot/05_d_資訊揭露表.md` / `05_plot/05_e_伏筆與回收表.md` 是否需要更新

## Claude 不可自行做

- 把 `REVIEW` 升 `FINAL`
- 把 `FINAL` 升 `LOCKED`
- 把破格資訊揭露直接刪除或改寫
- 把資訊提前揭露的破格判為 `PASS` 或單純 `REVIEW_REQUIRED`（必須走 `HUMAN_ARBITRATION_REQUIRED`）
- 把台詞內容直接回寫資訊揭露表 / 伏筆表 / Bible
- 修改已確認 `FINAL` / `LOCKED` 紀錄
- 推翻人類已下達的資訊裁決

# 19. 最終原則

資訊控制 QA 的目標，不是把劇情藏起來，而是讓資訊在最有力量的位置出現。

好的資訊控制應該讓玩家：

- 在該困惑時困惑
- 在該懷疑時懷疑
- 在該理解時理解
- 在該做選擇時知道足夠但不是全部

一段台詞若提前說出正確答案，可能會破壞數章的懸念、關係與伏筆。  
一段台詞若該揭露卻沒有揭露，也會讓玩家失去動機與公平理解。

因此資訊控制的核心不是「藏」或「說」，而是：

> 讓正確資訊，以正確形式，在正確時間，被正確的人說出或看見。
