## 1. 一句話總覽：這是什麼工具

這是一套 **商業級長篇遊戲劇本與台詞的製作系統**。

它的目標是：讓一個遊戲（或長篇故事）的世界觀、角色、劇情、台詞，能夠**有條理地一步步建立起來**，並且在過程中**自動維持一致性**——不會出現「同一個角色前後說話風格不一樣」「資訊在不該揭露的地方被講出來」「台詞讀起來很 AI、很僵硬」這類問題。

幾個關鍵特色，先記住這幾點，後面就好懂：

- **沒有傳統軟體那種圖形介面或資料庫**。所有內容都存成**純文字檔（Markdown 文件）**，用 Git 做版本控管。
- **操作方式是「打指令跟 AI 對話」**。你在 Claude Code（或類似的 AI 助手）裡輸入像 `/建立角色 小明` 這樣的指令，AI 就會照著固定流程跟你一問一答，最後幫你把檔案寫好。
- **流程是「先打地基，再蓋房子」**。一定要先有世界觀、角色、劇情大綱，才能生成台詞。不能跳過。
- **品質有「守門員」**。台詞寫完後要通過 8 道品質檢查才能定稿，而「定稿」這個最終決定永遠是人來按下，不是 AI 自己決定。

## 2. 核心理念：為什麼這樣設計

理解這 6 個理念，整套工具的所有設計都會變得理所當然。

1. **先建資料、後寫台詞。**
   不直接叫 AI「寫一場戲」，因為那樣很容易跑偏。要先把「這個世界長什麼樣、這個角色怎麼說話、這場戲要達成什麼」都先寫清楚，AI 才有足夠的依據去寫出對的台詞。

2. **用「概念」來追蹤進度，而不是用「檔案」。**
   一個角色的資料可能散在好幾個檔案裡（聲線卡、關係表、角色弧線……）。工具不是看「某個檔案寫完沒」，而是看「這個**角色**整體完成幾成」，把相關檔案的進度加權算出來。

3. **狀態跟流程分開記。**
   一份檔案「成熟到什麼程度」（草稿／審查中／定稿）、一場台詞「走到哪個製作階段」、「是哪一種試寫版本」、「跑了哪種品質檢查」——這幾件事各自獨立記錄，不會混在一起。

4. **改一個地方，自動找出受影響的地方。**
   每份檔案都記著「我依賴誰、誰依賴我」。所以當你改了世界規則，工具能立刻反查出「哪些角色、哪些台詞會受影響、需要重新檢查」。

5. **最終決定權留給人。**
   AI 負責提建議、出草稿、做診斷、列出問題。但「這份可以定稿了」「這個鎖定不再改」這種拍板，一定是人來做。

6. **內容永遠是純文字。**
   沒有專有格式、沒有資料庫綁死。你的劇本內容永遠是任何人都能打開讀的純文字檔，不怕工具壞了就拿不回來。

## 4. 第一大塊：資料庫（12 個編號資料夾）

這是工具的核心。資料夾用編號排序，**編號越小越「上游」（地基），越大越「下游」（成品）**。上游的資料是下游的依據。

下面逐層說明。每層我都標出「裝什麼、做什麼、餵給誰」。

### 00_protocol/ — 規則手冊層（12 份）
這一層不是劇本內容，而是**「做事的規則」**。它規範每個製作步驟該怎麼跑、該問什麼、該檢查什麼。可以想成「整套工具的使用說明書與標準作業流程」。

| 編號 | 名稱 | 管什麼 |
|---|---|---|
| 00_a | 台詞生產協議 | 定義什麼叫「好台詞」、常見毛病（AI 味、聲線混淆、提前劇透） |
| 00_b | 反 AI 味檢查表 | 怎麼判斷台詞「太像 AI 寫的」；上半部由各作品自己填，下半部是通用邏輯 |
| 00_c | 台詞輸出格式 | 不同階段（試寫／修稿／定稿）台詞檔該長什麼樣 |
| 00_d | 工作流總覽 | 「先做什麼、後做什麼」的全流程地圖 |
| 00_e | 世界觀創建協議 | `/建立世界觀` 怎麼跑 |
| 00_f | 角色創建協議 | `/建立角色` 怎麼跑 |
| 00_g | 大綱創建協議 | `/建立大綱` 怎麼跑 |
| 00_h | 細綱創建協議 | `/建立細綱` 怎麼跑 |
| 00_i | 專案初始化協議 | `/init-project` 怎麼跑（開新專案） |
| 00_j | 迭代協議 | 所有「修改既有內容」的指令共用的流程 |
| 00_k | 台詞生產流程協議 | 從任務包到定稿的下游流水線 |
| 00_l | 關係創建協議 | `/建立關係` 怎麼跑 |

### 01_world/ — 世界觀層（4 份）
裝這個作品的**世界設定**：世界規則、歷史、勢力、語言風格、整體文風。原則是「只記會影響劇情或角色的設定，純裝飾的背景不收」。
- `01_a` 世界觀總覽、`01_b` 世界語言規格、`01_c` 陣營與階級語言、`01_d` 文風樣本與指紋（作品整體的「文字口吻」）。
- **餵給**：詞彙、角色、劇情、台詞。是所有東西的最上游。

### 02_vocabulary/ — 詞彙層（3 份）
**統一管理用詞**：正式名詞、俗稱黑話、禁用詞。確保「一個東西只有一個正式名字」「角色不會講出他這時候還不該知道的詞」。
- `02_a` 專有名詞表、`02_b` 俗稱與黑話表、`02_c` 禁用詞與慎用詞表。
- **餵給**：角色台詞、劇情、品質檢查（禁用詞那關）。

### 03_characters/ — 角色層（模板 + main/minor/npc 子資料夾）
裝每個角色**怎麼說話、怎麼想、怎麼行動**的「聲線卡」。這是台詞生成時 AI 判斷「這個角色會怎麼講這句話」的依據。
- `03_a` 角色總表（索引）、`03_b` 主要角色聲線卡模板、`03_c` 次要角色／NPC 模板。
- main／minor／npc 子資料夾放實際角色檔（目前是空的範本庫狀態）。
- **餵給**：關係、角色弧線、任務包、台詞、品質檢查。

### 04_relationships/ — 關係層（2 份）
裝**兩個角色之間的關係**：誰對誰有權力、有沒有情感債、什麼話不能提、怎麼稱呼對方、關係隨劇情怎麼變。讓台詞反映「關係狀態」，而不只是角色本人。
- `04_a` 角色關係矩陣、`04_b` 關係變化時間線。

### 05_plot/ — 劇情層（5 份）
裝**整部作品的劇情骨架**，從大到小：
- `05_a` 主線大綱（一句話主線、核心衝突、主題）
- `05_b` 章節結構（拆成一章一章）
- `05_c` 角色弧線表（每個角色在每章的心理變化）
- `05_d` 資訊揭露表（什麼資訊該在第幾章才讓玩家知道）
- `05_e` 伏筆與回收表（哪裡埋、哪裡收）

### 06_scene_index/ — 場景索引層（1 份）
把章節再拆成**一場一場的戲**，每場標明：這場要幹嘛、開場什麼狀態、結束什麼狀態、誰出場。這是台詞生成的入口清單。

### 07_scene_tasks/ — 任務包層（1 份）
**「單場戲的完整出戰指令」**。把這一場需要的所有上下文（場景目的、角色當前聲線與關係、哪些資訊必說／只能暗示／禁說、角色弧線位置）壓縮成一份高密度文件，交給 AI 去寫台詞。這是上游與下游的交接點。

### 08_dialogue_outputs/ — 台詞成品層（2 份）
裝**實際生成的台詞**與它的版本管理規範。一場戲會經歷：試寫多版（v01A/B/C）→ 修稿 → 通過檢查 → 定稿。每一版都保留，可追溯。

### 09_quality_assurance/ — 品質檢查層（9 份）
台詞定稿前必須通過的**檢查報告模板**。其中 8 份是「必跑」，第 9 份是「定稿紀錄」：

| 檔 | 檢查什麼 |
|---|---|
| 09_a | AI 味（是否太工整、太像機器寫的） |
| 09_b | 角色聲線一致（拿掉名字還認得出是誰嗎） |
| 09_c | 禁用詞（有沒有講到不該講的詞） |
| 09_d | 資訊控制（有沒有提前劇透） |
| 09_f | 類型偏移（風格有沒有跑離作品調性） |
| 09_g | 節奏感（句子長短、呼吸感） |
| 09_h | 對話張力（你來我往的攻防夠不夠） |
| 09_i | 跨場一致（同一角色／伏筆在多場之間連不連得起來） |
| 09_e | 定稿變更紀錄（**不是檢查，是人類最終拍板的紀錄**） |

> 註：09_e 不算在 8 份必跑檢查內，它是「人來做最後定稿決定」時填的紀錄。

### 10_art_assets/ — 美術資產層（目前空）
預留給**搭配台詞的多媒體**：立繪（portrait）、背景（bg）、事件圖（cg）、音效（sfx）、音樂（bgm）、配音（voice）、介面文字（ui）。目前只有空資料夾結構，等實際製作時才放東西。

## 6. 邏輯實體：工具追蹤的「概念單位」

工具不是用「檔案」當追蹤單位，而是用「**實體**」——也就是一個個有意義的概念。每份檔案的開頭都記著「我貢獻了哪些實體、我依賴哪些實體」，靠這個織成一張依賴網。

> 下表是 `entity_type_registry`（型別總表）的**鏡像**，方便閱讀；型別集的**權威**永遠以 `entity_type_registry.yaml` 為準。目前共 11 種核心型別。Batch 6 加 REGISTRY-MIRROR marker，納入 `check_entity_type_consistency.py` 機器強制比對。

<!-- REGISTRY-MIRROR: entity-types -->
core 11 種：W-rules / W-language / W-style / V / C / R / P / CH / S / A / ORG
<!-- /REGISTRY-MIRROR -->

| 代號 | 中文 | 意思 | 例子 |
|---|---|---|---|
| W-rules | 世界規則 | 世界的核心設定法則 | 這個世界的魔法／生態規則 |
| W-language | 世界語言 | 世界的用語、方言、敬語規範 | 貴族跟平民講話的差別 |
| W-style | 文風樣本 | 作品整體的文字口吻、節奏 | 句子偏短、冷硬風 |
| V | 詞彙系統 | 名詞、禁用詞表 | 人名、地名、術語 |
| C-名字 | 角色 | 單一角色 | C-小明 |
| R-甲-乙 | 關係 | 兩角色間的關係 | R-小明-小華 |
| P | 主線 | 整部作品的劇情主線 | （一份） |
| CH-編號 | 章節 | 單一章節 | CH-01 |
| S-章-場 | 場景 | 單一場戲 | S-01-03 |
| A-類型-… | 美術資產 | 立繪／背景／配音等 | A-portrait-小明-生氣 |
| ORG-名字 | 組織 | 非人格的組織／制度／體系（不建聲線卡、不進台詞生成） | ORG-教會 |

**怎麼算完成度？** 每份檔案的成熟度換算成分數（草稿 25%、審查中 75%、定稿 100%），再依該檔案對某實體的「貢獻權重」加權平均。所以 `/進度` 能告訴你「小明這個角色整體完成 53%」，而不只是「某檔案還沒寫完」。

## 7. 第二大塊：指令技能（你能下哪些指令）

你透過「打指令跟 AI 對話」來操作工具。指令有英文原名與中文別名兩種寫法，**功能完全一樣**（中文別名只是轉接到英文主檔）。指令依製作階段分四組。

> 標示說明：**📖 純看（不改檔）** / **✍️ 會寫檔**。「會寫檔」的指令幾乎都會跑「診斷→探索→收斂→（你確認後）寫檔→驗證」五步，動筆前一定先給你看預覽。

### 第 A 組：開專案與查狀態（地基）
| 指令（中文別名） | 做什麼 | 改檔？ |
|---|---|---|
| `/init-project`（初始化專案） | 開一個新作品專案，建好初始結構 | ✍️ |
| `/create-world`（建立世界觀） | 建立世界規則、語言、詞彙 | ✍️ |
| `/status`（進度） | 列出每個實體完成幾成、建議下一步 | 📖 |
| `/check-gaps`（缺漏檢查） | 掃出待辦標記、缺漏的檔案、失效的視圖 | 📖 |

### 第 B 組：建立上游設定（世界、角色、劇情）
| 指令 | 做什麼 | 改檔？ |
|---|---|---|
| `/create-character`（建立角色） | 建一個角色的聲線卡 | ✍️ |
| `/create-relationship`（建立關係） | 建兩個角色之間的關係 | ✍️ |
| `/create-outline`（建立大綱） | 建主線大綱 | ✍️ |
| `/create-detailed-outline`（建立細綱） | 把主線拆成章節與場景清單 | ✍️ |

### 第 C 組：生產台詞（下游流水線）
| 指令 | 做什麼 | 改檔？ |
|---|---|---|
| `/scene-task`（場景任務包） | 為某一場戲組好「出戰指令」 | ✍️ |
| `/dialogue-write`（生成台詞） | 寫台詞，支援 4 種模式：試寫多版／破格實驗／收斂合併／單版打磨 | ✍️ |
| `/qa`（檢查） | 跑 8 份必跑品質檢查 | ✍️ |

### 第 D 組：查看、修改、匯出、整理（維護期工具）
**查看（純看，即時在對話裡組合給你）：**
- `/view-world`（查看世界觀）、`/view-character`（查看角色）、`/view-outline`（查看大綱）、`/view-detailed-outline`（查看細綱）— 全部 📖

**修改既有內容（會自動反查影響範圍）：**
- `/iterate-world`、`/iterate-character`、`/iterate-relationship`、`/iterate-outline`、`/iterate-detailed-outline`、`/iterate-scene`（迭代世界觀／角色／關係／大綱／細綱／場景）— 全部 ✍️
- 特色：改之前一定先列出「這個改動會牽動哪些角色、台詞、需不需要重跑檢查」。

**匯出（把查看結果寫成一份靜態檔，放在 `view/` 資料夾）：**
- `/export-world`、`/export-character`、`/export-outline`、`/export-detailed-outline`（匯出世界觀／角色／大綱／細綱）— 全部 ✍️（寫到 view/）

**診斷與整理：**
- `/diagnose`（診斷）📖 — 跨檔找問題：缺漏、矛盾、風格不合、待辦未清。
- `/integrate`（整理）✍️ — 把你的零散筆記整理進正式模板欄位，先給你看差異預覽、你逐項同意後才寫。

## 8. 三種「看內容」的方式（視圖三層）

同一份內容，工具提供三種不同的取用方式，各有適用場合：

1. **即時查看（`/view-*`）**：在對話裡當場把相關檔案組合起來給你看，不留檔。適合快速查閱、反覆調整。
2. **靜態匯出（`/export-*`）**：把組合結果寫成一份檔案放進 `view/` 資料夾，標記為「衍生檔（DERIVED）」。適合離線分享、保存快照。
3. **對外匯出（Layer 3，規劃中）**：產生結構化資料（JSON）＋給外部系統用的提示詞，用於系統整合或資料搬移。這一層的完整功能屬於未來工作。

## 9. 第三大塊：設計文件與支援工具

### 範本 vs 專案（很重要的一個區分）
- **範本庫（Template）**：就是你現在看到的這份快照。它只有骨架和規則，**沒有實際作品內容**。根目錄有一個 `.template_root` 標記檔，用來提醒「這是範本，不要直接在這上面開工」。
- **專案庫（Instance）**：把範本複製到新資料夾後，跑 `/init-project` 展開成的實際作品（例如某個具體遊戲）。每個專案各自獨立，但共用同一套規則。

### 設計文件（`_design/`，約 167 份）
這是工具的「藍圖與會議紀錄」，分兩類：
- **規格書（會一直被引用的權威文件）**：
  - `ARCHITECTURE.md`（整體架構）、`SPEC.md`（規格細節）、`DATA_FORMAT_SPEC.md`（檔案格式規範）、`UPSTREAM_DOWNSTREAM_SPEC.md`（上下游流程）、`INTEGRATION_CONTRACTS.md`（各部分之間的介面約定）、`DECISIONS_LOG.md`（所有設計決定的紀錄 D-001～D-055）、`REQUIREMENTS_LOCK.md`（最高需求基準）、`TASKS.md`（任務拆解）。
- **過程文件（開發歷程的紀錄，多為歷史參考）**：大量 `CODEX_*`（各輪審查報告）、`HANDOFF_*`（每一輪開發者的交接包）、`PHASE_*_COMPLETION_REPORT`（各階段完成報告）等。這些是「怎麼一路做到今天」的歷史，平常不需要讀。

### 設定登錄表（`*.yaml` 與 `_design/registries/`）
這些是「可調整的設定清單」，採三層設計：核心（鎖定不可改）＋ 專案擴充（你可加自訂項）。
- `entity_type_registry.yaml`：定義有哪些實體類型、命名規則。
- `qa_type_registry`（範本在 registries/）：定義有哪些品質檢查類型。
- `issue_type_registry`（範本在 registries/）：定義建立世界觀／角色等時，AI 會逐項問你的「議題清單」——你可以加自訂議題（例如恐怖類加「驚悚密度」）。
- `expected_entities.yaml`、`entity_exempt.yaml`：用來判斷「現在應該要有哪些實體」「哪些檔案豁免實體標記」。

### 檢查腳本（`scripts/`，3 支 Python，純讀取不改檔）
- `check_paths.py`：檢查檔案路徑引用有沒有寫錯／檔案缺漏。
- `check_headers.py`：檢查每份檔案開頭的 5 個必填欄位齊不齊。
- `parse_frontmatter.py`：核心解析工具，把所有檔案讀進來建索引，供前面兩支與各指令使用。

### 網頁檢視器（`_tools/frontend/`，約 31 份）
一個跑在本機的**網頁版檢視介面**（用 Python FastAPI 當伺服器，預設網址 127.0.0.1:8765）。
- 用途：用看板、場景列表、並排編輯、詞彙表等視覺化方式瀏覽專案。
- **目前狀態：早期骨架（alpha）**，能啟動、能看，但完整編輯功能還在開發中。
- 設計上是「**雙視窗工作流**」：網頁只負責「看」，真正的「改」還是在另一個視窗由 AI 助手執行。

### 使用手冊（`_user_manual/`，約 19 份）
寫給使用者（非開發者）的指南：10 分鐘快速上手、工具總覽、各組指令說明、資料結構、客製化、常見問題、疑難排解。另有 `08_workflows/` 子資料夾放 5 個情境教學（從零建專案、從手稿匯入、寫一場戲全流程、改設定怎麼處理、交付到下游）。其中 `skill_invocation_guide.md` 給那些不會自動讀指令的環境用，提供可直接複製貼上的指令範本。

### 其他
- `台詞/`（4 個 .tsv 檔）：示範用的台詞樣本資料（戰前／戰後對話），非正式內容。
- `archive/`（1 份）：已套用的歷史補丁，僅供回顧。
- `.gitignore`：排除暫存／工作目錄（含 `_sandbox/`，也就是這份快照所在的稽核工作區）。

## 10. 一份檔案是怎麼「成熟」的（狀態與流程）

工具用幾個獨立的標籤同時追蹤一份檔案／一場台詞的不同面向：

- **成熟度**：草稿（DRAFT）→ 審查中（REVIEW）→ 最終（FINAL）→ 鎖定（LOCKED）。另有已廢止（DEPRECATED）、已套用存檔（APPLIED）、自動衍生（DERIVED，只在 view/ 出現）。
- **製作階段（只給台詞）**：場景已索引 → 任務包草稿 → 台詞試寫 → 收斂 → 通過檢查 → 定稿 → 鎖定。
- **試寫模式（只給台詞）**：試寫多版／破格實驗／收斂合併／單版打磨等。
- **檢查類型（只給品質報告）**：前面講的 8 種檢查各有標籤。

這幾個面向互不干擾。例如一份試寫稿可以同時是「成熟度＝草稿、製作階段＝試寫、模式＝試寫多版」。

## 13. 一頁速查

- **這是什麼**：用純文字檔＋AI 指令做遊戲劇本與台詞的系統，重點是「先建設定、後寫台詞、全程保持一致、品質有關卡、人來定稿」。
- **內容在哪**：12 個編號資料夾，00 是規則、01～05 是設定與劇情、06～09 是台詞生產與檢查、10 是美術。
- **怎麼操作**：打指令（如 `/建立角色`），AI 照五步流程跟你對話、給預覽、你同意才寫檔。
- **怎麼保證一致**：靠每份檔案記的「依賴關係」自動反查影響範圍；靠 8 道品質檢查守門；靠「人類定稿」把關。
- **三大組成**：A 資料庫（內容）＋ B 指令技能（操作）＋ C 設計／支援（藍圖、腳本、前端、手冊、設定表）。
- **現況**：主體已封版（2026-05-23），目前在維護期，持續做稽核回補與前端開發；本資料夾是範本庫狀態的完整快照。
