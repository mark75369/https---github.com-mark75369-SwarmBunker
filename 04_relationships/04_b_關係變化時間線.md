# 1. 文件目的

本文件用於記錄角色關係如何隨劇情推進而變化。

`04A_角色關係矩陣.md` 定義角色之間的關係結構；本文件則定義這些關係在時間線上如何移動。

本文件的目的是避免：

- 角色突然信任
- 角色突然和解
- 角色突然改稱呼
- 角色突然告白或道歉
- 角色忘記前一章的衝突
- 關係破裂後下一場像沒事一樣
- 角色在尚未解禁的階段說出過度親密台詞
- AI 為了讓場景好看而提前完成關係變化

本文件應幫助 AI / Codex 判斷：

- 這兩個角色此時處於哪個關係階段？
- 哪些稱呼已解禁？
- 哪些話仍不能說？
- 這場戲會讓關係變近、變遠、停滯還是假性和解？
- 關係變化是否有足夠鋪墊？
- 關係變化是否影響後續台詞任務包？

# 3. 關係變化管理原則

## 3.1 關係變化必須有代價

角色關係不應無成本變好。

有效的關係變化通常伴隨：

- 讓步
- 犧牲
- 誤解
- 傷害
- 交換秘密
- 失去權力
- 共同犯錯
- 共同承擔後果
- 目睹對方脆弱
- 被迫合作
- 暫時停戰
- 不完整的原諒

若關係從敵對變成信任，必須有可追蹤節點。

---

## 3.2 不是所有變近都是真和解

關係變化可分為：

- 真正變近
- 表面變近
- 戰術合作
- 假性和解
- 暫時停戰
- 互相利用
- 情緒靠近但立場更遠
- 表面破裂但信任增加
- 一方變近，另一方退後

AI 生成台詞時不得把所有「衝突後」都寫成成熟和解。

---

## 3.3 稱呼變化是重要節點

稱呼、真名、暱稱、敬語、頭銜的變化通常代表關係移動。

稱呼變化應記錄：

- 何時第一次出現
- 誰先使用
- 對方是否察覺
- 玩家是否應察覺
- 是否可被角色點破
- 是否在後續穩定使用
- 是否在關係破裂後退回舊稱呼

稱呼變化不應隨意發生。

| 未相識 | 尚未建立直接關係 |

| 不完整修復 | 傷害仍在，但雙方願意繼續前進 |
| 最終選擇 | 關係在結局前完成不可逆定位 |

| 節點類型 | 說明 | 常見台詞影響 |

| 初遇 | 兩人第一次建立印象 | 稱呼正式、資訊少、試探多 |

| 假性和解 | 表面和解，問題未解 | 台詞客氣但有刺 |
| 真正靠近 | 行動承認關係 | 稱呼、沉默、讓步改變 |
| 告別 | 關係被迫暫停或終止 | 避免套語，常以小事遮掩 |
| 最終選擇 | 關係完成不可逆定位 | 不一定說破，但行動明確 |

## 11.2 信任不可跳躍規則

若信任從低變高，必須有：

- 具體事件
- 具體代價
- 一方冒險
- 一方讓渡資訊或權力
- 一方看見對方弱點
- 一方做出與過去相反行動

不得因單場對話說清楚就完成大幅信任跳躍。

## 12.3 不完整和解規則

和解不必完整。

不完整和解可以保留：

- 刺
- 沉默
- 不用道歉
- 稱呼不完全恢復
- 表面合作
- 情緒債未清
- 玩家知道雙方還在迴避

AI 不得自動把和解寫得太乾淨。

# 16. Codex 使用規則

當 Codex 修改或使用本文件時，應遵守：

1. 不得擅自改變 LOCKED 關係節點。
2. 不得自動讓角色關係和解、變近或告白。
3. 若新增關係節點，需填寫場景、事件、變化與台詞影響。
4. 若台詞中出現稱呼變化，需檢查是否已在本文件解禁。
5. 若任務包導致關係變化，需提醒更新本文件。
6. 若關係變化影響角色聲線，需提醒更新角色聲線卡。
7. 若關係真相揭露，需提醒更新 `05D_資訊揭露表.md`。
8. 若關係伏筆回收，需提醒更新 `05E_伏筆與回收表.md`。
9. 若關係變化涉及禁用詞或解禁詞，需提醒更新 `02C_禁用詞與慎用詞表.md`。
10. 修改後必須列出影響範圍。

# 19. 最終原則

角色關係不是靜態設定，而是隨著劇情被拉近、推遠、扭曲、破壞與修復的時間結構。

一段可信的關係變化應該讓玩家感覺到：

- 變化不是突然發生
- 稱呼有歷史
- 沉默有前因
- 道歉有代價
- 和解不一定乾淨
- 破裂不會下一場自動消失
- 親密不是靠台詞宣告，而是靠累積與回收

若角色在關係尚未到達某階段時說出過度親密、過度成熟、過度理解的台詞，應判定為關係時間線錯誤。

關係變化的核心不是「兩人說清楚了」，而是「兩人在付出代價後，仍然無法完全回到原本的位置」。
