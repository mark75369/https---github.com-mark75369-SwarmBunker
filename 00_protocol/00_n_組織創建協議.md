# 00_n 組織創建協議

## 1. 文件目的

本協議規範 `/create-org` / `/建立組織` 在組織實體創建過程中的執行流程。

本協議的目標是引導使用者建立 `ORG-<name>` 實體（組織 / 非人格反派 / 組織型對抗源：公司 / 制度 / 機構 / 體系 / 殘留或已解散組織），內容包含 ORG card 7 段固定 schema：

1. 組織本質（essence）
2. 對抗性質與來源（antagonism）
3. 組織結構/層級（structure）
4. 殘留型態（residual）
5. 影響範圍（influence）
6. 下游 hooks（hooks）
7. 文件語體 hint（hint→方向 B）

本協議只處理一個非人格組織節點。它**不**建立角色聲線卡、**不**設計完整劇情、**不**寫具體場景台詞、**不**把 ORG 當會說話的說話者。

**核心不變量（不可違反）：**
- **ORG-* 永無聲線卡**，不寫入 `03_characters/`。
- **ORG-* 不進 `/dialogue-write` 為說話者**（組織不會說話）；其「殘留文件語體」僅在 §7 留 hint hook，方向 B 才正式做 W-language 文件語體卡。
- ORG-* 是可被 `P` / `S` / `R` / `C` 以穩定 ID `depends_on` / cross-ref 的一級節點。

本協議完成後，產物寫入：

- `11_organizations/<name>.md`（唯一寫檔目標）

`.protocol_version.phase_log` 記錄 runtime log。

## 2. 啟動條件

使用者以 `/create-org` 或 `/建立組織` 啟動本流程時，agent 必須先確認下列條件：

| 條件 | 要求 | 不符合時處理 |
|---|---|---|
| 世界觀先決 | `W-rules` 至少 REVIEW（ORG 需世界觀脈絡判定對抗性質） | 進入「先決資料缺失」處理；建議先 `/create-world` 或 `/iterate-world` |
| 啟動指令 | 指令中明示要建立的組織名 | 要求使用者補組織名 |
| 目標性質 | 目標為「非人格組織 / 制度 / 體系對抗源」，不是會說話的角色 | 若描述顯示是會說話的角色 → 提示改用 `/create-character` |
| 組織 ID | `ORG-<name>`，name 非空 | agent 依組織名生成；空名拒絕 |
| 重複建立 | 同一 `ORG-<name>` 尚未建立 | 若已存在，提示改用 `/iterate-org` 並中止 |
| 寫入狀態 | 目標檔不是 LOCKED，或已有明確修改授權 | 若 LOCKED，停止並請使用者確認 |
| issue registry | `<instance_root>/issue_type_registry.yaml` 可讀；若不存在，從 `_design/registries/issue_type_registry.template.yaml` Template fallback + WARN；對應 skill key `00_n_organization` 必須存在 | YAML parse error、缺 `00_n_organization` skill key、議題 id 違規、locked 欄非 bool 等 schema 異常時，拒絕進階段 2 |

> **issue-ful 設計（D-074 amendment；§13 Q2 user 拍板）：** 本協議**讀** `issue_type_registry` 的 `00_n_organization` key，按 **`core + user_extensions − core_overrides`** 動態構建議題清單（比照 `00_l`）。7 個 core 議題對應 ORG card 7 段。

### 2.1 啟動後第一句

agent 不得直接寫檔。啟動後第一句固定為：

> 我們要建立組織 `ORG-<name>`。請先貼出你對這個組織的長段想法；可以包含它是什麼組織 / 制度 / 體系、為什麼是對抗源、內部結構或層級、現在以什麼形式存在（文件 / 條文 / 設施 / 流程 / 已破產解散）、影響哪些角色與地點、未來在主線怎麼被引用。

## 3. 階段 1：自由輸入診斷

### 3.1 使用者自由輸入

使用者可貼長段敘述、設定筆記、殘留文件素材（`_source_materials/` 可讀不可寫）。agent 在本階段只能診斷與提問，不得拆分寫檔。

### 3.2 診斷重點

agent 依 `00_protocol/00_a_台詞生產協議.md` §3.3 診斷模式輸出診斷報告，重點檢查：

| 診斷項 | 檢查內容 |
|---|---|
| 是否真為非人格 ORG | 目標是否為組織 / 制度 / 體系，而非會說話的角色（呼應 `/create-character` Stage 1.0 gate） |
| 對抗性質清晰度 | 為何是對抗源；對抗形式（制度壓迫 / 殘留債務 / 體系慣性…）是否可說清 |
| 組織結構 | 是否有可辨識的內部層級 / 派系 / 殘留網絡 |
| 殘留型態 | 以什麼形式存在 / 被引用；是否已破產 / 解散 |
| W-rules 合規 | 組織設定是否與世界規則一致 |
| 下游落點 | 影響哪些角色 / 地點 / 主線段；可交給哪個下游 skill |

### 3.3 診斷報告結尾

診斷報告結尾必須預告接下來會依 `00_n_organization` registry core 動態構建的議題清單提問（7 個 core 議題對應 ORG card 7 段；最終清單依 `user_extensions` / `core_overrides` 動態構建）。

若目標經診斷其實是「會說話的角色」，agent 必須停止並建議改用 `/create-character`，不得硬建 ORG。

## 4. 階段 2：逐題探索順序

## 4.0 議題清單動態構建（D-047 / Contract D §4a.2 對齊；D-074 amendment）

階段 2 啟動時，agent **必須**讀 `<instance_root>/issue_type_registry.yaml` 的 `00_n_organization` skill key 區段，按 **`core + user_extensions − core_overrides`** 動態構建議題清單：

1. **core 議題**先依 `id` 升序排列（id 1-99 為 core 保留範圍）。
2. **`user_extensions[*]`** 接在 core 議題後（id ≥ 100 為 user 自訂議題保留範圍）。
3. 套用 **`core_overrides[*].skip_id`** 過濾：
   - 對應 core entry `locked=true` → agent **忽略**該 override 並 **WARN**「locked 議題 id={n} 不可 SKIP」；該議題仍出現
   - 對應 core entry `locked=false` → 從議題清單中移除
   - `skip_id` 不存在於 core → WARN「core_overrides skip_id={n} 找不到對應 core 議題；條目被忽略」
4. 對每個議題：
   - `required_level=REQUIRED` → 跳階段 4 前**必答**；缺漏拒絕進階段 4
   - `required_level=STRONGLY_PREFERRED` → 允許但 phase_log 紀錄缺漏 + 對應 ORG card 段標 TODO
   - `required_level=OPTIONAL` → 允許不紀錄
5. `core[*].question_summary` 只作 opener；完整提問腳本見本檔 §4.1。

**Registry 異常處理：**
- registry 載入失敗（YAML parse error / 缺 `00_n_organization` key 等）→ agent **拒絕進階段 2**，請 user 修補 registry。
- `user_extensions[*].id < 100` 或與 core id 重複 → parser ERROR；agent 拒絕進階段 2。

## 4.1 議題順序預設表 + 專屬腳本

00_n core 共 **7 個 user-facing 議題**（registry 對應 id 1-7）。若 user 在 `user_extensions:` 加自訂議題，按 §4.0 接於 core 後。

| 順序 | 議題 | required_level | locked | ORG card 段 | 為什麼這時問 |
|---|---|---|---|---|---|
| 1 | 組織本質 | REQUIRED | true | §1 | 先定位組織是什麼，後續對抗 / 結構 / 殘留才有基準。 |
| 2 | 對抗性質與來源 | REQUIRED | true | §2 | ORG 之所以被獨立追蹤的理由；缺此段 ORG 無意義。 |
| 3 | 組織結構/層級 | STRONGLY_PREFERRED | false | §3 | 決定 ORG 內部如何被引用 / 哪一層是對抗主體。 |
| 4 | 殘留型態 | STRONGLY_PREFERRED | false | §4 | 決定下游如何引用本 ORG。 |
| 5 | 影響範圍 | STRONGLY_PREFERRED | false | §5 | 下游 depends_on / cross-ref 判定基線。 |
| 6 | 下游 hooks | STRONGLY_PREFERRED | false | §6 | 把 ORG 與下游 skill 銜接（不含台詞）。 |
| 7 | 文件語體 hint | OPTIONAL | false | §7 | 為未來方向 B 留 hook。 |

agent 一次最多問 3 題，每題說明影響哪個下游。允許 `跳這題`、`先跳到議題 X`、`中止`。每題輸出：為什麼問 / agent 怎麼問 / 使用者預期答什麼 / agent 怎麼整理寫檔 / 拒答跳題處理。

**§1 組織本質（REQUIRED / locked）**：問「這是什麼組織 / 制度 / 體系？一句話定位」；寫入 ORG card `§1`；必答，缺則拒絕進階段 3。

**§2 對抗性質與來源（REQUIRED / locked）**：問「為何是對抗源？對抗形式（制度壓迫 / 殘留債務 / 體系慣性 / 其他）」；寫入 `§2`；必答，缺則拒絕進階段 3。

**§3 組織結構/層級（STRONGLY_PREFERRED）**：問「內部結構 / 層級（高層 / 執行層 / 殘留網絡 / 派系等）？哪一層是對抗主體？」；寫入 `§3`；可跳，跳則標 TODO。〔§13 Q1 user 拍板新增段〕

**§4 殘留型態（STRONGLY_PREFERRED）**：問「以什麼形式存在 / 被引用（文件 / 條文 / 設施 / 流程）？是否已破產 / 解散？」；寫入 `§4`；可跳標 TODO。

**§5 影響範圍（STRONGLY_PREFERRED）**：問「影響哪些角色 / 地點 / 主線段？」；寫入 `§5` + 可補 frontmatter `depends_on` 提示；可跳標 TODO。

**§6 下游 hooks（STRONGLY_PREFERRED）**：問「交給 /create-outline·/scene-task·/create-relationship 哪些鉤子？（不含台詞）」；寫入 `§6`（僅登錄，不寫下游檔）；可跳標 TODO。

**§7 文件語體 hint（OPTIONAL）**：問「殘留文件『讀起來』是什麼語體（官腔 / 條文體 / 公告體…）？」；寫入 `§7`，明標「僅 hint；正式文件語體卡待方向 B」；可跳不紀錄。

## 5. 階段 3：收斂與寫入預覽

所有 REQUIRED 議題完成後，agent 必須輸出收斂預告稿並等待使用者確認。預告稿必須包含：

| 項目 | 要求 |
|---|---|
| 組織一句話 | 由議題 1/2 提煉一句定位，使用者拍板措辭 |
| ORG card 7 段寫入預覽 | 顯示 `11_organizations/<name>.md` 將寫入的 7 段 |
| frontmatter 預覽 | `entities: [ORG-<name>]` / `depends_on` / `weight` / 5 欄 header |
| TODO / INFERENCE / CONFLICT 清單 | 跳過的 STRONGLY_PREFERRED 議題與下游影響 |

未經使用者明確確認（`通過` / `OK` / `寫檔`），agent 不得進入階段 4。若有未解 CONFLICT，不得進入階段 4。

## 6. 階段 4：執行寫檔

使用者確認後，agent 才可寫入。**唯一寫檔目標為 `11_organizations/<name>.md`**（D-050 子裁決 2 新增 /create-org 列）。

寫檔前重讀目標檔 LOCKED 狀態；若 `11_organizations/` 目錄不存在，on-demand 建立（同 `10_art_assets/` 慣例）。

固定 7 段寫入順序：§1 → §2 → §3 → §4 → §5 → §6 → §7。

### 6.1 ORG card frontmatter 規範

```yaml
---
entities: [ORG-<name>]
depends_on: [W-rules]
weight:
  ORG-<name>: 1.0
---
```

- 5 欄中文 header 狀態 DRAFT。
- `depends_on` 至少含 `W-rules`；依 §5 影響範圍補相關 `C-*` / `V` 等。
- `weight` 預設 `{ORG-<name>: 1.0}`（與 D-071 parser scalar 1.0 行為一致）。

### 6.2 缺漏標記

```md
<!-- TODO(<議題 id>): <缺什麼、何時可補> -->
<!-- INFERENCE: <依據>; 待人類確認 -->
<!-- CONFLICT(<file>:<section>): 既有=<existing>, 本次想寫=<new>. 請人類拍板 -->
```

- 缺 REQUIRED 議題（議題 1 / 2）阻擋階段 4。
- 跳過的 STRONGLY_PREFERRED 議題以 TODO 標記 + phase_log 記缺漏。

## 7. 階段 5：驗證與狀態回報

寫檔完成後，agent 必須自動呼叫 `/status` 並回報：

| 項目 | 預期 |
|---|---|
| ORG-<name> | 新增為 DRAFT |
| 11_organizations/<name>.md | 新建，含 7 段 + frontmatter |
| .protocol_version.phase_log | 含 `phase: create-org` / `created_entities: [ORG-<name>]` |
| 殘餘標記 | 列剩餘 TODO / INFERENCE / CONFLICT |

建議下一步：

- 若 ORG 與某會說話角色有對抗 / 從屬關係 → 跑 `/create-relationship <C-*> <ORG-*>`（C↔ORG endpoint；至多一端 ORG）。
- 若需迭代本 ORG → `/iterate-org <name>`。
- 若需查看本 ORG → 目前直接讀 `11_organizations/<name>.md`。（view-world「勢力與組織」段 compose ORG / 獨立 `/view-org` `/export-org` 為 **後續批**，見 D-074 §13 Q5 決策佇列；本批未實作，勿指引使用者跑 /view-world 看 ORG。）

## 8. 禁止事項

本協議執行期間禁止：

1. 建立任何 `C-*` 聲線卡或寫入 `03_characters/`。
2. 把 ORG 當會說話的說話者、生成任何台詞、進 `/dialogue-write`。
3. 寫入 `11_organizations/` 以外任何目錄（含 `00_protocol/`、`04_relationships/`、`05_plot/` 等）。
4. 把使用者未確認的推論寫入 ORG card。
5. 同一 `ORG-<name>` 重跑時直接覆寫；必須提示改用 `/iterate-org`。
6. 對 LOCKED 檔案直接寫入。
7. 在 §7 正式撰寫 W-language 文件語體卡（屬方向 B；本協議只留 hint）。
