# 00_l 關係創建協議

## 1. 文件目的

本協議規範 `/create-relationship` / `/建立關係` 在關係實體創建過程中的執行流程。

本協議的目標是引導使用者建立 `R-<a>-<b>` 實體，內容包含：

- 關係類型
- 權力差
- 稱呼系統
- 情緒債
- 不能說出口的事
- 關係時間線錨點

本協議只處理一對角色之間的關係。它不負責創建角色、不改角色聲線卡核心內容、不設計完整劇情、不寫具體場景台詞。

本協議完成後，產物主要寫入：

- `04_relationships/04_a_角色關係矩陣.md`
- `04_relationships/04_b_關係變化時間線.md`

必要時補充：

- `02_vocabulary/02_c_禁用詞與慎用詞表.md`
- `05_plot/05_c_角色弧線表.md`

## 2. 啟動條件

使用者以 `/create-relationship` 或 `/建立關係` 啟動本流程時，agent 必須先確認下列條件：

| 條件 | 要求 | 不符合時處理 |
|---|---|---|
| 端點先決 | 兩端實體都至少 REVIEW，**其中至多一端為 ORG-***（C↔C 或 C↔ORG；不允許 ORG↔ORG，見 §2.2） | 進入「先決資料缺失」處理 |
| 啟動指令 | 指令中明示要建立的一對端點名（C 名或 ORG 名） | 要求使用者補端點名 |
| 關係 ID | `R-<a>-<b>`，a / b 取端點裸名依字典序排列 | agent 自動排序，不要求使用者手動修正 |
| 重複建立 | 同一對關係尚未建立 | 若已存在，提示改用 `/iterate-relationship` 並中止 |
| 寫入狀態 | 目標檔不是 LOCKED，或已有明確修改授權 | 若 LOCKED，停止並請使用者確認 |
| issue registry | `<instance_root>/issue_type_registry.yaml` 可讀；若不存在，從 `_design/registries/issue_type_registry.template.yaml` Template fallback + WARN；對應 skill key `00_l_relationship` 必須存在 | YAML parse error、缺 `00_l_relationship` skill key、議題 id 違規、locked 欄非 bool 等 schema 異常時，拒絕進階段 2 |

### 2.1 啟動後第一句

agent 不得直接寫檔。

啟動後第一句固定為：

> 我們要建立 `<C-a>` 與 `<C-b>` 的關係。請先貼出你對這兩人之間關係的長段想法；可以包含他們怎麼相遇、現在的狀態、為什麼會這樣、有沒有秘密、關係未來會怎麼變。可以從任一視角寫。

若一端為組織（ORG-*），第一句改為：

> 我們要建立角色 `<C-a>` 與組織 `<ORG-b>` 的關係。請貼出你對這個角色與這個組織之間關係的長段想法；可以包含對抗 / 從屬 / 受其影響的形式、現在的狀態、未來在主線怎麼推進。

### 2.2 C↔ORG endpoint 規則（F8 Phase 3 / D-074）

本協議容許關係的**一端為 ORG-***（非人格組織 / 制度 / 體系對抗源；見 `00_protocol/00_n_組織創建協議.md`）：

- **容許型態**：`C↔C`（既有，行為不變）或 `C↔ORG`（一端 C-* 一端 ORG-*；例 `C-瑟琳 ↔ ORG-清道夫`）。
- **不允許 ORG↔ORG**（至多一端 ORG；DECISIONS_LOG §6.25.2 item 6 預設）。若兩端皆 ORG，停止並說明本協議不支援。
- **端點先決**：兩端至少 REVIEW；其中至多一端 ORG-*。ORG 端的「REVIEW」指其 ORG card 至少 REVIEW。
- **ORG 端無聲線卡**：ORG-* 永無聲線卡（00_n 不變量）。因此 §6 階段 4 寫聲線卡關係段時，**只寫 C 端聲線卡的關係段，ORG 端不寫**（ORG 無聲線卡可寫）。`04_a` / `04_b` 關係矩陣 / 時間線照舊寫。
- **議題適配**：6 議題照跑，但「稱呼系統」「不能說出口的事」等以**角色對組織**的視角理解（組織不會回話）；agent 不得為 ORG 端虛構台詞或聲線。
- **frontmatter**：`04_a` 的 `entities` / `depends_on` 納入該 `ORG-<name>`（與 C-* 同等的一級節點）。

## 3. 階段 1：自由輸入診斷

### 3.1 使用者自由輸入

使用者可貼長段敘述、片段筆記、人物互動、未整理的關係想法。

agent 在本階段只能診斷與提問，不得拆分寫檔。

### 3.2 診斷重點

agent 依 `00_protocol/00_a_台詞生產協議.md` §3.3 的診斷模式輸出診斷報告，重點檢查：

| 診斷項 | 檢查內容 |
|---|---|
| 表面 vs 真實 | 對外呈現的關係與實際內裡是否一致 |
| 權力平衡 | 雙方在資訊、武力、地位、情緒主動權上誰握有籌碼 |
| 雙方視角 | 是否只寫了其中一方的感受或理由 |
| 台詞落點 | 是否已有稱呼、禁忌、不能說出口的事 |
| 時間推進 | 關係是否有清楚的變化節點 |

### 3.3 診斷報告結尾

診斷報告結尾必須預告接下來會依 registry core 問 6 個 user-facing 議題；最終議題清單依 `issue_type_registry.yaml` 的 `user_extensions` / `core_overrides` 動態構建：

1. 關係類型分類
2. 權力差
3. 稱呼系統
4. 情緒債
5. 不能說出口的事
6. 關係時間線錨點

若 `05_plot/05_b_章節結構模板.md` 尚未建立有效章節，agent 必須提醒：第 6 題可暫跳，之後在 00_h 後再補。

## 4. 階段 2：逐題探索順序

## 4.0 議題清單動態構建（D-047 / Contract D §4a.2）

階段 2 啟動時，agent **必須**讀 `<instance_root>/issue_type_registry.yaml` 的 `00_l_relationship` skill key 區段，按 **`core + user_extensions − core_overrides`** 動態構建議題清單：

1. **core 議題**先依 `id` 升序排列（id 1-99 為 core 保留範圍）。
2. **`user_extensions[*]`** 接在 core 議題後（id ≥ 100 為 user 自訂議題保留範圍）。
3. 套用 **`core_overrides[*].skip_id`** 過濾：
   - 對應 core entry `locked=true` → agent **忽略**該 override 並 **WARN**「locked 議題 id={n} 不可 SKIP，core_overrides 條目被忽略」；該議題仍出現在清單
   - 對應 core entry `locked=false` → 從議題清單中移除
   - `skip_id` 不存在於 core → WARN「core_overrides skip_id={n} 找不到對應 core 議題；條目被忽略」
4. 對每個議題：
   - `required_level=REQUIRED` → 跳階段 4 前**必答**；缺漏拒絕進階段 4
   - `required_level=STRONGLY_PREFERRED` → 跳階段 4 允許但 phase_log 紀錄缺漏 + 對應分拆檔段落標 TODO
   - `required_level=OPTIONAL` → 跳階段 4 允許不紀錄
5. `core[*].question_summary` 只作為 opener；agent 完整提問腳本詳本檔 §10 對應段（沿用 UD §1.5.2 完整內容）。
6. **拆分規則不入 registry** — 屬 agent 階段 4 寫檔 mechanic，由 §10.7 固定執行，不對 user 提問。

**Registry 異常處理：**
- 若 registry 載入失敗（YAML parse error / 缺 `00_l_relationship` key 等）→ agent **拒絕進階段 2**，請 user 修補 registry
- 若 `user_extensions[*].id < 100` 或 與 core id 重複 → parser ERROR；agent 拒絕進階段 2

## 4.1 議題順序預設表

00_l core 共 **6 個 user-facing 議題**（registry 對應 id 1-6）。若 user 在 `user_extensions:` 加自訂議題，按上述 §4.0 規則接於 core 後。

| 順序 | 議題 | required_level | locked | 為什麼這時問 |
|---|---|---|---|---|
| 1 | 關係類型分類 | REQUIRED | true | 先定義關係基底，後續權力、稱呼與禁忌才有判定基準。 |
| 2 | 權力差 | STRONGLY_PREFERRED | false | 關係張力與對話攻防方向需要先確定。 |

**§10.7 拆分規則**屬 agent-side execution mechanic（非 user-facing 議題），階段 3 收斂預告稿時固定列入寫檔計畫；user 只需在預告稿確認，不對拆分規則本身提問。詳本檔 §10.7。

agent 必須依 §4.0 動態構建後的清單提問，不得跳到拆分或寫檔。

每題都必須輸出：

- 為什麼問
- agent 怎麼問
- 使用者預期答什麼

- 拒答 / 跳題處理

所有必要議題完成後，agent 必須輸出收斂預告稿，並等待使用者確認。

收斂預告稿必須包含：

| 項目 | 要求 |
|---|---|
| 關係一句話 | 從最終 user-facing 議題清單提煉一句定位，由使用者拍板措辭 |

| 05_c 補充預覽 | 若有章節錨點，顯示角色弧線補充註記 |

## 6. 階段 4：拆分執行

使用者確認後，agent 才可依區段 10.7 的拆分規則寫入。

固定寫入順序：

1. `04_relationships/04_a_角色關係矩陣.md`
2. `04_relationships/04_b_關係變化時間線.md`
3. `02_vocabulary/02_c_禁用詞與慎用詞表.md`
4. `05_plot/05_c_角色弧線表.md`

`04_a` 與 `04_b` 是主要產物；`02_c` 與 `05_c` 只寫本關係衍生的禁語、稱呼限制或弧線註記，不可擴寫成新角色設定或新劇情。

- 若核心關係已齊，進入 00_h 細綱創建或下游 `/scene-task`

## 8. 禁止事項

本協議執行期間禁止：

1. 修改 `C-*` 聲線卡核心內容。
2. 擅自決定未來事件的具體發生方式。
3. 把時間線錨點寫成完整劇情段落。
4. 寫具體場景台詞。
5. 用關係說明替角色整理心理總結。
6. 將使用者未確認的關係推論寫入 04_a 或 04_b。
7. 同一對關係重跑時直接覆寫；必須提示改用 `/iterate-relationship`。
8. 對 LOCKED 檔案直接寫入。

## 10. 專屬議題腳本

### 10.1 關係類型分類

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 後續議題的判定基底。 |
| agent 怎麼問 | 關係類型兩題：a. 基底類型，可組合，選項包含對立、合作、血緣、戀愛、師徒、同袍、敵友、上下級、仇恨、救命、其他。b. 表面 vs 真實是否一致；兩人對外呈現的關係跟實際內裡是否一致？哪一方在演？ |
| 使用者預期答什麼 | 子題 a 給一到三個基底類型；子題 b 給一段說明。 |
| agent 怎麼整理寫檔 | 寫入 `04_a §<R-a-b> §1 關係類型`。若表面 vs 真實不一致，在 chat 印出下游台詞提醒，並記入「關係一句話」。 |
| 拒答 / 跳題 | 必答。 |

### 10.2 權力差

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 關係張力的源頭，影響台詞攻防方向。 |
| agent 怎麼問 | 權力差兩題：a. 權力四維評估，資訊、武力、社會地位、情緒主動權分別誰有更多籌碼。b. 權力會在哪些情境翻轉。 |
| 使用者預期答什麼 | 子題 a 給四維各一個答案；子題 b 給若干情境。 |

| 拒答 / 跳題 | 強烈建議不跳；若跳，下游 09_b 對攻防傾向的檢查缺基線。 |

### 10.3 稱呼系統

### 10.5 不能說出口的事

| 為什麼問 | 與 05_b 章節結構對齊，下游任務包可判定本場關係處於哪個狀態。 |
| 前置檢查 | 先看 `05_plot/05_b_章節結構模板.md` 是否已建立；已建才跑完整提問，未建則標 TODO 等 00_h 後補。 |
| agent 怎麼問 | 關係時間線錨點兩題：a. 哪幾章這個關係的狀態有顯著變化，逐條註明章節、狀態變化、緩變或突變。b. 從本協議建立到主線結束，這個關係有什麼絕對不會變。 |
| 使用者預期答什麼 | 子題 a 給若干錨點；子題 b 給一句或一段。 |
| agent 怎麼整理寫檔 | 子題 a 寫入 `04_b §<R-a-b> §章節錨點`。子題 b 寫入 `04_a §<R-a-b> §6 不變的部分` 與 `04_b §<R-a-b> §不變的部分`。若需要，於 `05_c` 加註此章該角色關係狀態。 |
| 拒答 / 跳題 | 05_b 未建時跳；已建仍可跳，但 04_b 必須標 TODO。 |

### 10.7 拆分規則

固定拆分計畫如下：

| 來源議題 | 目標 | 寫入方式 |
|---|---|---|
| 10.1 關係類型 | `04_a §<R-a-b> §1` | 覆寫該段 |
| 10.2 權力差 | `04_a §<R-a-b> §2` | 覆寫該段 |
| 10.3 稱呼系統 | `04_a §<R-a-b> §3` + `02_c §關係禁語 §<R-a-b>` | 覆寫 + append |
| 10.4 情緒債 | `04_a §<R-a-b> §4` + `04_b §<R-a-b> §情緒債` | 覆寫 + append |
| 10.5 不能說出口的事 | `04_a §<R-a-b> §5` + `02_c §關係禁語 §<R-a-b>` + `04_b §<R-a-b> §破戒錨點` | 覆寫 + append |
| 10.6 時間線錨點 | `04_b §<R-a-b> §章節錨點` + `04_a §<R-a-b> §6 不變的部分` + `05_c` 註記 | 覆寫 + append |

寫檔順序固定為：

1. `04_a`
2. `04_b`
3. `02_c §關係禁語`
4. `05_c` 註記

### 10.8 Frontmatter 規範

本協議自身通常不貢獻實體，可省略 YAML block。

依本協議寫入其他檔案時，目標檔 frontmatter 需依實際內容維護：

| 目標 | 規範 |
|---|---|
| 04_a | `entities` 隨關係累積；包含各 `R-*` 與相關 `C-*` |
| 04_a | `depends_on` 包含 `W-rules`、`V`、相關 `C-*` |
| 04_a | `weight` 可包含 `{R-*: 0.6, C-*: 0.1}` |
| 04_b | `entities` 包含各 `R-*-*` |
| 04_b | `depends_on` 包含 `P`、`CH-*`、各 `R-*-*` |
| 04_b | `weight` 可包含 `{R-*: 0.3}` |
| 02_c / 05_c | 只補相關依賴，不擅自新增 entity 類型 |

### 10.9 多次執行規則

| 情況 | 處理 |
|---|---|
| 同一對關係再次執行 | 視為迭代，提示改用 `/iterate-relationship` 並中止 |
| 不同對關係執行 | 追加到 `04_a` 新段落，並更新 `04_b` / `02_c` / `05_c` 相關段落 |
| 使用者要求一次建立多對關係 | 拆成多次 `/create-relationship`，每次只處理一對 |
