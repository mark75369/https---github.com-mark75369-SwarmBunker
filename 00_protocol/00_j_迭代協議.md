# 00_j_迭代協議

若本文件與舊聊天紀錄衝突，以本文件為準。

本協議規範 `/iterate-*` series（`/iterate-world` / `/iterate-character` / `/iterate-relationship` / `/iterate-outline` / `/iterate-detailed-outline` + `/iterate-scene --split-to-file`）的執行流程，被 5+1 個 `/iterate-*` skill 共同引用。本協議定義所有上游實體（W-rules / V / W-language / C-* / R-*-* / P / CH-* / S-*-*）的迭代邏輯共通基底；個別 skill 的差異段（讀檔位置、entity 類型、protocol 對齊）由各自 SKILL.md 處理。

`00_e` ~ `00_h` + `00_l` 規範「上游實體如何從零創建」；`00_k` 規範「下游台詞 pipeline」；本協議規範「**既有實體如何被使用者意圖**（含 Phase C/D 期間發現的設計缺漏 / Canon Delta 回寫 / per-scene 拆檔需求）**安全迭代 + 不破壞既有受影響檔**」。

# 1. 文件目的與適用範圍

本協議用於迭代已存在的上游實體與 Bible 分拆檔；不重新建立 entity。

本協議會：

- 識別變更點（哪些既有實體 / 分拆檔被改）
- 強制執行影響範圍評估（雙路反查 entities + depends_on；ARCH §5）
- 收斂變更（含拍板「只改 X、不改 Y / Z」選項）
- 執行變更（含 frontmatter / phase_log append）
- 驗證變更（自動 `/status`；不重生 view）

本協議不：

- 不建立新 entity（屬 `/create-*` scope）
- 不重新生成 view 整合檔（屬 `/export-*` scope；user 需要時手動跑 `/export-*`）
- 不執行下游台詞 pipeline 內容（屬 `/scene-task` / `/dialogue-write` / `/qa` scope）
- 不擅自跨入 LOCKED 實體（必須先走 LOCKED 解除流程；屬人類拍板）

**通用 vs 個別：**

- 本協議是「通用」基底；定義 5 階段框架 + 影響範圍評估 algorithm + 觸發語 + phase_log 紀錄欄位
- 各 `/iterate-*` SKILL.md 是「個別」實作；定義對應 entity 類型 / 對應 protocol（00_e/00_f/00_g/00_h/00_l）/ 對應分拆檔讀寫位置
- 個別 SKILL.md 必須引用本協議為共通基底，不重複本協議 substantive 內容

1. Instance 已 bootstrap，且 `.protocol_version` 存在。

# 3. 階段 1：變更點識別

agent 開場提示（範本；個別 SKILL.md 可依 entity 類型細化）：

> 請說明你想改什麼。可以是：
> - 既有實體的具體變更（例：「在 W-rules 加入宗教階級議題」）
> - 對既有設定的修補（例：「角色 C-林夜的聲線描述應從 'cold' 改為 'cold + 自嘲'」）
> - 來自 Canon Delta 回寫的具體 entry（例：「09_e 紀錄某場台詞建立了新事實 'X 神宗教階級'，需回寫 W-rules」）
> - 設計缺漏 / CONFLICT 處理（例：「05_d 規定本章不能透露 X，但任務包必須透露 X，需修 05_d」）
> 
> 越具體越好；空白處我會幫你補洞。

使用者貼完後，agent 依 00_a 診斷模式產出**變更點識別報告**。本報告只印在 chat，不寫檔。

變更點識別報告必含：

1. **變更目標 entity**：明示 entity ID（W-rules / V / W-language / C-<name> / R-<a>-<b> / P / CH-<n> / S-<ch>-<n>）
2. **變更類型分類**：
   - **新增**（既有 entity 內加新內容）
   - **修改**（既有 entity 內改舊內容）
   - **刪除**（既有 entity 內移除內容）
   - **重組**（既有 entity 內重新組織 / 重新分類）
3. **變更具體內容**：條列描述（不寫檔；只 chat）
4. **既有實體當前狀態**（DRAFT / REVIEW / FINAL / LOCKED）：
   - DRAFT / REVIEW：可直接迭代
   - FINAL：需 user 明示同意降回 REVIEW 才迭代
   - LOCKED：agent 拒絕；列出 LOCKED 解除流程指引
5. **預估影響範圍**（粗略 — 階段 2 強制完整評估）
6. **建議下一步與階段 2 提問清單預告**

診斷階段禁止：

- 寫入任何檔案
- 自動執行影響範圍評估（屬階段 2）
- 把使用者意圖自動延伸到未明示 entity

# 4. 階段 2：強制影響範圍評估

本階段是 `/iterate-*` 的核心差異 — `/create-*` 不需要，但 `/iterate-*` 必須執行。

依 ARCH §5 + SPEC 5.2 鎖定的 frontmatter schema，agent 執行**雙路反查**：

## 4.1 反查邏輯

```python
def find_affected_files(modified_entity, all_files):
    direct = []      # entities 含 modified_entity
    depends = []     # depends_on 含 modified_entity
    indirect = []    # 直接受影響檔案的次級實體被影響

    # 路徑 A — 直接受影響：「貢獻」關係
    for f in all_files:
        if modified_entity in f.entities:
            direct.append(f)

    # 路徑 B — 依賴受影響：「依賴」關係（C-9 拍板）
    for f in all_files:
        if modified_entity in f.depends_on:
            depends.append(f)

    # 路徑 C — 間接受影響：直接受影響檔案的其他 entities 對應的檔案
    secondary_entities = set()
    for f in direct:
        for e in f.entities:
            if e != modified_entity:
                secondary_entities.add(e)

    for f in all_files:
        if f in direct or f in depends:
            continue
        if any(e in f.entities for e in secondary_entities):
            indirect.append(f)

    return {
        "direct": direct,
        "depends": depends,
        "indirect": indirect,
    }
```

## 4.2 雙路反查紀律

- **`entities` 反查抓「定義該實體的檔案」**（直接受影響）
- **`depends_on` 反查抓「使用該實體的檔案」**（依賴受影響）
- 兩條路徑互補；初版 `depends_on` 可能不完整 → 靠 indirect 兜底
- 階段 2 結尾印「依賴受影響清單看起來是否完整？若有遺漏請補 frontmatter `depends_on`」

## 4.3 影響範圍呈現格式（範本）

```text
影響範圍評估：

直接受影響（entities 含 <modified_entity>）：
- <檔案 1>（將新增 XX 段落）
- <檔案 2>（將更新 XX 欄位）

依賴受影響（depends_on 含 <modified_entity>）：
- <檔案 3>（內容依賴本實體）
- <檔案 4>（內容依賴本實體）

間接受影響（次級實體的對應檔案；由 entities 反查補漏）：
- <檔案 5>（含本實體相關段落；可能不需要動）

請選擇本輪要處理的範圍：
1. 只改直接受影響
2. 改直接受影響 + 依賴受影響
3. 改直接受影響 + 依賴受影響 + 間接受影響
4. 全部都改

請順帶確認：以上「依賴受影響」清單看起來是否完整？
若有遺漏，請補上對應檔案的 frontmatter depends_on，下次反查就會抓到。
```

## 4.4 下游 pipeline 互鎖檢查（依 00_k §10.7.5）

若影響範圍含 C-* / R-*-* / P / CH-* / S-*-*，且 phase_log 含進行中的 `/scene-task` / `/dialogue-write` / `/qa` 引用本 entity，agent **必須**：

1. 印「進行中的下游 skill 清單」
2. 印 abort_reason 選項：
   - **等待完成**：暫停本次迭代；等下游 skill complete
   - **中止下游**：明示中止哪些下游 skill；對應 phase_log 補 `status: aborted` + `abort_reason: iterate_force`
   - **強制迭代**：本次迭代繼續；但 phase_log 標 `prereq_changed: true` 警示後續 /qa 可能 FAIL
3. 等 user 拍板才進階段 3

# 5. 階段 3：收斂

agent 整合階段 2 影響範圍評估 + 使用者拍板範圍 → 產出**收斂預告稿**。收斂預告稿只印在 chat，不寫檔。

收斂預告稿必含 5 欄位：

1. **變更目標 entity** + 最終結論一句話（對應階段 1 報告）
2. **本輪要處理的檔案清單**（含完整路徑；對應階段 2 user 拍板的範圍）
3. **每檔案的具體變更動作**：
   - 新增哪段內容（含 anchor 位置）
   - 修改哪段欄位（含 before / after）
   - 刪除哪段內容
   - frontmatter 更新（哪些 `entities` / `depends_on` / `weight` 動）
4. **TODO 清單**（本次無法完成需後續補的項；含「依賴受影響補 depends_on」建議）
5. **影響範圍預告**：
   - 是否動到任何 LOCKED 檔（必須 0；若 > 0 拒絕）
   - 是否動到任何 FINAL 檔（須 user 明示同意降回 REVIEW）
   - phase_log 將標的 `prereq_changed` 等下游 alert

等使用者明確「通過」/「OK」/「寫檔」才進階段 4。

未經使用者拍板，agent 不得寫檔。

# 6. 階段 4：執行

依 ARCH §6.7.1 區段 6 固定順序：

1. 計算目標路徑（依階段 3 拍板的檔案清單）
2. 對每檔讀現有 / 新建段落 / 找 anchor 寫入
3. frontmatter 補完 SPEC §5.2 規範（新增 `depends_on` entry / 更新 `weight` 等）
4. 00_b 寫入依 SPEC §17 規則（若變更目標含 W-rules 與作品專屬反 AI 味基線相關）
5. `.protocol_version.phase_log` append iterate 紀錄

**寫檔順序：** 主分拆檔 → 衍生分拆檔 → 00_b。任一步出錯：

- rollback 已寫的變更
- 印錯訊息
- 不更新 phase_log
- 提示 user 是否重試 / 拍板降級

## 6.1 phase_log entry 範例

```yaml
- phase: iterate-<entity_type>
  date: YYYY-MM-DD
  skill: /iterate-<entity_type>
  status: completed
  modified_entity: <entity_id>
  modified_files:
    - <檔案 1 路徑>
    - <檔案 2 路徑>
  scope_choice: 2  # 1/2/3/4 對應階段 2 user 拍板範圍
  affected_files_evaluated:
    direct: [<檔案清單>]
    depends: [<檔案清單>]
    indirect: [<檔案清單>]
  prereq_changed: false  # 若強制迭代且有進行中下游，標 true
  abort_reason: null  # 若 aborted，標原因
  customizations: []
```

## 6.2 View 重生紀律（O3 鎖定）

`/iterate-*` 完成後**不**自動重生 view 整合檔。

若使用者要看新狀態的整合視圖，須**手動**跑對應 `/export-*` skill（Phase D Wave 14 範圍）。

理由：view 重生屬 `DERIVED` 衍生物；自動重生會掩蓋使用者拍板細節 + 增加 race condition 風險。

# 7. 階段 5：實體驗證

每個 `/iterate-*` skill 結束都自動跑 `/status`。

**自動 `/status` 範圍：**

1. 列本次更新實體 ID + 完成度（**不**應升新等級；除非 user 明示）
2. 列寫入檔案清單（對應階段 4 寫的檔）
3. expected entity manifest 對照
4. 後續建議下一步 skill（含「需要重新看視圖請跑 /export-<entity_type>」 + 「若下游 pipeline 受影響請考慮重跑 /qa」）

**不可：**

- 自動跑下一個 skill
- 自動升 entity 狀態
- 自動重生 view

# 8. 禁止事項

1. agent 不擅自跳階段（特別是階段 2 影響範圍評估）
2. agent 不擅自迭代 LOCKED 實體
3. agent 不擅自升 entity 狀態（含階段 4 後自動升 REVIEW）
4. agent 不擅自重生 view 整合檔
5. agent 不擅自呼叫其他 skill（含 /export-*）
6. agent 不擅自新增實體類型 / enum 值
7. agent 不擅自跳過階段 2 影響範圍評估（即使 user 明示「直接寫檔」也須先列影響範圍）
8. agent 不擅自延伸變更到未明示 entity（「Fix one, find two」cascade pattern 預防）
9. agent 不擅自中止進行中的下游 skill（屬 user 拍板）
10. agent 不擅自處理 Canon Delta 回寫（屬人類拍板 09_e → /iterate-*；本協議不自動觸發）

# 9. 缺漏處理

## 9.1 影響範圍 depends_on 不完整

階段 2 反查時若發現多個 indirect 但 depends 為空，agent 必須印：

> 影響範圍中 `depends_on` 反查空，但 `entities` 反查有 N 個間接受影響。建議：補上對應檔案的 frontmatter `depends_on`，下次反查就會抓到。
> 
> 本次處理選項：
> 1. 接受 indirect 為「依賴受影響候選」（保守處理）
> 2. 補 depends_on 後重跑階段 2（推薦；長期維護）
> 3. 跳過 indirect（風險 — 下游 QA 可能 FAIL）

## 9.2 CONFLICT

迭代過程發現衝突（例：W-rules 改動跟既有 C-* 設定衝突）：

- 在階段 2 報告中標 `CONFLICT(<details>)`
- 在階段 3 收斂預告稿中明示「需 user 同時處理 C-* 才能 commit」
- 不擅自自動修 C-*；屬 user 拍板要不要平行 trigger `/iterate-character`

## 9.3 LOCKED 實體迭代請求

agent 拒絕並印：

> 目標 entity <ID> 狀態為 LOCKED。本協議不擅自迭代 LOCKED 實體。
> 
> LOCKED 解除流程（屬人類拍板）：
> 1. user 手動把實體 frontmatter 改為 REVIEW（明示理由）
> 2. user 在 09_e（若為下游 entity）或 phase_log（若為上游 entity）紀錄「解除 LOCKED 理由」
> 3. 重跑 `/iterate-<entity_type>`

## 9.4 過時依賴

若本次迭代會降級下游引用的 entity（例：iterate W-rules → C-* 需重審）：

- 階段 4 phase_log entry 標 `downstream_review_required: true`
- 階段 5 印「下游 entity 可能需重審 — 建議 user 拍板：保持 REVIEW / 降回 DRAFT 重審」

# 13. 觸發語字典（共通，沿用 ARCH §6.7.2）

所有 `/iterate-*` skill 必須穩定識別下列觸發語：

| 觸發語 | 行為 |
|---|---|
| `跳到階段 X` / `跳階段` | agent 印當前已確認資料快照、請使用者確認 → 跳到階段 X。**`/iterate-*` 專屬硬性紀律：禁止跳過階段 2 影響範圍評估；禁止跳過階段 3 user approval；階段 4 只能從已通過階段 3 進入。**若 user 跳階段方向觸及此硬性紀律 → agent 拒絕跳階段並印紀律提示 |
| `跳這題` / `跳這個議題` | 當前議題標 TODO 並進下一題 |
| `通過` / `OK` / `寫檔` | 從階段 3 進階段 4（僅在階段 3 有效）|
| `中止` / `取消` | 不寫檔離開協議；`.protocol_version.phase_log` 補 `status: aborted` |
| `重來` / `從頭來` | 棄置目前累積的所有變更，從階段 1 重啟（保留變更點識別報告）|
| `這部分先不要動` | 該變更對應的分拆檔在階段 4 跳過寫入 |
| `直接寫檔` | 從階段 1 直接跳階段 3（**仍須跑階段 2 影響範圍評估**；不可省略；對應上述硬性紀律）|

**硬性紀律（vs `/create-*` 共通骨架的差異）：**

`/iterate-*` skill 的階段 2 影響範圍評估 + 階段 3 user approval 是「**強制 gate**」，不像 `/create-*` 的「探索 / 收斂」可由 user 跳階段省略。理由：迭代是「改既有實體 + 可能 cascade 下游」性質；跳過影響範圍評估 = 失去 cascade 預防 + 失去 view / dialogue / QA 一致性保證。

具體禁止行為：

1. user 喊 `跳到階段 4` / `跳到執行` → agent 拒絕並印「`/iterate-*` 必須跑階段 2 + 階段 3 approval；不可跳過」
2. user 喊 `跳階段` 略過階段 2 → agent 拒絕並印同上
3. user 喊 `通過` 在階段 1 / 階段 2 內 → agent 拒絕並印「`通過` 僅在階段 3 收斂預告稿後有效」
4. user 喊 `直接寫檔` 在階段 1 → agent 走「階段 1 → 階段 2（強制）→ 階段 3 預告稿 → 等 user 通過」最短路徑；不繞過階段 2 / 階段 3
