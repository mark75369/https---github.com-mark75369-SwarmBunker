狀態：DRAFT

最後更新：2026-05-07
適用範圍：全作品
優先級：高

若本文件與舊聊天紀錄衝突，以本文件為準。  
本文件為通用輸出格式規範，不綁定任何特定作品世界觀、角色或劇情。

---

# 1. 文件目的

本文件定義 AI 與 Codex 在遊戲劇本台詞工作流中的標準輸出格式。

本文件的目的不是規定台詞內容，而是規定不同工作階段應如何輸出，避免發生以下問題：

- 輸出格式混亂
- 台詞、註解、分析混在一起
- 初稿與交稿格式不一致
- QA 報告缺少具體位置
- Codex 修改檔案後沒有說明影響範圍
- 多版本台詞難以比較
- 任務包欄位不完整
- 修稿方向與實際改動難以追蹤
- final 版混入分析文字

本文件適用於：

- 台詞試寫
- 多版本生成
- 台詞修稿
- 台詞壓縮
- 台詞 QA
- 場景任務包建立
- 角色聲線卡建立
- 世界語言規格整理
- Codex 修改報告
- 交稿版台詞輸出

# 2. 基本輸出原則

## 2.1 先判斷工作模式

每次輸出前，AI 應先判斷目前任務屬於哪一種模式：

- 開發模式
- 多版本試寫模式
- QA 模式
- 修稿模式
- 壓縮模式
- 交稿模式
- 文件建立模式
- Codex 修改報告模式

若使用者未指定模式，預設使用「開發模式」。

若使用者明確要求「只輸出台詞」「不要分析」「交稿版」，必須使用「交稿模式」。

---

## 2.2 不同模式不得混用

以下情況應避免：

- 交稿模式中附加分析
- QA 模式中直接改寫台詞但未說明
- 修稿模式中大幅改變劇情資訊
- 文件建立模式中混入角色台詞試寫
- 多版本試寫模式中只給一版
- Codex 修改報告中只說「已完成」但不列出修改檔案

模式混用會造成文件污染與版本混亂。

---

## 2.3 輸出應可被直接保存為 .md

除非使用者要求聊天式回答，否則所有正式輸出應盡量符合 Markdown 文件格式。

應避免：

- 不必要的閒聊
- 模糊說明
- 未標題分段
- 未標註版本
- 台詞與分析混在同一段
- final 台詞後附加「希望你喜歡」等聊天內容

---

## 2.4 輸出模式規格

### 2.4.1 輸出模式總表

| 輸出模式 | 用途 | 是否含分析 | 是否可保存 | 是否可 final |
|---|---|---|---|---|
| 診斷模式 | 分析資料與風格 | 是 | 可保存為報告 | 否 |
| 整理模式 | 將資料填入模板 | 可少量 | 可保存為 DRAFT | 否 |
| 探索模式 | 發想可能方向 | 是 | 可保存為素材 | 否 |
| 試寫模式 | 產出台詞候選 | 可附簡評 | 可保存為 ALT / DRAFT | 否 |
| 破格模式 | 尋找不安全亮點 | 必須附風險 | 可保存為素材 | 否 |
| 收斂模式 | 整合可用版本 | 可附變更說明 | 可保存為 REVIEW | 否 |
| QA 模式 | 檢查問題 | 是 | 必須保存 | 否 |
| 交稿模式 | 乾淨台詞 | 否或極少 | 可保存為 FINAL 候選 | 視 QA |
| Codex 執行模式 | 將已確認結論寫入 repo | 必須列修改摘要 | 可保存為修改紀錄 | 否 |
| 人類裁決模式 | 記錄例外、保留與定稿決定 | 可含裁決理由 | 可保存為裁決紀錄 | 可確認 FINAL |

### 2.4.5 交稿模式輸出限制

交稿模式只能輸出：

- 文件頭。
- 場景摘要。
- 台詞正文。
- 玩家選項 / 分支。
- 必要版本紀錄。

交稿模式不得混入：

- 診斷分析。
- 創作思路。
- 破格風險說明。
- 未確認 TODO。
- AI 自評。
- 額外世界觀補充。

# 3. 通用文件頭格式

所有新增 `.md` 文件，除非使用者明確排除，必須包含以下文件頭：

```md
# 文件名稱

狀態：DRAFT / REVIEW / LOCKED / DEPRECATED
版本：v0.1
最後更新：YYYY-MM-DD
適用範圍：全作品 / CHxx / CHxx_Sxx / 角色名 / 文件類型
優先級：高 / 中 / 低

若本文件與舊聊天紀錄衝突，以本文件為準。
```

## 3.1 狀態定義

| 狀態 | 意義 | AI / Codex 可做事項 |
|---|---|---|
| DRAFT | 草稿 | 可提出大幅修改、補充、重組 |
| REVIEW | 審核中 | 可提出建議與局部修改，但需標註不確定處 |
| LOCKED | 已鎖定 | 不得擅自修改，只能引用與遵守 |
| DEPRECATED | 棄用 | 不得引用，除非使用者明確要求回顧舊版 |

## 3.2 優先級定義

| 優先級 | 意義 |
|---|---|
| 高 | 生成與修稿時必須優先遵守 |
| 中 | 應遵守，但可被高優先級文件覆蓋 |
| 低 | 參考性文件，僅在相關任務中使用 |

# 4. 開發模式輸出格式

開發模式用於討論、分析、試寫前推敲。  
此模式可以包含分析，但不得假裝是 final 版。

## 4.1 使用時機

使用於：

- 討論場景方向
- 設計角色聲線
- 分析台詞問題
- 比較不同寫法
- 建立任務包前的概念整理
- 將創意轉成可執行規格

## 4.2 標準格式

```md
# 開發模式輸出

## 任務理解

簡述本次任務要解決什麼問題。

## 目前可用資訊

-
-
-

## 缺漏資訊

-
-
-

## 建議方向

1.
2.
3.

## 風險提醒

-
-
-

## 下一步建議

-
```

## 4.3 開發模式注意事項

- 可提出問題，但不要過度依賴反問。
- 若資料不足，可先列出缺漏。
- 若缺漏不影響初步分析，可提出低污染暫定假設。
- 不得將暫定假設寫成正式設定。
- 不得把討論結論直接視為 LOCKED 文件。

常用版本方向：

- 最冷

- 最有潛台詞
- 最像兩人都在說謊
- 最像吵小事掩蓋大事
- 最接近正式交稿
- 更符合角色A聲線
- 更符合角色B聲線

- 每版必須有明確差異，不得只是換詞。

- 每版都必須遵守禁止透露資訊。
- 設計註解不得混入正式台詞。
- 若要交給引擎或劇本格式，必須另行輸出交稿模式版本。

# 6. QA 模式輸出格式

## 6.1 使用時機

- 台詞初稿檢查
- 修稿後複檢
- 角色聲線一致性檢查
- 任務包完整性檢查
- 禁用詞檢查
- 資訊揭露檢查
- AI味檢查
- 場景功能檢查

## 6.4 QA 模式注意事項

- 若任務要求「只檢查」，不得改寫原文。
- 必須指出具體位置或具體句子。
- 不得只說「有 AI 味」，必須說明問題類型。
- 若問題來自任務包缺漏，應指出缺漏，而不是硬改台詞。
- 若無法判斷，應標註「需要人類審美裁決」。

# 8. 壓縮模式輸出格式

壓縮模式用於縮短台詞、刪除說明與提高密度。

## 8.1 使用時機

使用於：

- final 前最後整理
- 遊戲 UI 字數限制
- 戰鬥中短句
- NPC 短對話
- 手機畫面短文本
- 過場節奏過慢
- 台詞中有過多解釋

# 9. 交稿模式輸出格式

交稿模式用於最終可交付台詞。  
此模式必須乾淨，不得混入分析與註解。

## 9.1 使用時機

使用於：

- final 台詞
- 要貼進劇本主檔
- 要交給引擎、編輯或團隊
- 使用者明確要求「只輸出台詞」
- 使用者明確要求「不要分析」

---

## 9.2 標準格式

```md
# 場景ID_台詞_final

狀態：REVIEW / LOCKED
版本：v1.0
最後更新：YYYY-MM-DD
適用範圍：場景ID
優先級：高

角色A：  
角色B：  
角色A：  
角色B：  
```

---

## 9.3 純台詞格式

若使用者要求「只輸出台詞」，則輸出：

```md
角色A：  
角色B：  
角色A：  
角色B：  
```

不得輸出：

- 設計註解
- 潛台詞說明
- 修改摘要
- 風險說明
- QA 評分
- 額外建議
- 問候語

---

## 9.4 交稿模式注意事項

- 不得在台詞後補充「以上是版本」。
- 不得加入未指定的註解。
- 不得自動加入動作描寫，除非格式允許。
- 不得加入玩家看不到的心理說明。
- 若需要動作、停頓、音效，必須依專案格式輸出。

# 14. 最終原則

輸出格式的目標是讓創作流程可追蹤、可比較、可修正、可交付。

因此：

- 討論稿要能看出思考方向。
- 試寫稿要能比較版本差異。
- QA 報告要能定位問題。
- 修稿稿要能看出改動理由。
- 交稿稿要乾淨，不混入分析。
- Codex 修改要能追蹤檔案與影響範圍。

若使用者要求「正式交稿」，輸出必須乾淨。  
若使用者要求「開發討論」，輸出必須可診斷。  
若使用者要求「Codex 任務」，輸出必須可直接貼進 Codex 執行。  

格式是為了保護創作，不是取代創作。
