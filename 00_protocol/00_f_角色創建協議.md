# 00_f 角色創建協議

## 1. 文件目的

本協議規範 `/create-character` / `/建立角色` 在角色實體創建過程中的執行流程。

本協議用於建立 `C-<name>` 實體，並產出角色聲線卡、偏移檢查、合規檢查與下游 QA 所需基線。

本協議處理：

- 角色類型分類
- 聲線測試題
- 去名測試前置
- 與 W-rules / V / W-language 合規性檢查
- 髒話來源欄位
- 偏移檢查欄位
- 聲線污染檢查
- 與類型氣質合規檢查
- 既有劇本台詞聲線基準萃取（§B）
- 既有劇本聲線使用規則（§C）
- 拆分寫入規則

本協議不建立關係、不寫主線、不寫場景、不寫正式台詞。

## 2. 啟動條件

使用者以 `/create-character` 或 `/建立角色` 啟動時，agent 必須先確認下列條件：

| 條件 | 要求 | 不符合時處理 |
|---|---|---|
| 世界觀規則 | `W-rules` 至少 REVIEW | 啟動拒絕，或走 `prereq_waived: true` 共通流程 |
| 詞彙系統 | `V` 至少 REVIEW | 啟動拒絕，或走 `prereq_waived: true` 共通流程 |
| 語言規格 | `W-language` 至少 REVIEW | 啟動拒絕，或走 `prereq_waived: true` 共通流程 |
| 角色名稱 | 指令中明示要建立或補完的角色名稱 | 要求使用者補角色名 |
| 寫入狀態 | 目標聲線卡不是 LOCKED | 若 LOCKED，停止並請使用者確認 |
| issue registry | `<instance_root>/issue_type_registry.yaml` 可讀；若不存在，從 `_design/registries/issue_type_registry.template.yaml` Template fallback + WARN；對應 skill key `00_f_character` 必須存在 | YAML parse error、缺 `00_f_character` skill key、議題 id 違規、locked 欄非 bool 等 schema 異常時，拒絕進階段 2 |

使用者可一次指定多個角色。agent 不得擅自決定要建哪些角色。

## 3. 階段 1：自由輸入診斷

### 3.1 開場提示

啟動後，agent 必須先說：

> 我們要建立 `<角色名清單>`。請貼出你對這 `<N>` 個角色的長段想法，包含身份、年齡、職業、外觀、語言習慣、人格、創傷、目標、與其他角色的關係。關係內容此輪只會記為提及，不建立關係。可以全部貼一起，也可以一個一個來。

### 3.2 診斷方式

使用者貼完後，agent 依 `00_protocol/00_a_台詞生產協議.md` §3.3 診斷模式，對每個角色輸出子診斷段。

每個角色至少診斷：

| 診斷項 | 內容 |
|---|---|
| 身份語感推測 | 該角色的階級、陣營、教育程度、用詞可能來源 |
| 聲線初步輪廓 | 句長、攻防傾向、情緒遮掩方式 |
| W-rules 初判 | 角色設定是否違反世界規則 |
| V 初判 | 角色用詞是否涉及禁用或未解禁詞 |
| W-language 初判 | 是否符合世界語言層級與陣營語感 |
| 邊界提醒 | 哪些內容屬關係、主線、場景、台詞，暫不寫入 |

### 3.3 診斷結尾

診斷報告結尾必須預告：階段 2 會對每個角色跑 registry core 預設 10 個 user-facing 議題；最終議題清單依 `issue_type_registry.yaml` 的 `user_extensions` / `core_overrides` 動態構建。

若同輪有多個角色，agent 必須詢問使用者採用：

- 深度優先：逐角色跑完最終 user-facing 議題清單再換下一個角色
- 廣度優先：同一議題橫掃所有角色，再進下一題

N > 3 時，agent 必須提示廣度優先較易比較聲線差異。

## 4. 階段 2：逐題探索順序

## 4.0 議題清單動態構建（D-047 / Contract D §4a.2）

階段 2 啟動時，agent **必須**讀 `<instance_root>/issue_type_registry.yaml` 的 `00_f_character` skill key 區段，按 **`core + user_extensions − core_overrides`** 動態構建議題清單：

1. **core 議題**先依 `id` 升序排列（id 1-99 為 core 保留範圍）。
2. **`user_extensions[*]`** 接在 core 議題後（id ≥ 100 為 user 自訂議題保留範圍）。
3. 套用 **`core_overrides[*].skip_id`** 過濾：
   - 對應 core entry `locked=true` → agent **忽略**該 override 並 **WARN**「locked 議題 id={n} 不可 SKIP，core_overrides 條目被忽略」；該議題仍出現在清單
   - 對應 core entry `locked=false` → 從議題清單中移除
   - `skip_id` 不存在於 core → WARN「core_overrides skip_id={n} 找不到對應 core 議題；條目被忽略」
4. 對每個議題：
   - `required_level=REQUIRED` → 跳階段 4 前**必答**；缺漏拒絕進階段 4
   - `required_level=STRONGLY_PREFERRED` → 跳階段 4 允許但 phase_log 紀錄缺漏 + 對應分拆檔段落標 TODO
   - `required_level=OPTIONAL` → 跳階段 4 允許不紀錄
5. `core[*].question_summary` 只作為 opener；agent 完整提問腳本詳本檔 §10 對應段（沿用 UD §1.2.2 完整內容）。
6. **拆分規則不入 registry** — 屬 agent 階段 4 寫檔 mechanic，由 §10.9 固定執行，不對 user 提問。

**Registry 異常處理：**
- 若 registry 載入失敗（YAML parse error / 缺 `00_f_character` key 等）→ agent **拒絕進階段 2**，請 user 修補 registry
- 若 `user_extensions[*].id < 100` 或 與 core id 重複 → parser ERROR；agent 拒絕進階段 2

## 4.1 議題順序預設表

00_f core 共 **10 個 user-facing 議題**（registry 對應 id 1-10）。若 user 在 `user_extensions:` 加自訂議題，按上述 §4.0 規則接於 core 後。

| 順序 | 議題 | required_level | locked | 為什麼這時問 |
|---|---|---|---|---|
| 1 | 角色類型分類 | REQUIRED | true | 先決定角色模板與檔案分流，後續聲線卡密度才有基準。 |
| 2 | 聲線測試題（5 個極端場景） | REQUIRED | true | 以極端情境建立角色聲線核心，供 09_b 與後續台詞引用。 |
| 3 | 去名測試前置 | STRONGLY_PREFERRED | false | 先檢查拿掉名字後是否仍能分辨角色聲音。 |
| 4 | 與 W-rules / V / W-language 合規性檢查 | STRONGLY_PREFERRED | false | 角色設定若違反世界規則或詞彙系統，後續台詞會連鎖失準。 |
| 5 | 髒話來源欄位 | STRONGLY_PREFERRED | false | 咒罵與強感嘆最容易混同聲線，需要及早定基線。 |
| 6 | 偏移檢查欄位 | STRONGLY_PREFERRED | false | 直接形成 00_b §5 與 09_b 的角色偏移判定基線。 |
| 7 | 聲線污染檢查 | OPTIONAL | false | 補充辨識角色是否容易被寫成系統、旁白或其他角色。 |
| 8 | 與類型氣質合規檢查 | STRONGLY_PREFERRED | false | 避免角色設定把作品拖向 00_b 已列出的類型偏移。 |
| 9 | 既有劇本台詞聲線基準萃取 | STRONGLY_PREFERRED | false | 讀 _source_materials/dialogue/ + 08（只讀）+ speaker alias，擷取 8-12 句覆蓋 8 場景類型，寫入聲線卡 §B，供 /dialogue-write 對齊文筆。 |
| 10 | 既有劇本聲線使用規則 | STRONGLY_PREFERRED | false | 由 §B 基準歸納可複用 / 需調整 / 不可直接當台詞，寫入聲線卡 §C。 |

**§10.9 拆分規則**屬 agent-side execution mechanic（非 user-facing 議題），階段 3 收斂預告稿時固定列入寫檔計畫；user 只需在預告稿確認，不對拆分規則本身提問。詳本檔 §10.9。

agent 必須依 §4.0 動態構建後的清單提問，不得跳到拆分或寫檔。

每個議題都必須包含：

- 為什麼問
- agent 怎麼問
- 使用者預期答什麼
- agent 怎麼整理寫檔
- 拒答 / 跳題處理

## 5. 階段 3：收斂與聲線卡草稿

階段 2 完成後，agent 必須對每個角色產一份「聲線卡草稿」貼回 chat 供使用者檢視。

多角色同輪建立時，收斂預告稿必須包含：

| 項目 | 要求 |
|---|---|
| 角色列表 | 每個角色的等級、功能定位、預計檔案位置 |
| 聲線卡草稿 | 每個角色一份，含聲線輪廓與範例 |
| 去名測試結果 | 實際對比跨角色聲線；若太相近，標 CONFLICT |
| 合規問題 | 列出 W-rules / V / W-language 衝突 |
| 00_b §5 預覽 | 顯示每個角色會 append 的偏移檢查子節 |
| 09_a 補充預覽 | 顯示角色特定高風險句型 |

未經使用者明確確認，agent 不得進入階段 4。

## 6. 階段 4：拆分執行

使用者確認後，agent 才可依區段 10.9 寫入。

固定寫入順序：

1. 角色聲線卡，依角色類型分流到 `03_characters/main/`、`03_characters/minor/` 或 `03_characters/npc/`
2. `02_vocabulary/02_b_俗稱與黑話表.md`，只 append 該角色專屬詞
3. `02_vocabulary/02_c_禁用詞與慎用詞表.md`，只 append 該角色相關禁用或慎用詞
4. `04_relationships/04_a_角色關係矩陣.md`，只補該角色 row 標題，不建立關係內容
5. `05_plot/05_c_角色弧線表.md`，只補該角色 row 標題，不建立弧線內容
6. `00_protocol/00_b_反ai味檢查表.md` §5，append 該角色偏移檢查清單
7. `09_quality_assurance/09_a_ai味qa報告模板.md` 角色特定高風險表，append 該角色子節

本階段不得把角色資料寫入關係、主線、章節、場景或正式台詞檔。

## 7. 階段 5：驗證與狀態回報

拆分完成後，agent 必須回報：

| 項目 | 預期 |
|---|---|
| C-<name> | 新增或補完為 DRAFT |
| 聲線卡 | 已含聲線輪廓、範例、合規檢查、偏移檢查 |
| 00_b §5 | 已 append 該角色偏移檢查清單 |
| 09_a | 已 append 該角色高風險句型，若有 |
| 04_a / 05_c | 只補 row 標題，不含關係或弧線內容 |

建議下一步：

- 若主要角色未齊，繼續 `/create-character`
- 若主要角色齊全，進入 `/create-relationship`
- 若關係也齊全，後續再進大綱或細綱流程

## 8. 禁止事項

本協議執行期間禁止：

1. 在本協議建立關係內容。
2. 在本協議寫主線、章節、場景或正式台詞。
3. 擅自把角色升為主角。
4. 把使用者只提到一次的關係推論寫入 `04_a` 或 `04_b`。
5. 把角色聲線寫成旁白、系統提示或劇情總結。
6. 跳過聲線測試後仍標示該角色已通過聲線驗證。
7. 對 LOCKED 聲線卡直接覆寫。
8. 新增未定義的 entity 類型或 enum 值。

| 缺漏項 | 處理 |

| 議題 1 缺 | 拒絕進入階段 3，因無法決定模板 |
| 議題 2 缺 | 聲線卡標 `TODO(聲線測試未跑)`，狀態保持 DRAFT |
| 議題 3 缺 | 聲線卡標 `TODO(去名測試未跑)`，下游 09_b 需補測 |
| 議題 4 缺 | chat 印警告；聲線卡標合規檢查未完成 |
| 議題 5 缺 | 可跳，依作品髒話尺度決定是否標 TODO |
| 議題 6 缺 | 不寫入 00_b §5 該角色子節；09_b 缺偏移基線 |
| 議題 7 缺 | 09_a 缺角色特定高風險句型 |
| 議題 8 缺 | 09_f 會標該角色未做類型氣質合規 |
| 議題 9 缺 | 聲線卡 §B 既有劇本台詞聲線基準留 TODO；無既有劇本 source，/dialogue-write 缺最直接文筆對齊來源 |
| 議題 10 缺 | 聲線卡 §C 既有劇本聲線使用規則留 TODO；無 §B 基準時連帶留空 |

## 10. 專屬議題腳本

### 10.1 角色類型分類

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 決定該角色用哪種模板，影響聲線卡欄位密度與 QA 嚴格度。 |
| agent 怎麼問 | 角色 `<角色名>` 屬於哪個層級：主角級、副主 / 反派 / 夥伴、NPC 群類？同時請用一句話說明這個角色在作品裡的功能定位。 |
| 使用者預期答什麼 | 一個分類，外加一句功能定位。 |
| agent 怎麼整理寫檔 | 分類決定寫入 `03_characters/main/`、`03_characters/minor/` 或 `03_characters/npc/`；功能定位寫入聲線卡「角色定位」。 |
| 拒答 / 跳題 | 必答；若使用者堅持跳，預設中度模板並標 INFERENCE。 |

### 10.2 聲線測試題

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 透過極端場景測試聲線是否一致。 |
| agent 怎麼問 | 請以 `<角色名>` 身分回答 5 題，每題 1-3 行：失去重要物品、被陌生人冒犯、面對不想做但必須做的事、發現自己誤判、獨自一人時的自言自語或動作。 |
| 使用者預期答什麼 | 5 段具體台詞或動作。 |
| agent 怎麼整理寫檔 | 將分析結果寫入「聲線輪廓」，原始回答寫入「聲線範例」；若 5 段調性差異過大，標 `CONFLICT(聲線測試題不一致)`。 |
| 拒答 / 跳題 | 可跳，但聲線卡必須標「聲線測試未跑，下游 09_b 缺基線」。 |

### 10.3 去名測試前置

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 確保拿掉名字仍能分辨角色聲音。 |
| agent 怎麼問 | 讓使用者選：a. 使用者給情境，agent 用每個角色口吻寫一段；b. 使用者直接寫同一句話在不同角色口中的版本；c. 跳過。 |
| 使用者預期答什麼 | 選 a / b / c；若選 a，提供情境；若選 b，提供範例。 |
| agent 怎麼整理寫檔 | 寫入聲線卡「去名測試紀錄」。若兩個角色分不開，標 `CONFLICT(聲線辨識度)` 並回到 10.2。 |
| 拒答 / 跳題 | 可跳，但下游 09_b 檢查時需重跑去名測試。 |

### 10.4 與 W-rules / V / W-language 合規性檢查

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 先攔截角色設定與世界規則、詞彙、語言層級的衝突。 |
| agent 怎麼問 | agent 用角色聲線輪廓、身份與階級跑三層初判：用詞 vs V、概念 vs W-rules、層級 vs W-language / 01_c，並請使用者逐項確認或修正。 |
| 使用者預期答什麼 | 對初判逐項回應 OK 或修正。 |
| agent 怎麼整理寫檔 | 寫入「合規檢查紀錄」與「詞彙白名單 / 黑名單」；若有衝突，標 `CONFLICT(合規)`。 |
| 拒答 / 跳題 | 跳過時標警告；若不可跳但拒答，agent 只能用 INFERENCE 填初判。 |

### 10.5 髒話來源欄位

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 髒話、強感嘆、咒罵語感是角色聲線容易混同的地方。 |
| 前置檢查 | 先看 `00_b §2 髒話尺度`。低尺度時詢問替代反應；中尺度跑完整提問；高尺度加問強度上限。 |
| agent 怎麼問 | 觸發點、頻率、強度三題，並請列出 3-8 個慣用髒話或感嘆詞。 |
| 使用者預期答什麼 | 三子題與詞清單。 |
| agent 怎麼整理寫檔 | 寫入「髒話來源」段；詞清單 append 到 `02_b §<角色名>專屬詞`。若與禁用詞衝突，標 CONFLICT。 |
| 拒答 / 跳題 | 可跳；低尺度作品建議跳。 |

### 10.6 偏移檢查欄位

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 建立該角色不得偏移成什麼、必須保留什麼，供 00_b §5 與 09_b 使用。 |
| agent 怎麼問 | 兩題：a. 這個角色不得偏移成什麼，列 3-5 個。b. 這個角色應該保留什麼，列 3-5 個。 |
| 使用者預期答什麼 | 兩個 3-5 項清單。 |
| agent 怎麼整理寫檔 | 寫入聲線卡「偏移檢查」段；同時 append 到 `00_b §5 §<角色名>`。 |
| 拒答 / 跳題 | 強烈建議不跳；若跳，09_b 只能用聲線輪廓作基線。 |

### 10.7 聲線污染檢查

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 檢查角色是否容易被誤寫成系統、AI 助理、旁白或其他角色。 |
| agent 怎麼問 | 兩題：a. 這個角色容易被誤寫成什麼聲音。b. 被誤寫時最常見徵兆是什麼。 |
| 使用者預期答什麼 | 兩子題各一段。 |
