# 00_i_專案初始化協議

若本文件與舊聊天紀錄衝突，以本文件為準。  
本文件規範從 Template repo 建立 Instance repo 的一次性 Bootstrap 流程。  
本文件只處理專案初始化，不建立世界觀、角色、關係、大綱、細綱、任務包、台詞或 QA 報告。

---

# 1. 文件目的與適用範圍

本協議規範 `/init-project` skill 在「Instance 專案初始化」過程中的執行流程。協議定義 agent 應如何引導使用者從 Template repo 建立具體作品 Instance，並完成最小可追蹤的 `.protocol_version` 紀錄。

本協議適用於：

- 使用者已從 Template repo 建立一份新的 Instance repo。
- Instance 尚未寫入作品世界觀、角色、關係、主線、細綱或台詞。
- 使用者需要建立作品名、Template 來源、Bootstrap 日期與可微調文件紀錄。

本協議不適用於：

- 已經完成 Bootstrap 的 Instance 再次初始化。
- 作品世界觀建立；應使用 `00_protocol/00_e_世界觀創建協議.md`。
- 角色、關係、大綱、細綱建立；應使用對應 `/create-*` 協議。
- Template repo 本身的設計修補。

# 2. 啟動條件

agent 在進入任何階段前必須驗證：

1. **目前資料夾是 Instance repo 根目錄。**
   - 必須存在 `AGENTS.md`、`README.md`、`00_protocol/`、`01_world/` 至 `09_quality_assurance/`。
2. **Template 基礎檔已存在。**
   - `00_protocol/00_b_反ai味檢查表.md`
   - `00_protocol/00_c_台詞輸出格式.md`
   - `00_protocol/00_d_工作流總覽.md`
3. **尚未完成 Bootstrap。**
   - 若 `.protocol_version` 已存在且含 `phase: bootstrap` / `status: completed`，agent 必須停止，請使用者改用後續 `/create-world` 或手動審查現有初始化紀錄。
4. **尚未寫入作品內容。**
   - 若 `01_world`、`03_characters/main`、`05_plot`、`07_scene_tasks`、`08_dialogue_outputs` 等已有明顯作品內容，agent 必須提示「此 repo 可能已不是空 Instance」，請使用者確認是否仍要初始化。
5. **目前資料夾不是 Template repo（D-049 第一道防線：顯式 marker）。**
   - 不得存在 root level `.template_root` 標記檔。若存在，agent 必須停止並印「⏸ 條件未滿足 — 此目錄是 Template repo」：

     ```
     **What：** 偵測到 `.template_root` 標記檔；此目錄被標識為 Template repo。
     **Where：** repo root（cwd）
     **Why：** 不允許在 Template repo 上跑 /init-project（會污染 Template）。
     **下一步：** 若此目錄已 git clone 為新 Instance 並決定初始化：
       1. 移除 `.template_root` 標記檔（`rm .template_root` 或在 IDE / file explorer 內刪檔）
       2. 重新跑 `/init-project`
     若此目錄是 Template repo 本身：請先 `git clone <template-url> <new-project-dir>` 到新目錄後在新目錄內跑。
     ```
6. **~~registries template 殘留 + 未 bootstrap 推斷~~（**D-051 supersede — 移除**）。**
   - 原 D-049 第二道防線在 fresh Instance clone 場景必然 false-positive（兩條件對「Template 被誤刪 marker」與「Fresh Instance clone 主動刪 marker」file-state 完全相同，無法區分）→ 反而擋下合理 bootstrap。

# 3. 階段 1：診斷模式

agent 開場固定行為：

1. 印出 Bootstrap 先決檢查摘要。
2. 邀請使用者貼出新專案基本資料。
3. 依 00_a 診斷模式產出診斷報告，但不寫檔。
4. 預告階段 2 將討論 00_b/00_c/00_d 是否需要專案化微調。

## 3.1 使用者需提供資料

| 欄位 | 說明 | 是否必填 |
|---|---|---|
| 作品名 | Instance 的作品名稱 | 是 |
| repo 名稱 | 本地或遠端 repo 名稱 | 是 |
| Template 來源 | Template repo URL 或來源描述 | 是 |
| Template commit | Template commit SHA；未知時依 §9.2 守衛偵測，無法確證才標 TODO | 建議 |
| Bootstrap 日期 | 預設今天 | 是 |
| 作品類型初步描述 | 一句話描述即可，正式世界觀由 00_e 處理 | 建議 |
| 預期篇幅 / 規模 | 章節數、場景量級或製作規模 | 選填 |
| 語氣偏好 | 僅作 00_b 微調候選，不視為正式設定 | 選填 |
| 輸出格式偏好 | 僅作 00_c 微調候選 | 選填 |

## 3.2 診斷報告格式

```md
# Bootstrap 診斷報告

## 一、已確認基本資料

## 二、缺漏資料

## 三、可能需要微調的 Template 文件

## 四、不可在 Bootstrap 階段處理的內容

## 五、建議下一步
```

診斷階段禁止寫檔，禁止把使用者的作品類型描述寫成正式世界觀。

# 4. 階段 2：探索 / 微調候選

Bootstrap 允許微調的 Template 文件限定為：

| 文件 | 可微調內容 | 不可微調內容 |
|---|---|---|
| `00_protocol/00_b_反ai味檢查表.md` | 作品名占位、初步類型語氣提示、髒話尺度候選、死亡處理偏好候選 | 不得新增正式世界規則、角色、場景 |
| `00_protocol/00_c_台詞輸出格式.md` | Instance 想使用的輸出格式提示、命名偏好、交稿格式備註 | 不得新增 skill，不得建立引擎匯出格式框架 |

- 00_a 台詞生產協議。
- 00_e–00_l 的全部創建、迭代與下游流程協議。

- scripts 目錄。

範例：

> 第一輪先確認三件事：

> 1. 作品名要寫入 `.protocol_version` 的 `project_name` 欄位，請確認精確名稱。

> 2. 你是否要在 00_b 先放入「初步語氣偏好」作為 TODO？

>    影響：`00_protocol/00_c_台詞輸出格式.md`

# 5. 階段 3：收斂模式

agent 把階段 1–2 累積結果整合成「Bootstrap 收斂預告稿」。

預告稿必含：

1. **基本資料最終版**
   - project_name
   - template_source
   - template_commit
   - bootstrap_date
2. **微調清單**
   - 僅可列 00_b / 00_c / 00_d
   - 每項列出 file / type / note / action
3. **不處理清單**
   - 使用者提到但不屬 Bootstrap 的內容，例如角色設定、世界規則、大綱、台詞片段。
4. **將寫入的檔案**
   - `.protocol_version`
   - 可能修改的 00_b / 00_c / 00_d
5. **TODO / INFERENCE / CONFLICT 統計**

使用者必須回覆下列其中一種：

- `通過` / `OK` / `寫檔`：進階段 4。
- 指定修改某項：agent 修改後重印收斂預告稿。
- `中止` / `取消`：不寫檔，若 `.protocol_version` 不存在則不新增；若已進入部分執行則 rollback。

未取得明確「通過」前不得寫檔。

# 6. 階段 4：Codex 執行模式

階段 4 只可執行以下寫入：

1. 寫入 `.protocol_version`。
2. 依使用者已拍板的微調，局部修改 `00_protocol/00_b_反ai味檢查表.md`。
3. 依使用者已拍板的微調，局部修改 `00_protocol/00_c_台詞輸出格式.md`。
4. 依使用者已拍板的微調，局部修改 `00_protocol/00_d_工作流總覽.md`。
5. **從 Template 複製三 registry 到 Instance root**（對齊 D-043 + D-047；詳 §10.5）：
   - `_design/registries/entity_type_registry.template.yaml` → `<instance_root>/entity_type_registry.yaml`
   - `_design/registries/qa_type_registry.template.yaml` → `<instance_root>/qa_type_registry.yaml`
   - `_design/registries/issue_type_registry.template.yaml` → `<instance_root>/issue_type_registry.yaml`
6. **建立 `10_art_assets/` 目錄結構**（對齊 D-044 7 subtype + Contract A.3；詳 §10.6）：
   - 7 個 subtype 子目錄：`portrait/` / `bg/` / `cg/` / `sfx/` / `bgm/` / `voice/` / `ui/`
   - 7 個 subtype 索引檔（空 stub）：`portrait/index.md` ~ `ui/index.md`
7. **建立 Instance root `.gitignore`** — 至少含 `export/` 一行（對齊 D-038 附帶第 3 項：export 產物不入版控）。

## 6.1 寫檔順序

1. 先寫 `.protocol_version`。
2. 再從 Template 複製三 registry → Instance root（§6 step 5；step 5 失敗時 rollback `.protocol_version`，不繼續後續步驟）。
3. 建立 `10_art_assets/` 目錄結構（§6 step 6）。
4. 寫 Instance root `.gitignore`（§6 step 7）。
5. 再套用 00_b / 00_c / 00_d 微調。
6. 最後重新讀取上述全部檔案 + 目錄，確認檔案存在且 header 未破壞。

# 9. 缺漏處理規則

## 9.1 必填資料缺漏

若作品名、Template 來源或 Bootstrap 日期缺漏，agent 不得進階段 4。

處理方式：

- 標 TODO。
- 在收斂預告稿列為阻塞。
- 請使用者補齊後重印預告稿。

## 9.2 Template commit 未知

Template commit 偵測順序（須守衛，避免把 Instance 自身 commit 誤當 Template commit）：

1. 確認 git 可用且 Instance 有 `.git`。
2. 確認 Instance 自 clone 後**尚未產生任何本地 commit**——此時 `git rev-parse HEAD` 仍指向 clone 下來的 Template commit。盡量以 `git remote -v` 對應 `template_source`、或確認 HEAD 等同 remote 追蹤分支，佐證該 SHA 確為 Template commit。
3. 僅在上述全部成立時，才把該 SHA 寫入 `template_commit`（確定值）。

任一條件不成立（git 不可用 / `.git` 缺失 / 已有本地 commit / 無法確證 HEAD 即 Template commit），一律 fallback：

```yaml
template_commit: TODO
```

並在驗證報告中以 WARN 提示使用者補回實際 Template SHA（呼應 §8 #9：不得把無法確證的 SHA 當確定值）。

## 9.3 使用者提供超出 Bootstrap 的內容

若使用者在初始化時貼入角色、世界觀、主線或台詞草稿，agent 不得寫入 01–09 文件。

處理方式：

- 在「不處理清單」列出。
- 建議下一步用對應 `/create-*` 協議處理。
- 不把內容保存到正式文件。

## 9.4 微調衝突

若使用者要求微調 00_b/00_c/00_d 的內容與既有 Template 規則衝突，agent 必須標 CONFLICT，請使用者確認是否仍要作為 Instance 專屬偏好保留。
