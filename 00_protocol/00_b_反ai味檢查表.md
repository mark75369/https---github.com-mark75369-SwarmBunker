# 00_b_反ai味檢查表

若本文件與舊聊天紀錄衝突，以本文件為準。

本文件是 Template repo 的反 AI 味與類型偏移 QA 通用骨架，不綁定任何特定作品、世界觀、角色、陣營或劇情。
本文件中的範例只示範檢查方法，不代表任何作品的最終語氣；Instance bootstrap 與上游協議完成前，不得把範例當成作品設定。

# 0. 使用方式

本文件分成兩層：

1. `## 1` 至 `## 7` 是 SPEC §17.1 鎖定的作品專屬寫入區，Template 端只放佔位說明。
2. `# 8` 之後是通用 QA 檢查邏輯，供所有作品在台詞生成、修稿、QA 與 Codex 檢查時引用。

Instance repo 在 `/init-project` 與上游 `00_e`、`00_f`、`00_g`、`00_h` 協議跑完後，才可依 SPEC §17.2 把第 1 至第 6 節逐步擴充為作品專屬內容；第 7 節由日常使用或人類手動累積。

本檔不是台詞生產流程；流程層見：

- `00_protocol/00_a_台詞生產協議.md`
- `00_protocol/00_c_台詞輸出格式.md`
- `00_protocol/00_k_台詞生產流程協議.md`

本檔用途：

- 檢查台詞是否過度工整、禮貌、完整。
- 檢查角色是否替玩家整理劇情。
- 檢查潛台詞是否被寫成明台詞。
- 檢查角色聲線是否混同或被旁白污染。
- 檢查資訊是否被角色朗讀成設定說明。
- 檢查髒話、死亡、告別、覺醒等高風險場景是否被套語化。
- 檢查作品類型是否被台詞拖偏。
- 保護不規則但有角色生命力的句子，避免 QA 機械磨平。

## 2. 髒話尺度與死亡處理偏好

<!-- 由 /create-world（00_e）議題 10.9 子題 b/c 寫入；寫入時間戳：2026-06-03 -->

**髒話尺度：中至高。**

- 前線傭兵、外圍居民、黑市 / 狩獵隊為高尺度（髒話是基本語感的一部分）。
- 官方終端 / AI 客服 / 合約文書為零髒話（冷感制式語體）。
- 髒話來源：世俗化粗話、軍事 / 黑市黑話、厭世吐槽（**無宗教咒罵源**，本作無神無宗教）。
- 限制：髒話需有對象、壓力與功能；同一角色不應每次用同一罵法解決所有情緒；不同角色不能只靠髒話多寡區分聲線。

**死亡處理偏好：流程化為主，殘留化為輔。**

- 流程化：死亡就是死亡，少抒情；上層視死亡為「人員損耗」「績效檔案更新」。
- 殘留化：死後留下物件 / 空位 / 未完成的約定 / 對活人的影響，而非完整臨別自白。
- **避免**：煽情遺言、說太多感謝 / 對不起、直接要求對方「活下去」、把關係總結完、角色臨死突然變得過度成熟。
- 告別場景限制：可表面談小事、讓一方不知這是告別、把告別藏在命令 / 玩笑 / 爭吵 / 物件交接裡、不說再見。

# 8. AI 味的工作定義

「AI 味」不是指文字一定寫得差，而是指文字呈現出以下傾向：

- 太完整。
- 太平均。
- 太禮貌。
- 太安全。
- 太會總結。
- 太像作者在解釋。
- 太像企劃書或主題作文。
- 太像角色已經理解自己。
- 太像所有角色共用同一個說話腦袋。

AI 味的核心不是某個詞本身，而是文字缺乏：

- 角色壓力。
- 個人偏差。
- 局部不合理。
- 情緒造成的語言變形。
- 關係中的遮掩。
- 說不出口的事。
- 社會身份與階級痕跡。
- 不成熟、不禮貌、不完整的反應。

因此本檢查表不只檢查「詞」，也檢查「句型、功能、資訊、情緒、角色、場景」。

# 9. 快速總檢查

每段台詞生成後，先用以下 12 項快速檢查。

若命中 3 項以上，該段台詞應進入修稿。
若命中 5 項以上，該段台詞應重寫。
若命中 8 項以上，代表任務包或角色聲線資料可能不足，應回到前置文件修正。

| 編號 | 檢查項目 | 是否命中 |
|---|---|---|
| 1 | 角色是否直接說出自己的心理總結？ |  |
| 2 | 是否出現主題宣言式台詞？ |  |
| 3 | 是否有工整轉折句？ |  |
| 4 | 是否有過度完整、過度禮貌的段落？ |  |
| 5 | 是否所有角色拿掉名字後分不出差異？ |  |
| 6 | 是否有角色替玩家整理劇情？ |  |
| 7 | 是否用抽象詞代替具體行動？ |  |

# 10. 高風險詞檢查

以下詞彙並非永久禁止，但在 AI 生成台詞中高度容易導致抽象、通用、主題化表達。
除非角色聲線卡、任務包、作品專屬 `00_b` 或世界語言規格明確允許，預設應慎用或禁用。

- 羈絆
- 守護
- 希望
- 黑暗
- 光明
- 救贖
- 代價
- 選擇
- 真正
- 重要
- 意義
- 勇氣
- 內心
- 未來
- 過去
- 力量
- 信念
- 靈魂
- 奇蹟

## 10.2 成長總結詞

- 終於

- 我明白了
- 我懂了
- 我不再逃避
- 我會面對
- 我已經不是以前的我
- 這一次我會
- 從今以後

## 10.3 關係總結詞

- 我一直都相信你
- 我一直都在乎你
- 你是我最重要的人
- 我們早就是夥伴
- 我們的羈絆
- 我不想失去你

## 10.4 犧牲與告別高風險詞

- 活下去
- 別死
- 等我回來
- 我會回來
- 你一定要活下去
- 對不起
- 謝謝
- 再見
- 保重
- 這是我最後能做的事

## 10.5 修正原則

若出現高風險詞，不要優先做同義詞替換。

錯誤修法：

- 守護 -> 保護
- 希望 -> 光芒
- 命運 -> 宿命
- 羈絆 -> 連結

優先改成：

- 角色行動。
- 物件交接。
- 稱呼變化。
- 站位變化。
- 突然沉默。
- 不合時宜的提醒。
- 對小事的執著。
- 拒絕回答。
- 用玩笑或攻擊遮掩真心。

結構示意：

| 高風險表達 | 問題 | 修正方向 |
|---|---|---|
| 我會守護你 | 主題化、抽象 | 改成角色用站位、命令、物件或行動表現。 |
| 我一直都相信你 | 直接說出關係 | 改成角色把關鍵物品交給對方，或在爭吵中做出讓步。 |
| 你一定要活下去 | 犧牲場景套語 | 改成角色談一件小事，或用命令讓對方不要回頭。 |

此表僅示範檢查方式，不代表任何作品語氣。

# 11. 反 AI 味檢查不是反風格

反 AI 味檢查的目的不是消滅漂亮句、宣言句、情緒直說、重複句或風格化語言。

這些元素在某些作品中可能是缺陷，在另一些作品中可能是風格核心。

因此本文件所有檢查項都應理解為：

```text
風險提示，不等於絕對禁用。
```

除非某詞、句型或表達已在 `02_vocabulary/02_c_禁用詞與慎用詞表.md`、作品專屬 `00_b` 或單場任務包中明確標為禁用，否則 QA 不得自動刪除。

## 11.1 模式差異下的 AI 味判定

| 模式 | AI 味規則強度 | 處理方式 |
|---|---|---|
| 探索模式 | 低 | 可指出風險，但不限制發散。 |
| 試寫模式 | 中 | 標記高風險句，暫不強制刪除。 |
| 破格模式 | 很低 | 允許違規，重點找亮點。 |
| 收斂模式 | 高 | 修正明顯 AI 味，但保留亮點。 |
| QA 模式 | 最高 | 逐句標記問題與風險。 |
| 交稿模式 | 最高 | 不得保留未確認高風險句。 |

## 11.2 高風險句型示例，不等於禁用句

以下句型在通用情況下高風險，但不代表永遠禁用：

- 我會守護你。
- 你不是一個人。
- 這就是我的選擇。
- 我終於明白。
- 真正重要的是。
- 我不想失去你。
- 我一直都相信你。
- 你一定要活下去。
- 這就是我們的選擇。
- 這就是我們的命運。

這些句型應根據以下條件判斷：

| 判斷項目 | 問題 |
|---|---|
| 作品風格 | 本作是否允許宣言式、誓言式、情緒直說？ |
| 角色聲線 | 這個角色是否會用這種句型？ |
| 關係階段 | 此時兩人的關係是否允許說到這麼滿？ |
| 弧線階段 | 角色是否已經能理解或承認這件事？ |
| 場景壓力 | 場景是否支撐這種表達？ |
| 代價 | 這句話是否有行動或代價支撐？ |
| 重複度 | 同類句型是否已過度使用？ |

## 11.3 條件式判定範例

| 句型 | 通用風險 | 條件允許情況 | 禁用情況 |
|---|---|---|---|
| 我會守護你 | 通用、熱血套語 | 王道角色、誓言場景、後期關係節點。 | 冷感角色、前期關係、無代價場景。 |
| 你不是一個人 | 通用安慰 | 特定角色笨拙安慰、反諷、後期回收。 | 高情緒場景中替代角色真實反應。 |
| 我終於明白 | 心理總結 | 明確覺悟節點，且角色聲線允許。 | 弧線未完成、前中期、破防初期。 |
| 這就是我的選擇 | 主題宣言 | 結局選擇、角色以行動付出代價後。 | 無行動支撐、只為說主題。 |
| 真正重要的是 | 作者腔 | 特定導師、宗教、儀式語境。 | 一般角色突然總結主題。 |

## 11.4 可保留的不規則亮點

反 AI 味 QA 必須加入「可保留亮點」檢查。

```md
## 可保留的不規則亮點

| 位置 | 內容 | 為何值得保留 | 是否違反規則 | 建議處理 |
|---|---|---|---|---|
```

可保留亮點包括：

- 不漂亮但很像角色的句子。
- 語法不完整但符合情緒的句子。
- 看似突兀但能形成記憶點的反應。
- 重複但有角色執念感的詞。
- 粗糙但符合階級、職業、年齡的說法。
- 不完全合規但符合本作美學的句子。

## 11.5 不建議機械修正項

```md
## 不建議機械修正項

| 位置 | 內容 | 為何不應機械修正 | 需誰確認 |
|---|---|---|---|
```

以下情況不應由 AI 自行修掉：

- 人類明確指定保留的句子。
- 具有強烈角色感但語法不標準的句子。
- 本作風格需要的宣言式台詞。
- 角色刻意說謊、表演或自我欺騙的句子。
- 為回收伏筆而重複的句子。
- 為形成儀式感而使用的固定句。

# 12. 高風險句型檢查

以下句型容易讓台詞呈現 AI 式完整、工整、總結感。
除非任務明確要求，預設應避免。

## 12.1 對稱句型

- 不是 A，而是 B
- 這不只是 A，更是 B
- 與其說是 A，不如說是 B
- A 並非終點，而是開始
- A 不是弱點，而是力量

問題：

- 太像論述。
- 太像主題作文。
- 太像作者替角色下結論。
- 角色不一定會在壓力下說出這種工整句。

檢查問題：

- 角色此刻真的有能力整理成這麼工整嗎？
- 角色此刻是在說服對方，還是在替玩家理解主題？
- 能否改成更短、更偏執、更不完整？

## 12.2 成長句型

- 我曾經以為……但現在……
- 我終於明白……
- 我不想再逃避……
- 我會面對自己的……
- 從今以後，我會……

問題：

- 角色直接說出角色弧線。
- 角色過度理解自己。
- 戲劇變化被摘要化。

修正方向：

- 用行動取代覺悟。
- 用稱呼改變取代成長宣言。
- 用一句不完整的話取代完整自白。
- 讓旁人看出變化，而不是角色自己說出來。

## 12.3 關係確認句型

- 我一直都相信你
- 我們是夥伴
- 你對我很重要
- 我不能失去你
- 我們一起走到這裡
- 你不是一個人

問題：

- 直接確認關係。
- 缺乏遮掩。
- 容易變成情緒標籤。

修正方向：

- 改成對方才懂的舊稱呼。
- 改成爭吵中的讓步。
- 改成物件交接。
- 改成不願承認的關心。
- 改成語氣中的破綻。

## 12.4 說教句型

- 真正重要的是……
- 你必須明白……
- 這就是……的意義
- 人之所以……是因為……
- 如果我們放棄，就等於……

問題：

- 角色像旁白。
- 主題過度外顯。
- 玩家被動接受答案。

修正方向：

- 改成場景中的具體選擇。
- 改成失敗的說服。
- 改成一句有偏見的話。
- 改成角色基於自身經驗的錯誤判斷。

# 13. 心理總結檢查

AI 台詞常見問題是「直接命名情緒」，而不是讓情緒改變語言。

## 13.1 高風險心理直說

以下表達預設應檢查：

- 我很害怕
- 我很痛苦
- 我很孤獨
- 我很後悔
- 我只是想被理解
- 我不想再失去任何人
- 我不知道該怎麼辦
- 我其實一直很在乎你
- 我只是裝作不在乎

這些句子不是絕對不能用，但必須符合以下條件之一：

1. 角色聲線卡允許直白自白。
2. 場景任務包明確要求情緒說破。
3. 這是角色極罕見的破防時刻。
4. 這句話本身會造成不可逆後果。
5. 這句話不是成熟表達，而是失控、錯誤或傷人的表達。

## 13.2 情緒應如何進入語言

情緒不一定要被說出來，應優先影響：

- 句長。
- 停頓。
- 詞彙選擇。
- 主詞省略。
- 重複。
- 打斷。
- 轉移話題。
- 稱呼變化。
- 禮貌程度。
- 攻擊方向。
- 對小事的執著。

結構示意：

| 情緒 | 不佳表達 | 可改方向 |
|---|---|---|
| 擔心 | 我很擔心你 | 改成檢查對方裝備、責備對方慢、要求對方站後面。 |
| 愧疚 | 我一直很後悔 | 改成避開某個地名、拒絕談過去、過度補償。 |
| 害怕失去 | 我不想失去你 | 改成不讓對方同行、故意激怒對方離開。 |

# 14. 潛台詞檢查

## 14.1 潛台詞是否被說破

檢查每句重要台詞：

| 檢查問題 | 是 / 否 |
|---|---|
| 角色是否直接說出真實意圖？ |  |
| 角色是否直接說出真實情緒？ |  |
| 角色是否直接說出兩人關係？ |  |
| 角色是否直接說出作者想表達的主題？ |  |
| 角色是否沒有任何遮掩？ |  |

若多數答案為「是」，台詞可能缺乏潛台詞。

## 14.2 表層目的與真實目的是否分離

高品質台詞通常有兩層：

- 表層：角色正在談什麼。
- 深層：角色真正想做什麼。

檢查：

| 項目 | 說明 |
|---|---|
| 表層話題 | 角色表面上正在談的事。 |
| 真實目的 | 角色真正想達成的事。 |
| 遮掩方式 | 角色如何避免暴露真心。 |
| 對方理解 | 對方是否聽懂、誤解或假裝沒聽懂。 |
| 玩家理解 | 玩家是否能從縫隙裡讀出真相。 |

若表層與深層完全一致，且角色沒有理由直接說破，應修稿。

# 15. 髒話檢查

髒話不是強度開關，而是角色、階級、職業、年齡、地域、陣營與當下壓力的語言結果。

檢查時應先對照作品專屬 `00_b §2`，再判斷單句是否成立。

| 檢查項目 | 是 / 否 |
|---|---|
| 髒話是否符合該角色聲線？ |  |
| 髒話是否有明確情緒或攻防功能？ |  |
| 髒話是否來自作品語言規格或角色生活背景？ |  |
| 髒話是否只是為了讓台詞看起來更激烈？ |  |
| 不同角色的髒話來源是否混同？ |  |
| 髒話是否破壞作品類型語氣？ |  |

通用占位範例：

| 類型 | 可填占位 |
|---|---|
| 角色慣用髒話 | `<該角色慣用髒話 1>`、`<該角色慣用髒話 2>` |
| 陣營或職業髒話 | `<陣營 A 慣用罵法>`、`<職業群體慣用罵法>` |
| 禁止或慎用髒話 | `<本作慎用髒話 1>`、`<本作禁用髒話 1>` |

修正方向：

- 髒話應有對象、壓力與功能。
- 同一角色不應每次用同一罵法解決所有情緒。
- 不同角色不能只靠髒話多寡區分聲線。
- 若髒話會破壞場景壓力，優先改成沉默、打斷、動作或稱呼變化。

# 16. 死亡與告別處理檢查

死亡、犧牲、告別、瀕死、遺言等場景高度容易變成通用煽情句。

檢查時應先對照作品專屬 `00_b §2` 與 `00_b §6`。

高風險問題：

- 說太像遺言。
- 說太多感謝。
- 說太多對不起。
- 直接要求對方活下去。
- 把關係總結完。
- 角色突然變得過度成熟。
- 死亡被寫成主題宣言或情緒教學。

檢查方向：

- 是否可以表面談小事？
- 是否可以讓一方不知道這是告別？
- 是否可以讓告別藏在命令、玩笑、爭吵或物件交接裡？
- 是否可以讓角色不說再見？
- 是否可以讓死亡的重量留在後果、空位、未完成的約定，而不是完整自白？

通用占位範例：

| 情境 | 高風險寫法 | 可改方向 |
|---|---|---|
| `<角色 A>` 犧牲前 | 直接說「你一定要活下去」。 | 改成塞給 `<角色 B>` 一個物件，命令對方帶走。 |
| `<NPC>` 死前 | 把身世、願望、道歉一次說完。 | 改成說一個錯稱、半句未完的日常話。 |
| `<反派>` 臨終 | 突然替主題下結論。 | 改成仍維持偏執、否認、嘲諷或錯誤理解。 |

# 17. 角色聲線與污染檢查

## 17.1 去名測試

拿掉角色名後，是否仍能分辨誰在說話？

| 項目 | 結果 |
|---|---|
| 句長是否不同？ |  |
| 詞彙來源是否不同？ |  |
| 稱呼習慣是否不同？ |  |
| 情緒表達方式是否不同？ |  |
| 攻擊方式是否不同？ |  |
| 防禦方式是否不同？ |  |
| 是否有人使用不屬於自己的詞？ |  |
| 是否所有人都像同一個旁白？ |  |

若拿掉名字後無法區分角色，應回到角色聲線卡或角色關係矩陣修正。

## 17.2 聲線漂移檢查

角色在不同場景中可變化，但變化必須有原因。

合理變化原因：

- 劇情階段改變。
- 關係改變。
- 情緒壓力改變。
- 面對不同對象。
- 權力位置改變。
- 角色刻意偽裝。
- 角色破防。
- 角色失去控制。

不合理漂移：

- 無原因突然變文藝。
- 無原因突然變成熟。
- 無原因突然開始說教。
- 無原因使用其他角色的詞。
- 無原因使用不符合階級或教育背景的語氣。

檢查時應對照：

- 角色聲線卡。
- 角色關係矩陣。
- 劇情階段。
- 任務包中的情緒狀態。
- 作品專屬 `00_b §5`。

## 17.3 聲線污染檢查

聲線污染指角色被外部語氣覆蓋，導致說話方式不再像自己。

常見污染來源：

| 污染來源 | 徵兆 |
|---|---|
| 作者腔 | 角色開始替作品說主題。 |
| 系統腔 | 角色像在輸出任務簡報或規格。 |
| 旁白腔 | 角色替玩家整理情緒、關係或背景。 |
| 其他角色腔 | 使用不屬於自己的稱呼、髒話、節奏或攻擊方式。 |
| 過度成熟 | 角色突然能完整理解自己與他人。 |
| 過度禮貌 | 高壓場景仍維持無摩擦溝通。 |

檢查問題：

- 這句話是否只有 `<角色 A>` 會說？
- 如果換成 `<角色 B>`，是否幾乎不用改？
- 這句話是否更像作者、系統、旁白或 QA 評語？
- 這句話是否失去該角色的階級、職業、年齡、教育背景或關係痕跡？

## 17.4 口頭禪依賴檢查

角色差異不應只靠口頭禪。

若一個角色拿掉口頭禪後就與其他角色沒有差異，代表聲線設計不足。

應檢查：

- 該角色是否有獨特句法？
- 該角色是否有獨特詞彙來源？
- 該角色是否有獨特遮掩方式？
- 該角色是否有獨特攻擊方式？
- 該角色是否有獨特沉默方式？

# 18. 資訊控制檢查

## 18.1 是否把世界觀說成百科

檢查是否出現以下問題：

- 角色直接解釋歷史背景。
- 角色直接朗讀組織設定。
- 角色說出不自然的正式名詞。
- 角色為玩家整理人物關係。
- 角色在緊張場景中講解太多。
- 角色說了自己不需要說的背景資訊。
- 對話像任務簡報，不像角色互動。

修正方向：

- 改用俗稱。
- 改用誤稱。
- 改用片段。
- 改用偏見。
- 改用角色不知道全貌的說法。
- 改用物件或場景線索。
- 改用對方反應讓玩家推理。

## 18.2 是否提前劇透

檢查：

| 項目 | 是 / 否 |
|---|---|
| 是否出現任務包禁止說破的資訊？ |  |
| 是否出現後續章節才應出現的真相？ |  |
| 是否有角色知道不該知道的事？ |  |
| 是否稱呼暴露了尚未揭露的關係？ |  |
| 是否用了後期才該使用的詞？ |  |

若任一項為「是」，應立即修正。

## 18.3 必須透露資訊是否過於直白

即使是必須透露的資訊，也不一定要直接說明。

檢查：

- 是否可以用角色反應透露？
- 是否可以用地名、物件、傷口、缺席者透露？
- 是否可以用對方誤解透露？
- 是否可以讓玩家晚一拍才理解？
- 是否可以只透露結果，不透露原因？

# 19. 場景功能檢查

## 19.1 對話是否造成變化

檢查：

| 項目 | 說明 |
|---|---|
| 開始狀態 | 對話開始時角色、關係、資訊狀態。 |
| 結束狀態 | 對話結束時是否發生變化。 |
| 變化類型 | 關係、資訊、任務、情緒、權力、誤解。 |
| 不可逆性 | 是否有無法完全回到原點的結果。 |

若沒有變化，台詞可能只是閒聊或說明文。

## 19.2 每句是否有功能

重要台詞至少應具備一個功能：

- 攻擊
- 防禦
- 試探
- 隱瞞
- 命令
- 拒絕
- 確認
- 拖延
- 誤導
- 安撫
- 求救
- 切斷
- 挽留
- 羞辱
- 轉移話題
- 暴露破綻

檢查表：

| 台詞 | 表層功能 | 隱藏功能 | 可刪除？ |
|---|---|---|---|
|  |  |  |  |

若多句功能相同，應合併或刪除。

## 19.3 是否缺乏攻防

AI 常讓角色過快互相理解。
但劇本對話通常需要攻防。

檢查：

- 是否一方提出，另一方立刻接受？
- 是否衝突太快被解決？
- 是否角色太會說出合理立場？
- 是否缺乏誤解、迴避或反擊？
- 是否每句都像善意溝通？

修正方向：

- 加入試探。
- 加入錯誤指控。
- 加入轉移話題。
- 加入不成熟反應。
- 加入權力壓迫。
- 加入沉默或打斷。

# 20. 重複與壓縮檢查

## 20.1 詞彙重複

檢查：

| 類型 | 高頻項目 |
|---|---|
| 名詞 |  |
| 動詞 |  |
| 形容詞 |  |
| 抽象詞 |  |
| 情緒詞 |  |
| 世界觀名詞 |  |

若同一段對話重複使用同一抽象詞，應修正。

## 20.2 句型重複

檢查是否反覆出現：

- 我只是……
- 你知道……
- 也許……
- 但至少……
- 這就是……
- 如果……那麼……
- 不是……而是……
- 我曾經……但現在……

若句型重複，不應只替換詞彙，而應改變角色思考方式。

## 20.3 情緒重複

若多句都在表達同一情緒，應檢查是否可以壓縮。

例如：

- 一句表達憤怒。
- 一句表達失望。
- 一句表達受傷。
- 一句表達不信任。

如果四句其實都在說「我很痛」，可合併成更具體、更有攻防的一句。

## 20.4 可刪句檢查

每段台詞應問：

- 這句刪掉後，劇情是否仍然成立？
- 這句刪掉後，關係變化是否仍然成立？
- 這句是否只是重複上一句情緒？
- 這句是否只是替玩家整理？
- 這句是否只是讓角色顯得更會說話？

若刪除不影響功能，應刪。

## 20.5 解釋刪除優先級

優先刪除：

1. 心理總結。
2. 主題宣言。
3. 世界觀解釋。
4. 重複情緒。
5. 轉折說明。
6. 角色已經知道的資訊。
7. 玩家可自行理解的資訊。
8. 工整漂亮但無功能的句子。

優先保留：

1. 有攻防的句子。
2. 有角色聲線的句子。
3. 有關係變化的句子。
4. 有資訊功能的句子。
5. 有潛台詞的句子。
6. 不完美但像活人的句子。

# 21. 特殊場景檢查

以下分類是通用 QA 結構，不等於作品專屬 `00_b §6` 已寫入。正式判斷時應以 Instance 端 `00_b §6` 為準。

## 21.1 告別場景

高風險問題：

- 說太像遺言。
- 說太多感謝。
- 說太多對不起。
- 直接要求對方活下去。
- 把關係總結完。
- 角色突然變得過度成熟。

檢查方向：

- 是否可以表面談小事？
- 是否可以讓一方不知道這是告別？
- 是否可以讓告別藏在命令、玩笑或爭吵裡？
- 是否可以讓角色不說再見？

## 21.2 和解場景

高風險問題：

- 道歉太完整。
- 原諒太快。
- 雙方太理解彼此。
- 衝突被語言解決得太乾淨。

檢查方向：

- 是否可以只達成暫時停戰？
- 是否可以仍保留刺？
- 是否可以用行動代替道歉？
- 是否可以一方不承認自己原諒了？

## 21.3 反派理念場景

高風險問題：

- `<反派>` 像在寫論文。
- `<反派>` 直接解釋世界觀。
- `<反派>` 把理念說得太完整。
- `<反派>` 替 `<角色 A>` 指出主題。

檢查方向：

- 理念是否可以從具體選擇中呈現？
- `<反派>` 是否有私人偏執？
- `<反派>` 是否說謊或美化自己？
- `<反派>` 是否故意省略關鍵資訊？

## 21.4 主角覺醒場景

高風險問題：

- `<角色 A>` 說出自己成長。
- `<角色 A>` 突然講主題句。
- 覺醒只靠台詞，不靠行動。
- 所有角色見證並肯定 `<角色 A>`。

檢查方向：

- 是否可以用選擇表現覺醒？
- 是否可以不說出覺醒？
- 是否可以讓覺醒帶來代價？
- 是否可以讓旁人誤解 `<角色 A>` 的選擇？

# 23. Codex 使用規則

當 Codex 使用本文件進行檢查時，應遵守：

1. 若任務要求「只檢查」，不得修改台詞文件。
2. 若任務要求「修正」，應先輸出問題摘要，再修改指定文件。

# 24. 最終判斷原則

一段台詞是否有 AI 味，不只看詞彙，而看它是否缺乏活人說話時的壓力、偏差與遮掩。

檢查時應優先問：

- 這句話只有這個角色會說嗎？
- 這句話是否受當下壓力影響？
- 這句話是否有表層目的與隱藏目的？
- 這句話是否暴露了角色不想暴露的事？
- 這句話是否太像作者幫角色整理？
- 這句話是否可以更短、更偏、更不完整？
- 這句話是否雖然正確，但不像活人？

若台詞「全都對，但沒有壓痕」，仍應判定為需要修稿。

高品質台詞不是把情緒講出來，而是讓情緒改變角色說話的方式。
