# 00_e_世界觀創建協議

若本文件與舊聊天紀錄衝突，以本文件為準。  
本協議規範 `/create-world` skill 在世界觀實體創建過程中的執行流程。協議定義 agent 應如何引導使用者從零開始建立 W-rules / W-language / V 三個實體與作品專屬 00_b 骨架，並按拆分規則寫入 Bible 對應分拆檔。

---

# 1. 文件目的與適用範圍

本協議用於建立作品的世界觀、世界語言、詞彙系統與作品專屬反 AI 味基線。

本協議會產出或更新：

- `01_world/01_a_世界觀總覽.md`
- `01_world/01_b_世界語言規格.md`
- `01_world/01_c_陣營與階級語言.md`
- `02_vocabulary/02_a_專有名詞表.md`
- `02_vocabulary/02_b_俗稱與黑話表.md`
- `02_vocabulary/02_c_禁用詞與慎用詞表.md`
- `00_protocol/00_b_反ai味檢查表.md` 的 §1 與 §2

本協議不建立角色、關係、主線、章節、場景、任務包或台詞。

---

# 2. 啟動條件

啟動前 agent 必須確認：

1. Instance 已 bootstrap，且 `.protocol_version` 存在。
2. `.protocol_version.phase_log` 不存在 `phase: create-world` 且 `status: completed` 的紀錄。
   - 若存在，請使用者改跑 `/iterate-world`。
3. 目標檔案若已存在且狀態為 LOCKED，agent 不得擅自覆寫。
4. 目前沒有針對 W-rules / W-language / V 的進行中協議。
5. **`<instance_root>/issue_type_registry.yaml` 可讀**（D-047 + Contract D §4a 對齊）：
   - registry 不存在 → 從 `_design/registries/issue_type_registry.template.yaml` Template fallback + WARN
   - registry schema 異常（缺 `00_e_world` skill key / 議題 id 違規 / locked 欄非 bool 等）→ agent 拒絕進階段 2

本協議是上游第一份創建協議，沒有先決實體。

啟動失敗時，agent 必須列出缺漏與下一步，不得自動呼叫其他 skill。

# 3. 階段 1：診斷模式

agent 開場提示：

> 世界觀是這個專案的第一塊基石。請貼出一段你對這個世界的完整想法，可以包含類型、規則、能量觀、社會結構、人民生活、價值觀、宗教、語言、語感等任何你已經想過的。越長越好；空白處我會幫你補洞。

使用者貼完後，agent 依 00_a 診斷模式產出診斷報告。診斷報告只印在 chat，不寫檔。

診斷報告必含：

1. 作品類型與敘事氣質初判。
2. 世界規則可確認內容。
3. 世界語言與台詞風格推測。
4. 已確認內容。
5. 不可確認 / 需人類確認內容。
6. 建議下一步與階段 2 提問清單預告。

診斷階段禁止：

- 寫入任何檔案。
- 建立角色、主線或場景。
- 把推測寫成正式設定。
- 把使用者提到的片段自動補完成世界規則。

# 4. 階段 2：探索 / 補洞對話

## 4.0 議題清單動態構建（D-047 / Contract D §4a.2）

階段 2 啟動時，agent **必須**讀 `<instance_root>/issue_type_registry.yaml` 的 `00_e_world` skill key 區段，按 **`core + user_extensions − core_overrides`** 動態構建議題清單：

1. **core 議題**先依 `id` 升序排列（id 1-99 為 core 保留範圍）。
2. **`user_extensions[*]`** 接在 core 議題後（id ≥ 100 為 user 自訂議題保留範圍）。
3. 套用 **`core_overrides[*].skip_id`** 過濾：
   - 對應 core entry `locked=true` → agent **忽略**該 override 並 **WARN**「locked 議題 id={n} 不可 SKIP，core_overrides 條目被忽略」；該議題仍出現在清單
   - 對應 core entry `locked=false` → 從議題清單中移除
   - `skip_id` 不存在於 core → WARN「core_overrides skip_id={n} 找不到對應 core 議題；條目被忽略」
4. 對每個議題：
   - `required_level=REQUIRED` → 跳階段 4 前**必答**；缺漏拒絕進階段 4
   - `required_level=STRONGLY_PREFERRED` → 跳階段 4 允許但 phase_log 紀錄缺漏 + 對應分拆檔段落標 TODO
   - `required_level=OPTIONAL` → 跳階段 4 允許不紀錄
5. `core[*].question_summary` 只作為 opener；agent 完整提問腳本詳本檔 §10 對應段（沿用 UD §1.1.2 完整內容）。
6. **拆分規則不入 registry** — 屬 agent 階段 4 寫檔 mechanic，由 §10.11 固定執行，不對 user 提問。

**Registry 異常處理：**
- 若 registry 載入失敗（YAML parse error / 缺 `00_e_world` key 等）→ agent **拒絕進階段 2**，請 user 修補 registry
- 若 `user_extensions[*].id < 100` 或 與 core id 重複 → parser ERROR；agent 拒絕進階段 2

## 4.1 議題順序預設表

00_e core 共 **10 個 user-facing 議題**（registry 對應 id 1-10）。若 user 在 `user_extensions:` 加自訂議題，按上述 §4.0 規則接於 core 後。

| 順序 | 議題 | required_level | locked | 為什麼這時問 |
|---|---|---|---|---|
| 1 | 世界類型快速分類 | REQUIRED | true | 後續所有議題的判定基底 |
| 2 | 世界規則最小集 | REQUIRED | true | W-rules 主體 |
| 3 | 科技水平 | STRONGLY_PREFERRED | false | 影響 V 詞彙與 W-language |
| 4 | 人民生活水準 | STRONGLY_PREFERRED | false | 影響角色語感與階級語 |
| 5 | 各項價值觀 | STRONGLY_PREFERRED | false | 影響角色行為合理性 |
| 6 | 宗教 | OPTIONAL | false | 影響語感、咒罵與日常用語 |
| 7 | 語言層級切片 | REQUIRED | true | W-language 主體 |
| 8 | 陣營與階級語言 | STRONGLY_PREFERRED | false | 影響後續角色聲線 |
| 9 | 類型語氣定位 | REQUIRED | true | 決定作品專屬 00_b §1 / §2 |
| 10 | 越界禁區 | STRONGLY_PREFERRED | false | 確認本協議邊界 |

**§10.11 拆分規則**屬 agent-side execution mechanic（非 user-facing 議題），階段 3 收斂預告稿時固定執行；user 只需在預告稿確認，不對拆分規則本身提問。詳本檔 §10.11。

提問規則：

- 每次最多丟 3 題。
- 每題必附「為什麼問」與「答完會影響哪個分拆檔」。
- 使用者可說 `跳這題`、`先跳到議題 X`、`中止`。
- 每完成一題，agent 印議題進度條。
- 若答案與既有設定衝突，先討論 CONFLICT，不得繼續假裝已確認。

# 5. 階段 3：收斂模式

agent 將階段 2 累積結果整合成「世界觀收斂預告稿」。

預告稿必含：

1. 10 個 user-facing 議題的最終結論（§10.1 ~ §10.10），每題一句話；§10.11 拆分規則屬階段 4 mechanic，由 agent 列出寫檔計畫供 user 確認。
2. 拆分計畫，列出每個結論將寫入哪個檔案與段落。
3. 作品專屬 00_b §1 / §2 將被覆寫的內容預覽。
4. TODO / INFERENCE / CONFLICT 清單。
5. 本次將新建或更新的實體：W-rules、W-language、V。
6. 三個實體的 frontmatter 預設權重。
7. 不屬本協議的內容清單，例如角色、主線、場景、台詞片段。

使用者必須回覆：

- `通過` / `OK` / `寫檔`：進階段 4。
- 指定修改某項：agent 修改後重印預告稿。
- `中止` / `取消`：不寫檔；若需要，phase_log 補 aborted 紀錄。

CONFLICT 數量大於 0 時不得自動進入階段 4。

# 6. 階段 4：Codex 執行模式

階段 4 依第 10.11 節拆分規則寫檔。

寫檔順序固定：

1. `01_world/01_a_世界觀總覽.md`
2. `01_world/01_b_世界語言規格.md`
3. `01_world/01_c_陣營與階級語言.md`
4. `02_vocabulary/02_a_專有名詞表.md`
5. `02_vocabulary/02_b_俗稱與黑話表.md`
6. `02_vocabulary/02_c_禁用詞與慎用詞表.md`
7. `00_protocol/00_b_反ai味檢查表.md` 的 §1 / §2

執行規則：

- 若目標檔不存在，新建並補標準中文 header 5 欄與必要 YAML block。
- 若目標檔存在，先讀現有內容，找到本協議負責的段落 anchor 後局部覆寫或 append。
- 00_b 只可覆寫 §1 / §2，且附本次寫入時間戳。
- frontmatter 依 SPEC §5.2 canonical schema 補齊。
- 任一步失敗即 rollback 本次新建或修改的檔案，不更新 phase_log。
- LOCKED 檔案不得擅自改核心內容；需 dump blocked write 到 `_design/create_world_blocked_writes.md` 並請使用者裁決。

# 9. 缺漏處理規則

## 9.1 TODO

使用者未提供必要資訊時，寫入對應段落：

```md
<!-- TODO(<議題編號>): <一句話描述缺什麼、agent 何時可補> -->
```

TODO 必須同時列入收斂預告稿與 phase_log。

## 9.2 INFERENCE

agent 從上下文推導得出但使用者未明確確認時，寫入：

```md
<!-- INFERENCE: <推導依據>；待人類確認 -->
```

INFERENCE 不得替代 TODO。

## 9.3 CONFLICT

本次資訊與既有設定衝突時，寫入：

```md
<!-- CONFLICT(<既有檔案>:<段落>): 既有=<X>，本次想寫=<Y>。請人類拍板 -->
```

並 dump 到 `_design/create_world_conflicts.md`。CONFLICT 未解前不得進階段 4。

## 9.4 00_e 特殊缺漏

- 跳過議題 9 類型語氣定位：階段 4 不寫 00_b §1 / §2，並標 TODO。
- 跳過議題 7 或 8：階段 4 不寫 01_b / 01_c；W-language 完成度應標為 0% 或明顯不足。
- 跳過議題 10：不得進階段 3，因為邊界未確認。
