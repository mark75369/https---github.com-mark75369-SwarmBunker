# 00_h 細綱創建協議

## 1. 文件目的

本協議規範 `/create-detailed-outline` / `/建立細綱` 在章節與場景索引實體創建過程中的執行流程。

本協議用於把主線骨架展開為：

- 章節節奏分布
- 角色弧線 × 章節矩陣
- 資訊揭露時間軸
- 伏筆與回收佈點
- 高風險場景識別
- 高風險場景處理方式
- 場景索引層 `S-<ch>-<n>`

完成後主要寫入：

- `05_plot/05_b_章節結構模板.md`
- `05_plot/05_c_角色弧線表.md`
- `05_plot/05_d_資訊揭露表.md`
- `05_plot/05_e_伏筆與回收表.md`
- `06_scene_index/06_a_場景索引模板.md`
- `00_protocol/00_b_反ai味檢查表.md` §6

本協議不寫具體場景台詞、不建立場景任務包、不改章節數。

## 2. 啟動條件

使用者以 `/create-detailed-outline` 或 `/建立細綱` 啟動本流程時，agent 必須先確認：

|---|---|---|

| 詞彙系統 | `V` 至少 REVIEW | 啟動拒絕，或走 `prereq_waived: true` 共通流程 |
| 語言規格 | `W-language` 至少 REVIEW | 啟動拒絕，或走 `prereq_waived: true` 共通流程 |
| 主線 | `P` 至少 REVIEW | 啟動拒絕，或走 `prereq_waived: true` 共通流程 |
| 主要角色 | 所有主要 `C-*` 至少 REVIEW | 列出缺漏角色並請使用者裁決 |
| 規模與偏移基線 | `00_b §3` 與 `00_b §4` 已存在 | 要求先完成大綱流程 |

## 3. 階段 1：自由輸入診斷

### 3.1 開場提示

agent 啟動後必須先說：

> 細綱是把主線從一句話變成章節 × 場景的骨幹。請告訴我你希望這次做到哪個程度：(a) 章節節奏 + 弧線對齊；(b) 上述 + 場景索引；(c) 完整細綱，含資訊揭露、伏筆、高風險場景處理方式。預設跑 (c) 完整版。

若使用者已有細綱草稿，agent 要求貼出草稿後再診斷。

### 3.2 診斷重點

agent 依 `00_protocol/00_a_台詞生產協議.md` §3.3 診斷模式分析草稿。

診斷至少包含：

| 診斷項 | 檢查內容 |
|---|---|
| 節奏分布 | high / mid / low 是否過密或過散 |
| 主角弧線 | 是否與 00_g 的想要 vs 需要對齊 |
| 其他角色弧線 | 主要角色是否有章節切點或穩定狀態 |
| 資訊揭露 | 哪些資訊可能提前外洩 |
| 伏筆回收 | 是否有只埋不回、提前說破、回收太密 |
| 高風險場景 | 是否有過密的告別、覺醒、揭露、關係破裂等場景 |

### 3.3 診斷結尾

診斷報告結尾必須預告階段 2 會跑 registry core 預設 6 個 user-facing 議題；最終議題清單依 `issue_type_registry.yaml` 的 `user_extensions` / `core_overrides` 動態構建：

1. 章節節奏分布
2. 角色弧線 × 章節矩陣
3. 資訊揭露時間軸
4. 伏筆與回收佈點
5. 高風險場景識別
6. 高風險場景處理方式

拆分規則屬 §10.7 agent-side execution mechanic，不列入 registry user-facing 議題。

## 4. 階段 2：逐題探索順序

## 4.0 議題清單動態構建（D-047 / Contract D §4a.2）

階段 2 啟動時，agent **必須**讀 `<instance_root>/issue_type_registry.yaml` 的 `00_h_detailed_outline` skill key 區段，按 **`core + user_extensions − core_overrides`** 動態構建議題清單：

1. **core 議題**先依 `id` 升序排列（id 1-99 為 core 保留範圍）。
2. **`user_extensions[*]`** 接在 core 議題後（id ≥ 100 為 user 自訂議題保留範圍）。
3. 套用 **`core_overrides[*].skip_id`** 過濾：
   - 對應 core entry `locked=true` → agent **忽略**該 override 並 **WARN**「locked 議題 id={n} 不可 SKIP，core_overrides 條目被忽略」；該議題仍出現在清單
   - 對應 core entry `locked=false` → 從議題清單中移除
   - `skip_id` 不存在於 core → WARN「core_overrides skip_id={n} 找不到對應 core 議題；條目被忽略」
4. 對每個議題：
   - `required_level=REQUIRED` → 跳階段 4 前**必答**；缺漏拒絕進階段 4
   - `required_level=STRONGLY_PREFERRED` → 跳階段 4 允許但 phase_log 紀錄缺漏 + 對應分拆檔段落標 TODO
   - `required_level=OPTIONAL` → 跳階段 4 允許不紀錄
5. `core[*].question_summary` 只作為 opener；agent 完整提問腳本詳本檔 §10 對應段（沿用 UD §1.4.2 完整內容）。
6. **拆分規則不入 registry** — 屬 agent 階段 4 寫檔 mechanic，由 §10.7 固定執行，不對 user 提問。

**Registry 異常處理：**
- 若 registry 載入失敗（YAML parse error / 缺 `00_h_detailed_outline` key 等）→ agent **拒絕進階段 2**，請 user 修補 registry
- 若 `user_extensions[*].id < 100` 或 與 core id 重複 → parser ERROR；agent 拒絕進階段 2

## 4.1 議題順序預設表

00_h core 共 **6 個 user-facing 議題**（registry 對應 id 1-6）。若 user 在 `user_extensions:` 加自訂議題，按上述 §4.0 規則接於 core 後。

| 順序 | 議題 | required_level | locked | 為什麼這時問 |
|---|---|---|---|---|
| 1 | 章節節奏分布 | REQUIRED | true | 全局節奏曲線是章節、弧線與資訊揭露安排的時間基底。 |
| 2 | 角色弧線 × 章節矩陣對齊 | REQUIRED | true | 角色在每章的狀態會影響聲線、行動與任務包條件。 |
| 3 | 資訊揭露時間軸 | REQUIRED | true | 核心資訊何時解禁會直接決定 09_d 與場景任務包欄位。 |
| 4 | 伏筆與回收佈點 | STRONGLY_PREFERRED | false | 伏筆埋點與回收節奏會影響任務包和資訊控制。 |
| 5 | 高風險場景識別 | REQUIRED | true | 先標出容易出戲的場景，後續才能設定 risk_type 與處理規則。 |
| 6 | 高風險場景的「作品專屬正確處理方式」 | STRONGLY_PREFERRED | false | 將高風險場景的處理基線寫入 00_b §6，供 09_f 使用。 |

**§10.7 拆分規則**屬 agent-side execution mechanic（非 user-facing 議題），階段 3 收斂預告稿時固定列入寫檔計畫；user 只需在預告稿確認，不對拆分規則本身提問。詳本檔 §10.7。

agent 必須依 §4.0 動態構建後的清單提問，不得跳到拆分或寫檔。

每題都必須輸出：

## 10. 專屬議題腳本

### 10.1 章節節奏分布

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 全局節奏曲線是後續所有議題的時間基底。 |
| agent 怎麼問 | 兩題：a. 對每章標 high / mid / low。high 是衝突高潮、揭露、犧牲、告別、覺醒；mid 是情節推動、關係變化、資訊釋出；low 是呼吸、鋪墊、日常、反差。b. 判斷全局節奏曲線，例如漸進高潮、先 low 後爆、波浪起伏、全篇高張。 |
| 使用者預期答什麼 | 子題 a 給每章標籤；子題 b 給一句節奏定位。 |
| agent 怎麼整理寫檔 | 寫入 `05_b §各章節奏` 與 `05_b §全局節奏曲線`；若 high > 40% 或 low > 50%，印出疲勞或鬆散警示。 |
| 拒答 / 跳題 | 必答；若只給曲線一句話，agent 可 INFERENCE 逐章標籤並請使用者確認。 |

### 10.2 角色弧線 × 章節矩陣對齊

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 角色在每章的弧線階段會影響後續聲線與資訊控制判定。 |
| agent 怎麼問 | 兩題：a. 主角從想要走向需要的過程，在哪幾章發生關鍵轉折，請用階段起訖標示。b. 其他主要 `C-*` 同樣標示弧線切點；無顯著弧線則標全篇穩定。 |
| 使用者預期答什麼 | 主角與主要角色的弧線階段切點。 |
| agent 怎麼整理寫檔 | 整理成 `05_c` 的章節 × 角色矩陣；若兩條主弧線同章重疊，提示可能擁擠。 |
| 拒答 / 跳題 | 主角必答；其他角色可跳並標 TODO。 |

### 10.3 資訊揭露時間軸

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 決定哪個資訊在哪一章解禁，直接影響 09_d 判定。 |
| agent 怎麼問 | 三題：a. 列 5-15 條核心資訊。b. 每條標解禁章節或永遠不直接揭露。c. 標揭露方式，例如角色說、環境揭示、玩家看出、結局頓悟。 |
| 使用者預期答什麼 | 核心資訊清單、解禁章節、揭露方式。 |
| agent 怎麼整理寫檔 | 寫入 `05_d` 表格；再計算各章節應透露 / 禁止透露資訊，寫入 `05_d §各章節資訊狀態`。 |
| 拒答 / 跳題 | 強烈建議不跳；若跳，09_d 與任務包資訊欄位都缺基線。 |

### 10.4 伏筆與回收佈點

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 影響任務包的潛伏伏筆欄位，也影響 09_d 與 09_a。 |
| agent 怎麼問 | 兩題：a. 列 3-10 條伏筆，每條註明埋於哪章、回收於哪章，或不回收。b. 是否有刻意誤導觀眾的點，在哪章揭穿。 |
| 使用者預期答什麼 | 伏筆清單與誤導 / 公平性清單。 |
| agent 怎麼整理寫檔 | 寫入 `05_e`；再計算各章該埋、該回、該避免說破的伏筆，寫入 `05_e §各章節伏筆狀態`。 |
| 拒答 / 跳題 | 可跳，但任務包與 09_d 對伏筆相關判定缺基線。 |

### 10.5 高風險場景識別

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 標出告別、和解、背叛、犧牲、告白、主角覺醒、反派理念揭露、世界觀重大揭露、關係破裂、重大資訊揭露、結局前對話等容易出戲的場景。 |
| agent 怎麼問 | 請列出本作所有高風險場景，每場註明 `S-<ch>-<n>` 或「CH-N 第 ? 場」加場景類型。 |
| 使用者預期答什麼 | N 條場景 ID 與類型配對。 |
| agent 怎麼整理寫檔 | 寫入 `05_e §高風險場景清單` 或併入 `06_a` 對應場景行，並為每個高風險場景加 `risk_type`。若高風險場景 > 30%，提示密度過高。 |
| 拒答 / 跳題 | 可跳，但 10.6 無法進行。 |

### 10.6 高風險場景處理方式

| 欄位 | 內容 |
|---|---|
| 為什麼問 | 對每種高風險場景類型建立作品專屬正確處理方式，寫入 `00_b §6`。 |
| agent 怎麼問 | 對 10.5 出現過的每種類型詢問處理方式。死亡 / 犧牲可選殘留化、流程化、諷刺化、臨死遺言式或自定義；真相揭露可選拼湊式、一次交代、環境揭露或自定義；關係推進可選行動式、稱呼切換式、不承認式、直接告白式或自定義；系統 / AI 對話、覺醒、反派理念揭露各依 UPSTREAM 選項提問。 |
| 使用者預期答什麼 | 對每個高風險類型給一個選項或自定義。 |
| agent 怎麼整理寫檔 | 每個類型 append 到 `00_b §6 高風險場景處理方式`，並同步寫入 `05_e §高風險場景處理規則`。 |
| 拒答 / 跳題 | 強烈建議不跳；若跳，09_f 在高風險場景時缺直接基線。 |

### 10.7 拆分規則

| 來源議題 | 目標 | 寫入方式 |
|---|---|---|
| 10.1 章節節奏分布 | `05_b §各章節奏` + `05_b §全局節奏曲線` | 補完空殼 |
| 10.2 弧線 × 章節矩陣 | `05_c` 整檔 | 創建或覆寫 |
| 10.3 資訊揭露時間軸 | `05_d` 整檔 | 創建或覆寫 |
| 10.4 伏筆與回收 | `05_e §伏筆清單` + `05_e §各章節伏筆狀態` | 創建或覆寫 |
| 10.5 高風險場景識別 | `06_a` 場景索引含 `risk_type` 註記 | 創建或覆寫 |
| 10.6 高風險處理方式 | `00_b §6` + `05_e §高風險場景處理規則` | append 子節 + 覆寫 |

寫檔順序固定為：

1. `05_b`
2. `05_c`
3. `05_d`
4. `05_e`
