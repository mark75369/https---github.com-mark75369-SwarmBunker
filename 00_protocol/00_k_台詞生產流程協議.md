# 00_k 台詞生產流程協議

## 1. 文件目的

本協議統合下游台詞生產 pipeline 的執行流程，被 `/scene-task`、`/dialogue-write`、`/qa` 三個 skill 共同引用。

本協議定義：

- pipeline 五個內部階段：任務包建立、 多版本試寫、QA、修稿與定稿、實體驗證
- 場景的 `pipeline_state` 狀態機與轉移條件
- 與既有 `00_protocol/00_a_台詞生產協議.md` 模式系統的對應
- 任務包必填欄位、多版本方向、**八份 QA 報告閱讀順序**（v0.2 / D-043；原五份 supersede）、跨 skill 呼叫關係
- 多場景並行時的鎖、衝突與 abort 紀錄規範

`00_a` 規範「什麼算好台詞」與模式方法；本協議規範「流程怎麼跑、狀態怎麼轉、skill 怎麼串」。本協議不重寫 00_a 的品質規則，只引用。

## 2. 啟動條件

下游 pipeline 啟動前，agent 必須確認最小條件：

| 條件 | 要求 |
|---|---|
| 上游基礎 | `W-rules`、`V`、`W-language` 至少 REVIEW |
| 角色 | 涉及本場的所有 `C-*` 至少 REVIEW |
| 關係 | 涉及本場的所有 `R-*-*` 至少 REVIEW；若本場無關係可略 |
| 主線與章節 | `P` 與本場對應 `CH-NN` 至少 REVIEW |
| 場景索引 | `S-<ch>-<n>` 已存在於 `06_scene_index/06_a_場景索引模板.md` |
| Bootstrap | `.protocol_version` 存在 |
| 作品專屬 00_b | `00_b §1` 到 `§6` 都已有內容 |

任一條件缺漏時，pipeline 預設拒絕進入，列出缺漏並建議補完路徑。

### 2.1 例外路徑

| 例外 | 行為 |
|---|---|
| `prereq_waived: true` | 由使用者明示；允許某項缺漏，但必須寫入 phase_log |
| 任務包已存在但依賴降級 | `/dialogue-write` 啟動時拒絕，要求先把依賴升回 REVIEW 或由使用者明示 waiver |

waiver 產物不得自動升 FINAL。

## 3. 階段 1：任務包建立

本階段由 `/scene-task` 執行，目標是把 `S-<ch>-<n>` 場景索引轉成單場台詞任務包。

### 3.1 內部子階段 1：診斷

agent 讀取 `06_scene_index/06_a_場景索引模板.md`，定位該場景行，抽取：

- 場景 ID
- 場景名稱
- 出場角色
- 地點與時間
- 章節
- 場景節奏或 risk_type

agent 必須檢查所有先決實體狀態，並列出：

- 未達 REVIEW 的 W / V / C / R / P / CH
- `05_plot/05_d_資訊揭露表.md` 中該章必須透露 / 禁止透露候選
- `05_plot/05_e_伏筆與回收表.md` 中該章伏筆候選
- 任務包欄位草稿與 TODO 欄位

### 3.2 內部子階段 2：探索與抽取

agent 從下列來源抽取本場必要資訊：

| 來源 | 抽取內容 | 填入任務包欄位 |
|---|---|---|
| 01_a | 本場需要的世界規則摘要 | 世界規則約束 |
| 01_b / 01_c | 相關陣營、階級、語言層級 | 語言層級 |
| 02_a / 02_b / 02_c | 禁用詞、慎用詞、解禁詞、俗稱黑話 | 禁用詞與禁用句型 |
| 03_characters | 出場角色聲線卡引用與偏移重點 | 出場角色聲線卡引用 |
| 04_a / 04_b | 本場關係狀態、稱呼規則、禁語 | 本場關係狀態 |
| 05_b | 本章節奏 high / mid / low | 章節節奏與風格要求 |
| 05_c | 本場出場角色的弧線階段 | 角色弧線階段 |
| 05_d | 該章該透露 / 禁止透露清單 | 必須透露資訊 / 禁止透露資訊 |
| 05_e | 本場潛伏伏筆 / 回收任務 | 潛伏的伏筆 |
| 00_b §6 | 高風險場景處理規則 | 風格要求 |

本階段只抽取與整理，不寫台詞。

### 3.3 內部子階段 3：收斂

agent 把抽取資料整合成單場任務包草稿，按 `07_scene_tasks/07_a_單場台詞任務包模板.md` 填寫。

收斂預告稿必須包含：

- 任務包 markdown 預覽
- TODO 欄位清單
- 核心欄位是否完整
- 每個缺漏欄位會影響哪一份 QA
- 是否接受帶 TODO 寫檔

未經使用者確認，不得寫檔。

### 3.4 內部子階段 4：執行

任務包檔名依場景 ID 命名；路徑樣式如下，實作時應用實際章節與場景號取代 placeholder：

```text
07_scene_tasks/CH<n>_S<m>_台詞任務包.md
```

frontmatter 必須含標準中文 header 與下游欄位：

```yaml
狀態：DRAFT
版本：v0.1
最後更新：YYYY-MM-DD
適用範圍：CH<n>_S<m> 台詞任務包
優先級：中

---
entities: [S-<n>-<m>]
depends_on: [W-rules, V, W-language, <相關 C-*>, <相關 R-*-*>, P, CH-<n>]
weight: {S-<n>-<m>: 1.0}
scene_id: S-<n>-<m>
source_task: null
source_dialogue: null
source_dialogues: null
pipeline_state: TASK_DRAFT
mode_tag: null
qa_decision: null
qa_type: null
---
```

寫檔失敗時 rollback，不得留下半成品。

### 3.5 內部子階段 5：驗證

任務包寫完後：

- 自動 `/status`
- 確認 `S-<n>-<m>` 完成度上升
- 在 `.protocol_version.phase_log` append `/scene-task` 紀錄
- 印建議下一步：人類審任務包；通過後跑 `/dialogue-write`

phase_log 紀錄格式：

```yaml
- phase: scene-task
  date: YYYY-MM-DD
  skill: /scene-task
  scene_id: S-<n>-<m>
  task_path: 07_scene_tasks/CH<n>_S<m>_台詞任務包.md
  todo_count: <N>
  customizations: []
  status: completed
```

## 4. 階段 2：多版本試寫

本階段由 `/dialogue-write` 執行，目標是依任務包產出多個台詞試寫版本。

### 4.1 啟動前置：任務包 REVIEW gate

`/dialogue-write` 啟動時必須檢查任務包狀態：

| 任務包狀態 | pipeline_state | 行為 |
|---|---|---|
| DRAFT | TASK_DRAFT | 拒絕；需先過任務包 review gate |
| REVIEW | TASK_REVIEW | 允許執行 |
| REVIEW | 其他 | 拒絕；狀態欄與 pipeline_state 不一致 |
| FINAL / LOCKED | 任意 | 允許但提示本次不修改任務包 |

agent 不得自動把任務包從 DRAFT 升 REVIEW。

### 4.2 內部子階段 1：診斷

agent 讀任務包並檢查核心欄位：

- 出場角色
- 必須透露資訊
- 禁止透露資訊
- 出場角色聲線卡引用
- 角色表層目標
- 角色真實目標

任一核心欄位缺，`/dialogue-write` 拒絕執行。非核心 TODO 可保留，但必須寫入 phase_log 的 incomplete_fields。

### 4.3 內部子階段 2：探索

探索是可選階段，只有使用者明示才執行。

探索內容：

- 產 1-2 段短篇語氣方向片段
- 不寫檔
- 讓使用者選哪個方向較接近本場目標

### 4.4 內部子階段 3：試寫

預設產 3 版本：

| 版本 | 方向 | 句長偏好 | 潛台詞密度 | 攻防強度 | 情緒外露 |
|---|---|---|---|---|---|
| v01A | 克制、短句、強潛台詞 | 短 | 高 | 中 | 低 |
| v01B | 攻防更強、衝突更尖 | 中 | 中 | 高 | 中 |
| v01C | 情緒更重、但避免直說 | 中到長 | 中到高 | 中 | 中到高 |

任務包可在「輸出模式要求」欄指定額外方向，如 v01D / v01E / v01F。

台詞檔 frontmatter 範例：

```yaml
---
entities: [S-<n>-<m>]
depends_on: [<本場相關 C-*>, <本場相關 R-*-*>]
weight: {S-<n>-<m>: 1.0}
scene_id: S-<n>-<m>
source_task: 07_scene_tasks/CH<n>_S<m>_台詞任務包.md
source_dialogue: null
source_dialogues: null
pipeline_state: DIALOGUE_TRIAL
mode_tag: DRAFT_TRIAL
qa_decision: null
qa_type: null
---
```

### 4.5 內部子階段 4：破格

破格是可選階段，只有使用者明示才執行。

規範：

- 依 00_a 破格模式產出 experimental 版本
- `mode_tag: EXPERIMENTAL`
- `pipeline_state: DIALOGUE_TRIAL`
- 狀態保持 DRAFT
- 不混入正式 v01A / v01B / v01C

### 4.6 內部子階段 5：收斂

收斂由使用者選亮點後觸發，典型指令為 `/dialogue-write --converge`。

收斂條件：

- 至少有 2 個 trial 版本可比較，除非使用者明示只收斂 1 個版本
- 使用者明確指出要保留、刪除、混合或重寫哪些片段
- agent 產出收斂版，狀態通常為 REVIEW
- `pipeline_state: DIALOGUE_CONVERGED`
- `mode_tag: CONVERGENCE`

未經使用者選版，agent 不得自行挑選亮點。

### 4.7 內部子階段 6：驗證

完成試寫或收斂後：

- 自動 `/status`
- 在 `.protocol_version.phase_log` append `/dialogue-write` 紀錄
- 印下一步：人類挑亮點 + 收斂，或例外直接 `/qa`

phase_log 範例：

```yaml
- phase: dialogue-write
  date: YYYY-MM-DD
  skill: /dialogue-write
  scene_id: S-<n>-<m>
  dialogue_paths: [<v01A path>, <v01B path>, <v01C path>]
  mode_tag: DRAFT_TRIAL
  status: completed
```

## 5. 階段 3：QA（v0.2 / D-043 — 8 份必跑；原 5 份 supersede）

本階段由 `/qa` 執行，目標是對目標台詞檔產出 **8 份 QA 報告**（09_a/b/c/d/f/g/h/i）。

### 5.1 啟動前置

| 台詞檔狀態 | pipeline_state | 行為 |
|---|---|---|
| DRAFT | DIALOGUE_TRIAL | 允許例外 QA；提示 QA_PASSED 後不能直接升 FINAL |
| REVIEW | DIALOGUE_CONVERGED | 允許，標準路徑 |
| 其他 | 其他 | 拒絕 |

### 5.2 內部子階段 1：診斷

agent 必須：

- 讀目標台詞檔
- 從 `source_task` 反查任務包
- 從 `entities` / `depends_on` 反查 C-*、R-*-* 等聲線卡與關係矩陣
- 印本次 QA 對象元資訊

### 5.3 內部子階段 2：執行 8 份報告（v0.2 / D-043；原 5 份 supersede）

8 份檢查並行執行，但報告必須依下列序列印出（依 UD §2.5.3 v0.3 權威序列）：

| 印出順序 | 報告 | qa_type | 原因（依 v0.3 邏輯重排） |
|---|---|---|---|
| 1 | 09_f 類型偏移檢查 | GENRE_DRIFT | 最優先 — 類型錯了，其他判定基準會跑掉 |
| 2 | 09_d 資訊控制檢查 | INFO_CONTROL | 資訊洩漏先於 character 層 |
| 3 | 09_h 對話張力檢查 | DRAMATIC_TENSION | 張力強度標準依類型；09_f 之後跑 |
| 4 | 09_b 角色聲線一致性 | VOICE_CONSISTENCY | 角色 OOC 判定 |
| 5 | 09_g 節奏感檢查 | RHYTHM | 節奏依賴聲線；09_b 之後 |
| 6 | 09_a AI 味檢查 | AI_FLAVOR | 表層字面層 |
| 7 | 09_c 禁用詞檢查 | FORBIDDEN_WORD | 機械詞表比對 |
| 8 | 09_i 跨場一致性檢查 | CROSS_SCENE_CONTINUITY | 最後 — 需所有 per-scene 結果交叉比對 |

若 09_f 命中重大問題，agent 必須提示：後面 **7 份**報告的判定基準可能要重看（v0.2 / D-043；原「後 4 份」supersede）。

09_g/h/i 為 v0.2 新預設併入並行清單（取代 v0.1「可選跑」描述）。每份報告依各自模板生成；frontmatter 含完整下游 8 欄。

### 5.4 內部子階段 3：彙整（v0.2 / D-043）

agent 統合 **8 份報告**中的最高優先問題，計算整體 `qa_decision`：

| qa_decision | 條件 |
|---|---|
| PASS | **8 份全 PASS**（v0.2 / D-043；原「5 份全 PASS」supersede） |
| FAIL | 任一份 FAIL |
| ARBITRATE_REQUIRED | 人類要保留違規亮點；不由 agent 自動產生，需使用者後續裁決 |

### 5.5 內部子階段 4：寫檔（v0.2 / D-043）

QA 寫檔規則：

- **8 份報告**各寫一份到 `09_quality_assurance/`
- 檔名使用 `<base>_<QA 類型>_報告.md`
- 同步更新目標台詞檔 frontmatter 的 `pipeline_state` 與 `qa_decision`
- 更新台詞檔時只動 frontmatter，不動台詞內容
- 不產生 `09_e`；09_e 是人類 final-gating 紀錄

### 5.6 內部子階段 5：驗證（v0.2 / D-043）

QA 完成後：

- 自動 `/status`
- append phase_log（含 8 份 qa_report_paths）
- 印 QA 結論與下一步建議

phase_log 範例（v0.2 / D-043；對齊 UD §2.5.6）：

```yaml
- phase: qa
  date: YYYY-MM-DD
  skill: /qa
  status: completed
  scene_id: S-<n>-<m>
  target_dialogue: <台詞檔路徑>
  qa_report_paths:
    - <09_a 報告路徑>      # qa_type: AI_FLAVOR
    - <09_b 報告路徑>      # qa_type: VOICE_CONSISTENCY
    - <09_c 報告路徑>      # qa_type: FORBIDDEN_WORD
    - <09_d 報告路徑>      # qa_type: INFO_CONTROL
    - <09_f 報告路徑>      # qa_type: GENRE_DRIFT
    - <09_g 報告路徑>      # qa_type: RHYTHM           — v0.2 新增
    - <09_h 報告路徑>      # qa_type: DRAMATIC_TENSION — v0.2 新增
    - <09_i 報告路徑>      # qa_type: CROSS_SCENE_CONTINUITY — v0.2 新增
  qa_decision: PASS  # 或 FAIL（v0.2 / D-043：8 份全 PASS 才 PASS）
```

## 6. 階段 4：修稿與定稿

本階段不由單一 skill 完成，由人類在 chat 與 agent 互動處理。

### 6.1 QA_FAILED 處理路徑

| 路徑 | 行為 |
|---|---|
| 修稿 | agent 依 QA 報告產出建議修正；使用者拍板；採納後產新版本，回到 DIALOGUE_CONVERGED 並再跑 QA |
| 重生 | 使用者要求整個重來；既有版本不刪除，狀態降 DEPRECATED 並寫 09_e 棄用紀錄 |
| 人類裁決保留 | 使用者保留 QA 標違規但有價值的版本；不修台詞；填 09_e 留證後可升 FINAL |

### 6.2 QA_PASSED 到 FINAL（v0.2 / D-043 — 9 種 status 齊全 gate）

QA_PASSED 後：

1. 使用者人類確認台詞可定稿。
2. 必須填 `09_quality_assurance/09_e_定稿變更紀錄模板.md` 對應紀錄。
3. 09_e 含最終措辭、QA 結果摘要、是否保留違規或破格句、人類修改摘要、影響範圍評估。
4. 使用者把台詞檔改為 `狀態：FINAL`、`pipeline_state: DIALOGUE_FINAL`、`mode_tag: FINAL_CANDIDATE`。

agent 可輔助格式化，但不得自行完成 final gate。

**FINAL gate 9-status 齊全條件（v0.2 / D-043；對齊 UD §2.6.2）：**

| # | status | 來源 | 通過條件 |
|---|---|---|---|
| 1 | 09_a AI_FLAVOR | `/qa` 產出 | `qa_decision: PASS` 或 `ARBITRATE_REQUIRED + 09_e 已紀錄` |
| 2 | 09_b VOICE_CONSISTENCY | `/qa` 產出 | 同上 |
| 3 | 09_c FORBIDDEN_WORD | `/qa` 產出 | 同上 |
| 4 | 09_d INFO_CONTROL | `/qa` 產出 | 同上 |
| 5 | 09_f GENRE_DRIFT | `/qa` 產出 | 同上 |
| 6 | 09_g RHYTHM | `/qa` 產出（v0.2 新增） | 同上 |
| 7 | 09_h DRAMATIC_TENSION | `/qa` 產出（v0.2 新增） | 同上 |
| 8 | 09_i CROSS_SCENE_CONTINUITY | `/qa` 產出（v0.2 新增） | 同上 |
| 9 | 09_e final-gating 紀錄 | 人類填寫 | 已存在 + 涵蓋本版本 |

**Gate 行為：**

- 9 項全通過 → 允許升 FINAL
- 任一項缺漏 / FAIL 未 ARBITRATE → 拒絕升 FINAL；印缺漏清單

### 6.3 FINAL 到 LOCKED

使用者確認該版本不再修改時，才可升 LOCKED。

LOCKED 後若還要改，必須走迭代與 delta 流程；不得直接覆寫。

### 6.4 Trial 直接 QA 例外

使用者可明確命令直接對 trial 版跑 QA。

規則：

- 跳過收斂，直接 QA trial 版
- QA_PASSED 後 `pipeline_state` 可升 `QA_PASSED`
- 不能直接升 `DIALOGUE_FINAL`
- 必須先回到 `DIALOGUE_CONVERGED`，再走標準 FINAL 路徑

## 7. 階段 5：實體驗證

每個 skill 結束都自動跑 `/status`。

下游 pipeline 對 `/status` 的完成度期望：

| 階段 | 場景完成度期望 |
|---|---|
| 00_h 後 | 20% |
| /scene-task 後 | 35% |
| /dialogue-write trial 後 | 60% |
| /dialogue-write 收斂後 | 75% |
| /qa PASS 後 | 90% |
| FINAL | 100% |

若完成度未上升但 phase_log 已標完成，`/status` 應印「下游狀態不一致」警示。

每階段都需有對應 phase_log entry；缺者印「pipeline 紀錄缺失」警示。

## 8. 禁止事項

1. agent 不得擅自把 DRAFT 升 FINAL 或 LOCKED。
2. agent 不得跳過 QA gate 直接定稿。
3. agent 不得在高風險場景自行裁決定案；必須走 09_e 人類裁決紀錄。
4. agent 不得擅自修改 LOCKED 場景的台詞內容。
5. agent 不得把破格版混入正式 v01A / v01B / v01C。
6. agent 不得擅自刪除使用者標記保留的句子。
7. agent 不得新增 pipeline_state / mode_tag enum 值。
8. agent 不得擅自產生 09_e。
9. agent 不得跳過 8 份 QA 報告任一份（v0.2 / D-043；原「5 份」supersede）。
10. agent 不得在任務包核心 TODO 缺漏時執行 `/dialogue-write`。

## 9. 缺漏處理

### 9.1 任務包不完整

| 缺漏 | 處理 |
|---|---|
| 核心欄位缺 | `/dialogue-write` 拒絕 |
| 非核心欄位缺 | 可執行，但 phase_log 標 `incomplete_fields`；QA 報告附缺漏警示 |

核心欄位是任務包欄位 3 / 9 / 10 / 12 / 14 / 15。

### 9.2 上游實體未達 REVIEW

預設拒絕。

若使用者明示 `prereq_waived: true`：

- phase_log 必須標明原因
- 該版本不可升 FINAL
- QA 報告需提示依賴缺口

### 9.3 CONFLICT

任務包與既有實體衝突時，agent 必須在 `/scene-task` 階段 1 診斷標 CONFLICT。

例：05_d 規定本章不能透露 X，但任務包必須透露 X。

若使用者堅持寫：

- 在任務包標 `CONFLICT(資訊揭露)`
- 寫入衝突理由
- 下游 09_d 會持續報警，直到使用者修 05_d 或修任務包

### 9.4 過時依賴

任務包寫完後，若某 C-* / R-*-* 被 iterate 並降級：

- `/dialogue-write` 啟動時偵測
- 印出依賴已降級警示
- 使用者拍板繼續或暫停
- 若繼續，phase_log 標 `prereq_changed: true`

## 10. 專屬區段

### 10.1 場景狀態機

狀態機固定如下：

```text
SCENE_INDEXED
  -> /scene-task
TASK_DRAFT
  -> 人類審任務包；agent 不可自動升
TASK_REVIEW
  -> /dialogue-write
DIALOGUE_TRIAL
  -> 人類挑亮點 + /dialogue-write --converge
DIALOGUE_CONVERGED
  -> /qa
QA_PASSED 或 QA_FAILED
  -> QA_PASSED: 人類確認 + 填 09_e
DIALOGUE_FINAL
  -> 人類確認 LOCKED
DIALOGUE_LOCKED
```

trial 直接 QA 是例外路徑：

```text
DIALOGUE_TRIAL
  -> 使用者明示直接 /qa
QA_PASSED 或 QA_FAILED
  -> QA_PASSED 後不能直接升 FINAL
  -> 必須回 DIALOGUE_CONVERGED
```

轉移觸發條件：

| 從 | 到 | 觸發 | 檢查項 |
|---|---|---|---|
| SCENE_INDEXED | TASK_DRAFT | /scene-task | 先決實體 REVIEW、場景在 06_a 存在 |
| TASK_DRAFT | TASK_REVIEW | 人類手動 | 任務包 review gate；agent 不可自動 |
| TASK_REVIEW | DIALOGUE_TRIAL | /dialogue-write | 核心欄位齊全 |
| DIALOGUE_TRIAL | DIALOGUE_CONVERGED | /dialogue-write --converge | 人類選亮點；提供 trial 路徑 |
| DIALOGUE_TRIAL | QA_PASSED / QA_FAILED | /qa + 使用者明示 | 直接 QA trial |
| DIALOGUE_CONVERGED | QA_PASSED / QA_FAILED | /qa | 8 份報告完整（v0.2 / D-043；原 5 份 supersede） |
| QA_PASSED | DIALOGUE_FINAL | 人類手動 + 09_e | 不可由 agent 自動 |
| DIALOGUE_FINAL | DIALOGUE_LOCKED | 人類手動 | 通常整章或 milestone 鎖時做 |
| QA_FAILED | DIALOGUE_CONVERGED | 修稿路徑 | 新版本需再 QA |
| QA_FAILED | DIALOGUE_TRIAL | 重生路徑 | 既有版本 DEPRECATED |
| QA_FAILED | DIALOGUE_FINAL | 人類保留 | 09_e 留證 |

### 10.7 多場景並行注意事項

#### 10.7.1 並行容忍度

| 操作 | 容忍度 | 衝突類型 |
|---|---|---|
| /scene-task 同時對不同場景 | 完全並行可 | 無 |
| /dialogue-write 同時對不同場景 | 完全並行可 | 無 |
| /qa 同時對不同場景 | 完全並行可 | 無 |
| /scene-task 同時對同一場景 | 禁止 | 任務包寫入衝突 |
| /dialogue-write 同時對同一場景 | 禁止 | 台詞檔寫入、phase_log 衝突 |
| /qa 同時對同一台詞檔 | 禁止 | QA 報告覆蓋、frontmatter 衝突 |
| 同一場景 /scene-task + /dialogue-write | 禁止 | pipeline_state 競爭 |
| /iterate-* 與下游 skill 同時跑 | 禁止 | 上游實體狀態變動，下游檢查失效 |

#### 10.7.2 phase_log 寫入鎖

`.protocol_version.phase_log` 是所有 skill 共寫的 sequential log。每個 skill 在階段 5 寫入前必須取得檔案鎖。

規範：

1. 鎖檔位置：`.protocol_version.lock`
2. 鎖類型：排他鎖
3. 取得鎖後 append phase_log
4. append 後立即釋放鎖
5. 30 秒內取不到鎖，印「另一個 skill 正在寫 phase_log，請稍候重試」並 abort 自己的階段 5

fallback：

- 若環境不支援檔案鎖，可用 sentinel file。
- sentinel fallback 有 race window，只作單人使用的低風險替代。

#### 10.7.3 場景狀態升級鎖

`pipeline_state` 升級是 frontmatter 修改。agent 在升級前必須檢查：

- 預期狀態
- 實際狀態

兩者不一致時，印「狀態已被其他 skill 改變，請重新檢查 pipeline」並 abort。

寫 frontmatter 時：

- 使用 `<file>.lock` 排他鎖
- 30 秒 timeout
- 採整檔讀寫
- 用 tmp + rename pattern 完成 atomic 寫入

#### 10.7.4 任務包與台詞檔串行約束

同一場景 ID：

1. `/scene-task` 完全跑完後才能跑 `/dialogue-write`
2. 任務包需過 TASK_REVIEW gate
3. `/dialogue-write` 至少產出 1 版後才能跑 `/qa`
4. 三個 skill 不可對同一場景並行

#### 10.7.5 跨場景依賴檢查

若場景 A 與場景 B 同時跑下游 skill，且都依賴同一個 C-*，上游 `/iterate-*` 啟動時必須掃描 phase_log 中進行中的下游 skill。

若正在進行的下游 skill 引用了將被迭代的實體：

- `/iterate-*` 拒絕
- 列出使用該實體的下游 skill
- 請使用者拍板：等待完成、中止下游、或強制迭代

UPSTREAM §6.5 假設 phase_log 可標 `status: in_progress / completed / aborted`；這是 phase_log 內部結構假設，不新增本 repo 的 5.2 frontmatter enum。

#### 10.7.6 並行模式 skill 行為

不同場景並行時：

- 每個 skill 實例獨立跑自己的 5 階段
- Bible 上游檔案只讀可並行
- phase_log 與 frontmatter 寫入走鎖
- chat 呈現留給 UX 後續設計

#### 10.7.7 並行衝突紀錄

因鎖 timeout 或並行衝突 abort 時，phase_log 必須記錄：

```yaml
- phase: <skill>
  date: YYYY-MM-DD
  skill: /<skill>
  scene_id: <ID>
  status: aborted
  abort_reason: parallel_conflict
  detail: <具體哪個鎖、等多久、衝突的另一個 skill 是什麼>
```

`/status` 看到 `status: aborted` 時，列出「本場 pipeline 有未完成的 skill 嘗試」並建議重跑。

### 10.8 UI / UX 標記

下列事項留給 UX specialist 後續設計，不在本協議實作：
