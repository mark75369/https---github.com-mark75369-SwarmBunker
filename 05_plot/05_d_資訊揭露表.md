# 1. 文件目的

本文件用於控制故事資訊何時、由誰、以何種方式被玩家與角色知道。

在長篇遊戲劇本中，資訊揭露直接影響：

- 懸念
- 任務動機
- 角色關係
- 玩家理解
- 世界觀壓力
- 反派威脅
- 台詞自然度
- 伏筆與回收
- 劇情反轉
- 分支選擇

本文件的目的，是讓 AI / Codex 在生成大綱、場景任務包與台詞時能判斷：

- 玩家此時知道什麼？
- 玩家此時誤解什麼？
- 玩家此時絕對不能知道什麼？
- 角色此時知道什麼？
- 角色是否知道自己知道的是錯的？
- 這個真相何時可以被暗示？
- 這個真相何時可以被明說？
- 哪些台詞會提前劇透？
- 哪些正式名詞或稱呼尚未解禁？
- 哪些資訊只能透過行動、物件或環境呈現？

本文件是防止 AI 使用作者視角劇透的核心文件。

# 3. 資訊揭露管理原則

## 3.1 作者知道不等於角色知道

AI 最常見問題是使用作者視角。

生成台詞時，必須區分：

| 層級 | 說明 |
|---|---|
| 作者知道 | 創作者完整知道的真相 |
| AI / Codex 可讀 | 文件中存在，但不代表角色可說 |
| 玩家知道 | 玩家在當前劇情階段已掌握的資訊 |
| 玩家懷疑 | 玩家可能透過伏筆猜到，但尚未確認 |
| 角色知道 | 該角色在當前場景已知道的資訊 |
| 角色誤解 | 該角色相信但不正確的資訊 |
| 角色不能說 | 角色知道但因恐懼、利益、誓言或劇情限制不能說 |

任何台詞都必須使用「角色當前可用資訊」，不是使用作者完整資訊。

---

## 3.2 資訊可以分層揭露

一個真相不一定一次說完。

可分為：

1. 無跡象
2. 氣氛暗示
3. 物件暗示
4. 稱呼異常
5. 角色反應異常
6. 局部線索
7. 錯誤解釋
8. 玩家可懷疑
9. 角色可懷疑
10. 明確揭露部分真相
11. 完整揭露
12. 重新解讀前面線索

高品質揭露通常不是直接說明，而是讓玩家逐步修正理解。

---

## 3.3 角色說出真相必須有理由

角色不應因為劇情需要就直接說明真相。

角色說出重要資訊時，應至少符合以下條件之一：

- 角色想操控對方
- 角色被逼問到無法迴避
- 角色誤以為對方已經知道
- 角色在情緒失控下露出破綻
- 角色刻意說半真半假的資訊
- 角色以為這不是重要資訊
- 角色用真相換取利益
- 角色知道再不說會付出更大代價
- 角色臨死、離開或背叛前留下線索

若沒有角色動機，資訊揭露會變成作者旁白。

# 4. 資訊分類

## 4.1 資訊類型表

| 類型 | 說明 | 常見風險 |
|---|---|---|
| 世界真相 | 世界起源、制度謊言、災難原因 | 提前劇透世界觀 |
| 角色真相 | 身份、血緣、背叛、死亡、過去 | 提前改變關係 |
| 陣營真相 | 組織目的、反派計畫、內部背叛 | 讓反派失去懸念 |
| 任務真相 | 任務真正目的、委託人謊言 | 破壞任務反轉 |
| 關係真相 | 真實關係、舊債、未說出口的事 | 關係太快說清楚 |
| 系統真相 | 魔法、科技、污染、超自然規則 | 角色像在教學 |
| 結局真相 | 最終選擇、結局條件、核心代價 | 提前削弱結局 |
| 名詞真相 | 正式名、真名、禁稱、俗稱含義 | 稱呼提前解禁 |

---

## 4.2 資訊狀態定義

| 狀態 | 意義 | 台詞可用性 |
|---|---|---|
| HIDDEN | 完全隱藏 | 不可說，不可明顯暗示 |
| FORESHADOW | 可伏筆 | 可用物件、氣氛、反應暗示 |
| MISLEAD | 可誤導 | 可給錯誤說法或局部說法 |
| SUSPECT | 玩家可懷疑 | 可提供矛盾線索，但不可確認 |
| PARTIAL | 部分揭露 | 可說一部分，但不可說完整原因 |
| REVEALED | 明確揭露 | 可由角色或事件說明 |
| PUBLIC | 世界內公開 | 多數角色可自然提及 |
| LOCKED_SECRET | 永久保留或結局後才揭露 | 禁止在主線中說破 |

# 10. 揭露方式管理

## 10.1 揭露方式表

| 揭露方式 | 適合資訊 | 風險 | 注意事項 |
|---|---|---|---|
| 角色台詞 | 角色知道且有理由說出的資訊 | 容易說明化 | 必須有角色動機 |
| 角色失言 | 被壓力逼出的半真相 | 容易太巧 | 需有情緒或場景壓力 |
| 物件 | 歷史、關係、承諾、身份 | 玩家可能忽略 | 可後續回收 |
| 環境 | 世界規則、災難、階級壓迫 | 可能不明確 | 需搭配其他線索 |
| 任務結果 | 制度真相、選擇代價 | 需玩家參與 | 不要只靠 NPC 解釋 |
| 敵人行動 | 反派目的、威脅 | 容易直白 | 行動比演說更有力 |
| 文件 / 日誌 | 背景資訊 | 容易百科化 | 應有視角與偏見 |
| 玩家選項 | 玩家理解或誤解 | 容易劇透 | 選項不能知道不該知道的事 |

---

## 10.2 不建議揭露方式

以下方式應慎用：

- NPC 長篇解釋世界觀
- 反派完整講出計畫
- 角色突然自我分析
- 主角替玩家整理所有線索
- 任務簡報直接說出真相
- 對話中連續出現大量專有名詞
- 系統提示提前告知劇情意義

# 14. 高風險資訊類型

## 14.1 高風險劇透表

| 資訊類型 | 高風險原因 | 常見誤觸方式 | 檢查重點 |
|---|---|---|---|
| 主角真實身份 | 會重寫玩家理解 | 稱呼、反派台詞、文件名 | 真名與代稱 |
| 反派真正目的 | 會削弱懸念 | 反派演說、NPC 說明 | 反派只說局部理由 |
| 世界最大謊言 | 會破壞中後期反轉 | 世界觀解釋過多 | 前期只給裂縫 |
| 主要角色背叛 | 會破壞關係推進 | 角色說漏、旁白暗示太明顯 | 只能用行動不一致暗示 |
| 血緣 / 親密關係 | 會改變稱呼含義 | 暱稱、姓氏、舊物 | 稱呼解禁表 |
| 結局條件 | 會讓玩家功利選擇 | UI、任務提示 | 只提示情緒代價 |

## 14.2 高風險台詞情境

以下場景特別容易提前揭露資訊：

- 反派與主角首次長談
- 導師角色教學
- 任務委託說明
- 戰前作戰會議

# 16. Codex 使用規則

當 Codex 修改或使用本文件時，應遵守：

1. 不得擅自改變 LOCKED 資訊揭露時點。
2. 不得自動補完主線核心真相、反派目的、主角身份或結局條件。
3. 若新增資訊，需填寫資訊ID、類型、首次暗示、部分揭露、完整揭露與狀態。
4. 若台詞中出現未解禁資訊，應回報並建議修正，不得硬判通過。
5. 若修改資訊揭露時點，需提醒更新 `05A_主線大綱模板.md`、`05B_章節結構模板.md` 與相關場景任務包。
6. 若資訊涉及角色弧線，需提醒更新 `05C_角色弧線表.md`。
7. 若資訊涉及伏筆或誤導，需提醒更新 `05E_伏筆與回收表.md`。
8. 若資訊涉及稱呼、真名、俗稱，需提醒更新 `02A_專有名詞表.md` 與 `02B_俗稱與黑話表.md`。
9. 修改後必須列出影響範圍。

# 19. 最終原則

資訊揭露的目標，不是讓玩家越快知道越好，而是讓玩家在正確的時候知道正確的部分。

好的資訊揭露應該讓玩家：

- 前期以為自己懂
- 中期發現有裂縫
- 後期重新理解前面看過的東西
- 結局前知道足夠做選擇
- 結局後仍能回想起早期伏筆

AI 生成台詞時，最重要的不是「這句話是否正確」，而是「這句話在這個時間點能不能被這個角色說出來」。

若一句台詞說出了正確答案，但破壞了資訊節奏，它仍然是錯的。
